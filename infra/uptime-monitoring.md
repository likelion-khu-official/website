# 외부 가동 감시 — UptimeRobot (#83)

> 인프라 문서. #83(조용한 실패를 먼저 알아채기)의 ①②(사이트 외부 접속 불가·서버 전체 다운) 담당.

## 왜 UptimeRobot인가

①②는 "서버가 통째로 죽어도 우리가 먼저 안다"가 요구사항이라, 감시 주체가 서버 밖에 있어야 한다(같은 인스턴스 위에 감시를 올리면 인스턴스가 죽을 때 감시도 같이 죽어서 요구사항 자체가 성립 안 함).

검토한 대안과 제외 이유:

| 대안 | 제외 이유 |
|---|---|
| OCI Health Checks | Always Free 목록에 없고, 공식 가격표에 Basic ¥46.50/엔드포인트/월로 사용량 과금 명시 — 이슈의 "새 유료 서비스 금지" 원칙에 걸림 |
| OCI Load Balancer를 헬스체크 전용으로 nginx 앞에 추가 | 무료(Always Free 1개, 10Mbps)지만, 실 트래픽을 안 받는 "감시 전용 유령 리소스"가 됨 — 트래픽 경로에 제대로 넣으려면 nginx 재구성(SSL 종단·도메인 라우팅 이관)까지 번져서 이번 미션 범위 초과. 보류 |
| 자체 self-host 스택(Prometheus/Loki/Grafana 등) | 같은 인스턴스 위에 올리면 인스턴스가 죽을 때 감시도 같이 죽음 — ①②를 구조적으로 못 풂 |

→ 외부 무료 SaaS(UptimeRobot)로 결정. ③(디스크/메모리 사전경고)은 이 문서 범위 밖 — OCI Monitoring/Alarm/Notifications로 별도 구축됨(디스크는 custom metric, 메모리는 네이티브 `MemoryUtilization`).

## 계정 및 ToS

- **가입**: 동아리 공식 메일로 신규 가입 (개인 계정 아님 — 특정 인원 의존도 낮추기 위함)
- **ToS 확인**: UptimeRobot 무료 플랜은 2024-12부터 상업적 사용 금지("businesses, organizations... on behalf of" 문구)이나, **오픈소스·교육 목적·비영리 단체는 명시적 예외**. 대학 동아리(비영리, 수익 없음, 학생 홍보 목적)는 이 예외에 해당 — 위반 소지 낮음. 참고로 이건 형사법이 아니라 서비스 약관(위반 시 계정 정지 리스크 정도).

## 모니터링 대상 (4개)

| URL | 역할 | 확인한 정상 응답 |
|---|---|---|
| `https://api.prod.likelion-khu.com/actuator/health` | 백엔드 prod | `200 {"status":"UP"}` |
| `https://api.stage.likelion-khu.com/actuator/health` | 백엔드 stage | `200 {"status":"UP"}` |
| `https://likelion-khu.com` | 프론트 prod | `200` |
| `https://dev.likelion-khu.com` | 프론트 dev/stage | `200` |

백엔드는 루트(`/`)가 아니라 `/actuator/health`로 등록 — 루트는 인증 없이 찌르면 401/403이 정상 응답이라 다운 오탐 소지가 있어서 Spring Boot Actuator 표준 헬스 엔드포인트를 씀(내부 상태 이상 시 자동으로 `DOWN`/503 반환).

**모니터 타입**: HTTP(S) 방식 — 응답 본문의 특정 문자열(Keyword)을 보는 게 아니라 **HTTP 상태 코드**로 UP/DOWN을 판정함. 즉 `/actuator/health`가 정상이면 200, 내부 이상이면 503을 반환하는 것에 그대로 의존 — 만약 나중에 애플리케이션이 500대가 아닌 에러(예: 200인데 본문만 이상)를 내는 케이스가 생기면 이 방식으론 못 잡음(Keyword 방식으로 바꿔야 함).

백엔드 prod·stage가 물리적으로 같은 OCI 인스턴스 위에 있어서, 인스턴스가 통째로 죽으면 이 둘이 **동시에** DOWN으로 잡힘 — 두 개 이상 동시에 울리면 "앱 개별 문제가 아니라 인프라 전체 문제"로 바로 구분 가능.

## UP/DOWN 이메일 조건

- **체크 주기**: 5분 (무료 플랜 고정값)
- **DOWN 판정 기준**: 1회 실패 즉시 (연속실패 대기 없음)
- **UP(복구) 알림**: 켜져 있음 — 다운 감지뿐 아니라 복구 시점도 메일로 옴
- **Alert Contact**: 동아리 이메일 계정(가입 계정 자체) — 4개 모니터 전부 연결 확인됨. PM 개인 메일은 별도로 안 붙임(동아리 계정이 이미 공유 자산이라 불필요하다고 판단)
- **알림 채널**: 이메일 + **Discord 웹훅**(PM이 추가, 2026-07-08). 앞서 "Discord/Slack/웹훅은 Team 플랜($29/월)부터"로 조사했던 것과 안 맞았는데, **실제 다운/복구 테스트(2026-07-08, 아래 참고)로 둘 다 정상 수신 확인** — 무료 플랜에서 Discord 웹훅이 된다는 뜻. 조사 당시 정보가 부정확했거나 그 사이 정책이 바뀐 것으로 보임.
- **채널별 전달 지연 차이 있음(실측)**: Discord 웹훅은 거의 즉시(초 단위) 도착, 이메일은 SMTP 릴레이·수신서버 처리 때문에 몇 분 늦게 올 수 있음 — 같은 이벤트인데 두 채널 도착 시각이 다르다고 "뭔가 잘못됐다"고 오판하지 말 것.

## 한계 — 실측 사례 (2026-07-08)

prod 컨테이너 재생성 작업 중 태그 미지정으로 약 2~3분간 실제 장애가 발생(`exec format error` 크래시 루프, 05:19:21~05:19:39 UTC경 nginx 로그로 확인). **UptimeRobot 알림 이메일이 안 왔음** — 원인 분석 결과 버그가 아니라 체크 타이밍 문제:

| 시각(UTC) | UptimeRobot 체크 |
|---|---|
| 05:17:04 | 200 정상 (장애 직전 마지막 체크) |
| 05:19:21~05:19:39 | 실제 장애 구간 |
| 05:21:49 | 200 정상 (이미 복구된 뒤 다음 체크) |

**5분 체크 간격의 빈틈에 장애가 정확히 들어가서, 체크가 단 한 번도 장애 구간에 안 걸렸음.** 5분보다 짧은 장애는 순전히 타이밍 운에 따라 UptimeRobot이 통째로 놓칠 수 있다는 뜻 — 연속실패 기준(1회 즉시로 이미 최소로 맞춰둠)과는 별개의, 더 근본적인 무료 플랜의 한계. 짧은 flapping성 장애까지 잡고 싶다면 유료 플랜(체크 간격 단축)이나 다른 도구가 필요하다는 걸 감안할 것.

**같은 한계가 #83 재부팅 복구력 테스트(2026-07-09)에서 한 번 더 실증됨.** 인스턴스를 실제로 재부팅했고 SSH 재접속까지 약 20초, 헬스체크 정상 복귀까지 1분 이내로 자동 복구됐지만(→ [`observability.md`](./observability.md#증빙-로그-2026-07-09-실측) 재부팅 복구력 증빙), 그 재부팅 시각 전후로 UptimeRobot DOWN/UP 알림이 **전혀 오지 않음**(메일함 실측 확인) — 5분 체크 틈새에 통째로 들어가서 이번에도 놓친 것. 즉 **② "서버 전체 다운 감지"는 다운이 몇 분 이상 지속되는 경우에만 유효하고, 복구가 빠른 장애(재부팅 등)는 오히려 감지될 확률이 낮다** — 이건 아이러니지만 무료 플랜 구조상 불가피한 트레이드오프이며, 지금 수준에선 받아들이기로 함.

## 정상 동작 실측 — 의도적 다운/복구 테스트 (2026-07-08)

위 사례와 별개로, prod+stage를 의도적으로 동시에 `docker compose stop`한 뒤 정상적으로 감지·복구되는지 별도 테스트함:

- **DOWN 알림**: 이메일·Discord 웹훅 둘 다 정상 수신 확인
- **prod가 stage보다 먼저 죽음(약 1분 차)**: 동시에 `stop` 명령을 내렸는데도 nginx 로그상 prod가 먼저 502로 넘어감 — Spring Boot의 graceful shutdown(SIGTERM 수신 후 진행 중 요청을 정리하는 유예 시간) 때문에 즉시 끊기지 않고 잠깐 느려지다 죽는 것으로 추정. **"동시에 내렸으니 동시에 죽었을 것"이라고 가정하면 안 됨** — 서비스마다 종료 로직에 따라 실제 다운 시점이 갈릴 수 있음.
- **UP(복구) 알림**: 컨테이너를 다시 올린 뒤, 다음 UptimeRobot 체크 사이클(5분 내)에 정상 수신 확인
