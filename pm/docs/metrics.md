# 개발 지표 — 배포 전, 개발 과정에서 확인하는 수치

> 왜 이 지표들인가: 출시(8/28) 전엔 사용자 지표(방문·전환)가 존재하지 않는다. 그래서 v1 출시 전까지는
> **개발 과정 자체에서 매주 뽑을 수 있는 수치**만 본다 — 전부 기존 산출물(pm/qa·GitHub·CD)에서
> 파생되고, 새 계측 코드가 필요 없다. 방법론은 둘을 빌린다:
> **출시 준비도**(release readiness — GWT 성공기준 체계에서 파생)와 **DORA 지표 축약판**(개발 프로세스 건강).

## A. 출시 준비도 — "만들어야 할 것이 실제로 되는가"

`pm/qa/`(성공기준·검증)에서 그대로 계산된다.

| # | 지표 | 정의 | 출처 |
|---|---|---|---|
| 1 | **GWT 검증 커버리지** | 전체 시나리오 중 `not-run`이 아닌 비율 — "우리가 실제로 확인해 본 범위" | `pm/qa/verification/*.md` 집계 |
| 2 | **GWT pass율** | 검증된 시나리오 중 pass 비율 — "확인한 범위 안에서 되는 것" | 〃 |
| 3 | **fail 이슈화율** | fail 중 GitHub 이슈로 연결된 비율 — 검증 문서가 버그 무덤이 되지 않게 | 〃 + GitHub |
| 4 | **콘텐츠 채움율** | 공개 화면에서 플레이스홀더가 남은 노드 수 — 코드가 아니라 콘텐츠가 그림자 크리티컬 패스라는 브리프의 현실 반영 (LAND-005류 시나리오가 이미 잡아준다) | GWT 시나리오 + 트리 |

**①과 ②는 반드시 쌍으로 본다.** pass율만 목표로 삼는 순간 쉬운 시나리오만 늘리는 왜곡(굿하트의 법칙)이
생긴다 — "pass율 95%"는 커버리지 30%에서는 아무 뜻이 없다. 같은 이유로 **fail 수 자체를 벌점으로 쓰지
않는다** — fail을 많이 찾는 건 검증이 일을 잘한 것이다. 나쁜 신호는 fail이 아니라 "이슈화 안 된 fail의 방치"(③)다.

## B. 프로세스 건강 — DORA 축약판 (4개 중 2개만)

| # | 지표 | 정의 | 출처 |
|---|---|---|---|
| 5 | **리드타임** | 미션 이슈 생성 → PR 머지까지 일수(중앙값) | GitHub 이슈·PR 날짜 |
| 6 | **변경 실패율** | stage 배포 중 실패·롤백 비율 | CD run 기록 |

왜 DORA 4개 중 2개만 쓰나 — **배포 빈도**는 우리 구조상 머지가 곧 배포(dev→stage CD)라 리드타임과
사실상 중복이고, **MTTR**은 장애 표본이 적어(월 0~2건) 숫자가 노이즈다. 표본이 쌓이면 그때 추가한다.

## 운영 방식

- **최신값은 루트 `README.md` 최상단 한 줄**에 노출한다(구현 현황 트리 바로 위). 갱신 규칙은 트리와 동일 —
  검증 결과가 바뀐 턴에 같이 고친다. **수치 히스토리는 이 문서에 쌓지 않는다** — 상태는 썩는다(하네스 원칙).
  추이는 주 1회 고정 GitHub 이슈에 코멘트로 남긴다(코멘트 타임라인이 곧 히스토리).
- **행동 연결**: 지표는 보고용이 아니라 다음 미션 선정 입력이다 — 커버리지 구멍(0/n 노드)과 방치된 fail이
  곧 미션 백로그 후보.
- **경계**: 이건 팀 평가 도구가 아니다. 특히 리드타임을 개인 단위로 쪼개 보지 않는다 — 이 프로젝트의
  1순위 가치는 멤버 학습이고, 지표가 학습을 압박하는 순간 주객전도다.

## 출시 후 — 전환 예고

v1 출시 뒤엔 사용자 퍼널(AARRR 축약: 유입 → 활성 → 전환[알림 신청·지원서] → 실현["사이트 보고 왔다" 비율])로
전환한다. 그때 계측할 이벤트는 새로 발명하지 않고 **criteria의 `when`에서 고른다**(예: NOTI-003 제출 성공 =
전환 이벤트) — 스펙·검증·지표가 한 줄로 꿰지게. 유입 계측 도구(GA4 / 자가호스팅 / nginx 로그 집계)는
출시 직전에 트레이드오프를 보고 결정한다.
