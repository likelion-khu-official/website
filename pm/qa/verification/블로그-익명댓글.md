## 근거 (시나리오별 상세)

### CMT-001 — pass

`frontend/src/components/blog/CommentSection.tsx`: 마운트 시 `useEffect`(25-40행)가 `getComments(postId)`를 호출하는 동안 `loadState='loading'`이라 74-80행의 펄스 박스 2개(`Array.from({length:2})`)가 보인다. 성공하면 `loadState='ready'`, 헤더는 70-72행 `댓글 {loadState==='ready' ? comments.length : initialCount}` — 로드 전엔 서버가 내려준 `initialCount`(SSR `commentCount`), 로드 후엔 실제 배열 길이로 바뀐다. 실패 시(33-35행) `loadState='error'` → 82-93행 "댓글을 불러오지 못했어요." + "다시 시도" 버튼(`handleRetry`가 `reloadIndex`를 증가시켜 effect를 재실행). 0개면 97-98행 "첫 댓글을 남겨보세요." 각 항목(100-115행)은 `comment.nickname ?? '익명'` + `formatDate(comment.createdAt)` + `whitespace-pre-wrap`의 content.

백엔드 정렬은 `CommentService.list()`(`CommentService.java:31-35`) → `commentRepository.findByPostIdAndHiddenFalseOrderByCreatedAtAsc(postId)` — 숨김 제외 + 오래된 순, then과 일치.

### CMT-002 — pass

같은 파일 121-137행: 닉네임 `maxLength={50}`, 내용 `<textarea required maxLength={300}>`, 139행 `{content.length}/300` 카운터. 142행 `disabled={submitting || !content.trim()}` — 공백뿐이면 `trim()`이 빈 문자열이라 disabled. 제출 시 51행 `setSubmitting(true)` → 145행 버튼 텍스트 `'등록 중…'`. 성공(54-60행): `createComment()` 응답을 `setComments(prev => [...prev, created])`로 배열 끝에 즉시 추가하고 `setContent('')`·`setNickname('')` — `router.refresh()`나 `location.reload()` 같은 페이지 리로드 코드가 없다.

백엔드 `CommentCreateRequest`(파일은 못 열었지만 `CommentController.create()`가 `@Valid`로 검증) → `Comment.create(post, nickname, content)`가 저장 시 닉네임 정규화(공백/빈값→null="익명" 표시로 매핑)를 처리하는 것으로 보이며, 성공 시 `CommentController.create()`(`CommentController.java:20-25`)가 `201 CREATED` + `CommentResponse`를 반환한다.

### CMT-003 — pass

`CommentSection.tsx:61-62`: `catch (err) { setSubmitError(err instanceof FeedApiError ? err.message : '댓글 작성에 실패했어요.'); }` — `content`/`nickname` state는 이 catch 블록에서 건드리지 않으므로(성공 시에만 34-35행 `finally` 이전에 초기화) 입력값이 그대로 남는다. 148행에서 `{submitError && <p className="text-red-400">{submitError}</p>}`로 폼 하단에 표시된다.

### CMT-004 — pass

`CommentController.create()`(`CommentController.java:20-25`) → `CommentService.create()`(`CommentService.java:24-29`): `postRepository.findByIdAndStatus(postId, PostStatus.PUBLISHED).orElseThrow(() -> new ResponseStatusException(HttpStatus.NOT_FOUND, "글을 찾을 수 없어요."))` — 조회 조건 자체가 PUBLISHED로 고정이라 HIDDEN·DRAFT·존재하지 않는 postId 모두 이 한 줄에서 걸러진다. `ResponseStatusException`은 `GlobalExceptionHandler`에 대응하는 `@ExceptionHandler`가 없어(전체 파일 확인) Spring 기본 에러 핸들링으로 응답 — 커스텀 `{success,message}`가 아니다.

### CMT-005 — pass

`CommentController.list()`(`CommentController.java:27-30`) → `CommentService.list()`(`CommentService.java:31-35`)는 postId 존재 여부를 확인하는 코드가 전혀 없다. `findByPostIdAndHiddenFalseOrderByCreatedAtAsc(postId)`는 JPA 파생 쿼리라, 매치되는 행이 0개면 그냥 빈 리스트를 반환한다(예외 없음) — 컨트롤러가 그대로 `200 []`을 응답.

### CMT-006 — fail

CMT-005와 같은 이유로 `CommentService.list()`(`CommentService.java:31-35`)가 게시글 상태를 전혀 확인하지 않는다 — `Post`·`PostStatus`를 아예 참조하지 않고 `postId`만으로 댓글을 조회한다. 즉 HIDDEN이든 DRAFT든, 그 글의 숫자 postId만 알면 댓글 목록이 그대로 200으로 나온다. **글 숨김이 댓글 조회에 캐스케이드되지 않는다**는 then의 주장이 코드로 확정된다 — fail.

### CMT-007 — fail

`CommentController.hide()`(`CommentController.java:34-38`): `@PatchMapping("/admin/{commentId}/hide")`이고, 클래스 레벨 `@RequestMapping("/api/posts/{postId}/comments")`(14행)와 합치면 실제 경로는 **`/api/posts/{postId}/comments/admin/{commentId}/hide`**다. 이 메서드 위 주석(32행)이 `// ── 어드민 (TODO: 인증 추가) ──` — 코드 자체가 인증 미비를 인정하고 있다.

`SecurityConfig.java`의 matcher 순서를 확인했다:
- 60행: `.requestMatchers("/api/posts/*/comments/**").permitAll()` — "피드 댓글 — 공개 읽기·작성 + 어드민 숨기기"라는 주석과 함께, `/api/posts/{postId}/comments/` 아래 전부(`**`)를 permitAll로 열어둔다. `hide()`의 실제 경로(`/api/posts/*/comments/admin/*/hide`)가 이 패턴에 정확히 매치된다.
- 72행: `.requestMatchers("/api/admin/posts/**", "/api/admin/comments/**").authenticated()` — 이 matcher가 노리는 경로는 **`/api/admin/comments/...`**(admin이 posts보다 앞)인데, 실제 컨트롤러 경로는 **`/api/posts/.../comments/admin/...`**(admin이 뒤)라 문자열이 다르다 — 이 규칙에 안 걸린다.

Spring Security는 매치되는 규칙 중 **먼저 등록된 것**을 적용하므로, 60행이 72행보다 먼저 있고 애초에 72행 자체가 다른 경로를 가리켜 매치되지도 않는다 — 결과적으로 `hide()`는 인증 없이 호출 가능하다. `CommentService.hide()`(`CommentService.java:37-42`)도 postId를 파라미터로 받지도 않고 `commentId`만으로 `commentRepository.findById(commentId)`를 조회해 바로 숨긴다 — `@PathVariable Long postId`는 컨트롤러 시그니처에는 있지만(35행) 서비스 호출에 안 넘어간다(36행 `commentService.hide(commentId)`), 즉 postId는 라우팅 매칭에만 쓰이고 검증엔 전혀 안 쓰인다. then의 두 주장(인증 없이 204, postId 미검증) 모두 코드로 확정 — fail. **이슈 미등록 — 보안 갭이라 이슈화 필요.**

### CMT-008 — fail

`CommentController.create()`·`CommentService.create()`(위 CMT-002·004 참고) 어디에도 rate limit·동일 IP/닉네임 중복 체크·스팸 필터·CAPTCHA 검증 코드가 없다. `SecurityConfig.java`에도 요청 빈도를 제한하는 필터가 없다(전체 필터 체인은 `JwtAuthenticationFilter` 하나뿐). 코드상 무제한 등록이 사실이라 fail로 판정했지만, `pm/features/블로그-익명댓글.md`의 스펙 설명대로 v1 정책 자체가 "사전 필터 없음, 문제되면 관리자가 사후 숨김"이므로 이 fail은 **정책 위반이 아니라 정책이 의도한 상태**라는 점을 note로 남긴다.
