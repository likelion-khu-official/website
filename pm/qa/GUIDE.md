# 검증 결과 작성 지침

`pm/qa/` 밑은 두 종류 문서로 나뉜다. 섞지 않는다.

- **`criteria/<관점>.yaml`** — 성공 기준. given/when/then = "이래야 됐다고 본다"는 기대 동작. 스펙이 안 바뀌면 안 바뀐다. TDD의 테스트 코드에 해당.
- **`verification/<노드>.md`** — 검증 결과. 그 기대가 지금 실제로 성립하는지 + 그렇게 판단한 근거. QA 돌릴 때마다 갱신되는 쪽. TDD의 테스트 실행 로그에 해당.

이 문서는 `verification/`을 쓸 때 규칙.

## 모든 verification 문서 상단에 그대로 넣을 안전장치 문구

> **범위**: 이 검증은 `criteria/`의 given/when/then이 코드·API·화면에서 기대대로 동작하는지 — 즉 **기능 동작 기준으로 pass/fail**만 판정한다. **디자인 품질(비주얼 QA)·보안 침투 테스트·성능·접근성은 이 검증의 범위 밖**이다. 여기서 pass라고 해서 저것들까지 검수됐다는 뜻이 아니다.

## 시나리오 하나당 적어야 하는 것

| 필드 | 의미 |
|---|---|
| 결과 | `pass` / `fail` / `not-run` |
| 근거 | 실제로 뭘 보고 판단했는지. "그럴 것이다" 금지 — 관찰한 것만 |
| 확인 방법 | `코드 정적 분석` / `실제 요청·응답 관찰` / `브라우저 조작 관찰` / `추론(원문서 when 문구 기반, 재확인 안 함)` 중 하나 |
| 확인일자 · 확인자 | 언제, 누가(팀원 실명 또는 AI 세션) |

## 원칙

- **관찰한 것만 적는다.** 확인을 안 했으면 `not-run`이다. 안 해본 걸 했다고 쓰지 않는다 — 특히 AI 세션이 코드만 읽고 "브라우저로 확인함"이라고 쓰는 건 금지. 확인 방법이 불확실하면 `추론` 등급을 쓴다.
- **fail은 이유가 한 문장으로 있어야 한다.** 기대(then)와 실제 뭐가 다른지.
- **오래된 근거는 재확인 대상으로 표시한다.** 확인일자 이후 관련 코드가 바뀌었으면 `result`를 `not-run`으로 되돌리고 "재확인 필요" 표시.
- **fail인데 이슈가 없으면 이슈화 필요라고 적는다.** 검증 문서가 버그 목록으로 방치되지 않게.

## 파일 하나의 골격

```markdown
# <노드 이름> — 검증 결과

> pm/features/<노드>.md 의 성공 기준(pm/qa/criteria/<관점>.yaml) 검증 결과.

[안전장치 문구]

## <관점 이름> 관점

| ID | 결과 | 근거 | 확인 방법 | 확인일자 |
|---|---|---|---|---|
| ...
```
