# PM — 운영 하네스

> 김우진(PM·조직장)의 운영 베이스.

## 시작 전 읽기
1. 루트 `CLAUDE.md`
2. `pm/docs/`: `learnings.md` → `brief.md`

## PM 책임
- **결정** — 무엇을·왜. 방향·범위·우선순위.
- **운영** — 일정 조율, 막힌 것 해소, 스쿼드 통합. 진행은 GitHub 이슈(로드맵).
- **콘텐츠 수급** — 사진·소개글·자료를 운영진과 협의해 확보(전담 없음). 그림자 크리티컬 패스.
- **배운 것 관리** — *다음에도 쓸 통찰*이 생기면 `pm/docs/learnings.md`에. 결정·상태는 안 쌓음.
- **미션 박스 관리** — 미션 하나 = `pm/missions/<번호>-<슬러그>/` 상자(`proposal`→`log`→`result`). 제안·진행·결과의 정제된 맥락을 쌓는다(GitHub 복붙 아님). 운영은 mission 스킬.
- **기능 트리 = 미션의 출발점·도착점** — 미션은 `pm/features/`의 한 노드를 `Target`으로 걸고(무엇·왜는 노드 `## 개요`=위키 스펙에서 파생), **발주·재발주·담당자 변경·닫기 — 트리에 적힌 사실이 바뀌는 모든 순간에** 그 노드의 상태 색·`진행 상태`와 루트 `README.md` 트리를 그 턴 안에 갱신한다(닫을 때만이 아니다 — 2026-07-15에 발주 후 갱신을 미뤄 트리가 며칠간 사실과 다르게 남은 사고가 있었다). **새 미션은 트리의 ⚪ 미착수·🔵 계획 노드에서 고른다** — 위키(스펙) ↔ 트리(현황) ↔ 미션(전진)은 한 몸. 상세는 mission 스킬.

## 무엇을 learnings에 남기나
- 남긴다: 다음에도 쓸 **통찰**(팀·제품·기술·과정에서 배운 것), 왜가 드러나게.
- 안 남긴다: 일회성 결정·기술 스택 선택·진행 상태·대화 과정. (진행은 GitHub)

## 미션 이슈 발주 (필수 2단계)
미션을 GitHub 이슈로 쏠 때 `gh issue create`는 **Projects 커스텀 필드(Team·시작일·목표일)를 안 채운다** — 이슈는 보드에 자동 추가되고 Status=Todo까지만 된다. 그래서 발주는 **항상 2단계가 한 세트**:
1. `gh issue create …` (제목·본문·라벨·어사인)
2. `pm/scripts/mission-fields.sh <issue#> <디자인|프론트|백엔드|인프라> <시작 YYYY-MM-DD> <목표 YYYY-MM-DD>` — Team·시작일·목표일 세팅
2번을 빠뜨리면 보드 "Team" 칼럼에 **No Team**으로 뜬다(2026-07-05 실제 발생). 발주했으면 반드시 2번까지.

## 성공 기준·검증 (`pm/qa/`)
`pm/qa/`는 두 갈래로 나뉜다 — 섞지 않는다.
- **`criteria/<관점>.yaml`** — 성공 기준. given/when/then(기대 동작). 스펙이 안 바뀌면 안 바뀐다. TDD의 테스트 코드에 해당.
- **`verification/<노드>.md`** — 검증 결과. 그 기대가 지금 실제로 성립하는지(pass/fail/not-run) + 근거. QA 때마다 갱신. TDD의 테스트 실행 로그에 해당.

작성 규칙(안전장치 문구·근거 형식)은 `pm/qa/GUIDE.md`. 트리 노드 문서(`pm/features/<노드>.md`)의 "성공 기준(GWT)" 섹션과 루트 `README.md`의 "검증" 칸이 이 문서들로 링크된다 — **성공기준이 "됐다"의 실질 판정 기준**이다(색은 이슈/PR 기반 진행상태, 검증은 실제 동작 확인). 현재는 **익명 방문자 관점만** 구축됐다(`pm/qa/criteria/일반사용자.yaml`) — 로그인 이후(멤버·어드민·최고관리자) 노드는 아직 GWT 커버 대상이 아니다.

**범위 한정 — 안전장치.** 이 검증은 given/when/then이 코드·API·화면에서 기대대로 동작하는지, 즉 **기능 동작만** 판정한다. 디자인 품질(비주얼 QA)·보안 침투 테스트·성능·접근성은 범위 밖 — 별도 프로세스가 필요하다. verification 문서마다 이 문구를 상단에 명시한다.

## Git — 푸시 방식
- **기본 푸시 대상은 `dev` 브랜치다. `main`은 특별한 지시가 있을 때만.** 왜 — `main` = prod = 운영 배포라, main에 꽂으면 곧장 운영에 나간다(인프라 CI/CD: dev=stage, main=prod). 그래서 main 직접 푸시는 위험하고, 특별 지시가 있을 때만 한다.
- **PR은 김우진의 지시가 없는 한 올리지 않는다 — dev에 직접 푸시한다.** PM 운영(하네스·문서·미션 박스)은 혼자 도는 작업이라 PR 게이트가 오히려 마찰. PR이 필요하면 김우진이 그때 명시한다.
- **PM 하네스 산출물(스킬·`pm/` 문서·미션 박스·`ops.md`·이 `CLAUDE.md` 등)은 전역 "*.md 커밋 제외" 규칙의 예외** — 하네스가 곧 PM 운영의 결과물이라, 일반 작업처럼 **dev에 곧바로 커밋·푸시한다**(매번 별도 지시 불필요). `main` 푸시와 코드(FE/BE/infra) 변경은 여전히 지시·팀 규칙을 따른다.
- **`pm/` 아래 무언가 수정·생성되면 그 턴 안에 바로 dev로 커밋·푸시한다 — 서 있는(standing) 규칙, 매번 "푸시해" 지시를 기다리지 않는다.** (2026-07-14 김우진 재확인) PII(`pm/applicants/` 등 실명·개인정보)만 예외 — 이건 커밋하지 않고 사람에게 알린다.
- 비밀은 절대 커밋 금지 — `*.local`·비밀번호는 레포 밖에.
