# 배운 것 (Learnings)

> 이 프로젝트를 하며 얻은 *다음에도 쓸 통찰*을 쌓는다.
> 결정 기록·진행 상태가 아니라 **배운 것**. 새 통찰은 맨 위에, 한 줄로, 왜가 드러나게.

---

## 인프라 · CI/CD
- **certbot standalone 발급 시 Docker nginx를 먼저 내려야 한다.** certbot이 80 포트로 인증하는데 nginx 컨테이너가 잡고 있으면 실패 — `docker compose stop nginx` 후 발급, 완료 후 재기동. 두 도메인을 한 cert에 묶으려면 `-d domain1 -d domain2`로 한 번에 발급하면 첫 번째 도메인 경로에 저장된다.
- **서버 로컬 DNS 캐시가 느려도 글로벌 전파는 이미 됐을 수 있다.** `nslookup`이 실패해도 `dig @8.8.8.8`로 확인하면 실제 전파 여부를 알 수 있다 — certbot은 글로벌 DNS 기준으로 검증하므로 로컬 캐시와 무관하게 발급 가능.
- **compose `depends_on`이 있으면 named service 기동 시 dependency 이미지 태그도 재평가된다.** `docker compose up -d nginx`가 nginx의 `depends_on`인 backend-stage/prod의 이미지 태그가 바뀌었는지 확인하고, 달라졌으면 재생성한다. 동적 리졸버를 쓰는 nginx처럼 실제로 의존성이 없는 경우엔 `depends_on`을 아예 빼는 게 낫다.
- **compose에서 여러 서비스가 같은 변수를 공유하면 하나의 배포가 다른 환경을 오염시킨다.** `${BACKEND_TAG:-stage-latest}` / `${BACKEND_TAG:-prod-latest}` 처럼 fallback만 다르고 변수는 같으면, BACKEND_TAG가 세팅된 순간 fallback이 무시되고 두 서비스 모두 같은 값을 쓴다. 서비스마다 독립 변수(`STAGE_TAG` / `PROD_TAG`)를 써야 한 배포가 다른 쪽 컨테이너를 건드리지 않는다.
- **nginx를 compose에 넣으면 동시 운영 시 포트 충돌로 결국 앞단 레이어가 필요해진다.** 두 스택을 동시에 띄우는 게 기본이라면 native nginx가 더 단순하다.
- **롤백용 이전 태그를 `/tmp`에 두면 재부팅 시 유실된다.** deploy path 안에 두어야 영속성이 보장된다.
- **이미지 태그에 환경 prefix(`test-`/`prod-`)를 붙이면 하나의 레지스트리에서 두 환경이 독립적으로 관리된다.** `latest`만 쓰면 롤백 포인트가 없다.
- **브랜치를 환경과 1:1로 대응시키면(`dev`→test, `main`→prod) CI/CD 분기 로직이 브랜치 이름 하나로 결정된다.** 환경별 워크플로를 별도 파일로 나누는 것보다 단순하다.

## 팀 · 운영
- **학습은 페어여야 성립한다.** 1명은 "일하는 자리", 2명은 "배우는 자리" — 혼자는 막혀도 가르칠 짝이 없다.
- **일은 단계가 아니라 오너십+목표로 준다.** 결정을 멤버에게 남기는 게 곧 학습을 남기는 것. 단계까지 주면 받아쓰기가 되고, 그건 Claude가 더 잘한다.
- **핸드오프(던지기)는 안티패턴.** 계약을 먼저 정하고 병렬로 가면 통합이 싸다. "디자인 끝→FE→BE" 릴레이는 틈에서 깨진다.
- **PM은 팀 사이 유일한 채널이 되면 안 된다.** 인터페이스는 팀끼리 직접, PM은 우선순위·통합·막힘 해소. 중계자가 되면 전화게임·병목.
- **발산은 sync, 수렴은 텍스트.** 열린 결정·신생팀 정렬은 통화로, 정해진 것(계약·이슈)은 텍스트로.
- **PM이 "어떻게"까지 주면 학습이 죽는다 — 결정엔 "왜"를 붙여라.** "X 써"(받아쓰기)가 아니라 "이 제약이라 X"(근거→학습). 프로젝트 1순위가 멤버 학습이면 미션은 *무엇을·왜·제약*까지만 주고 *어떻게*는 비운다. 답을 주지 말고 제약을 줘라.
- **Walking skeleton도 전 팀 동시는 아니다.** 남의 산출물에 의존하는 팀(예: 배포는 띄울 앱이 있어야)은 그게 무르익은 *다음* 사이클이 제 타이밍 — 억지로 같은 줄에 넣으면 대기·빈작업이 된다. 한 줄은 "되는 곳까지"만 솔직히 긋는다.

## 제품
- **도구를 서비스 성격에 맞춰라.** 읽기 중심 콘텐츠/홍보 사이트엔 무거운 백엔드가 오버킬.
- **상시 모집이 아니면 사이트는 두 모드로 생각한다.** 평상시=인지·관심, 모집기간=전환. 한 모드만 만들면 평상시 방문이 휘발.
- **타겟은 좁힐수록 날카로워진다.** 카피·디자인·IA가 선명해진다.
- **BE는 비주얼을 안 보지만 거친 흐름은 본다.** 데이터는 브리프에서, 세부 기능은 와이어/IA에서.

## 기술 · 인프라
- **Docker 파일 마운트는 대상 경로가 컨테이너에 없으면 디렉터리를 만든다.** `./data/app.db:/app/data/app.db`처럼 파일 단위 마운트 시, 컨테이너 이미지에 `/app/data/` 디렉터리가 없으면 Docker가 마운트 포인트를 파일이 아닌 디렉터리로 생성한다 — SQLite가 열려는 경로가 디렉터리가 돼 CANTOPEN. 파일 하나를 마운트하려면 디렉터리 단위로 올리고(`./data:/app/data`) 파일명은 앱에서 결정하게 하는 게 안전하다.
- **Hibernate 6.x에서 `spring.jpa.database-platform`은 dialect로 전달되지 않는다.** `spring.jpa.properties.hibernate.dialect`로 직접 설정해야 한다. `database-platform`은 Spring Boot 자동 구성이 처리하던 방식인데 Hibernate 6.6.x에서 동작이 바뀌었다.
- **Walking skeleton의 버그는 순서대로 숨는다.** 앞 레이어 에러가 뒤 레이어를 가린다 — 플랫폼 불일치(arm64), 엔드포인트 없음(actuator), 앱 기동 실패(DB env), 볼륨 버그, Dialect 설정 순으로 5개가 차례로 드러났다. 파이프라인 첫 성공까지는 각 단계가 독립 실패 포인트다.
- **오래 사는 브랜치(main·dev)엔 squash·rebase 머지 금지.** 계속 사는 브랜치는 공통 조상이 함께 전진해야 다음 머지가 깔끔하고, 브랜치 간 진행 추적·롤백이 된다. squash/rebase는 새 SHA를 만들어 히스토리를 갈라놓고 공통 조상이 못 움직여 → 머지 충돌이 반복된다(실제로 dev→main에서 터짐). 버려지는 브랜치(feat·doc)는 squash해도 무방하나 설정이 레포 단위라, "히스토리가 지저분해짐 vs 추적 가능"의 트레이드오프에서 *추적*을 택했다(일반 머지만 허용).
- **프론트↔백엔드는 서버사이드 rewrite 프록시로 한 도메인에 통합한다.** 브라우저는 항상 한 도메인으로만 요청하고, 프론트 서버(Next.js)가 그걸 백엔드로 프록시. 효과 둘 — ① CORS 설정이 아예 불필요(서버↔서버 통신이라 브라우저 CORS 정책이 적용 안 됨), ② 실제 백엔드 주소가 외부에 노출 안 됨(은닉). 서브도메인 분리(api.…)보다 단순·안전 → 그래서 이 방식을 택함.
- **stage/prod를 나누는 진짜 목적은 "환경이 둘"이 아니라 데이터 격리다.** 테스트 요청이 운영 DB를 오염시키지 않게 dev=테스트·main=운영으로 가른다. 분리 자체가 목적이 아니라 *데이터 혼동 방지*가 목적 — 그래서 프론트도 테스트는 테스트 백엔드로 보낸다.
- **이미 떠 있는 환경엔 로컬에 복제하지 말고 직접 붙어라.** 프론트는 로컬에 백엔드를 띄우지 않고 staging 서버에 바로 연결해 개발 — 셋업 마찰을 없애고 통합을 늘 실물로 검증. 환경이 상시 떠 있다는 전제가 깔리면 "각자 로컬에 남의 서버 재현"이 사라진다.
- **왜 SQLite인가 — DB 선택은 트래픽 규모가 아니라 읽기/쓰기 비율로 먼저 가른다.** 우리 홈피는 정적 콘텐츠(소개·멤버·활동·블로그) 위주라 *읽기가 대부분, 쓰기는 드물다*. 그래서 DB 역할 자체가 작고, SQLite의 single-writer 제한이 안 아프다 — 거기에 단일 무료 노드에 별도 DB를 안 늘려 운영이 단순해지는 이득까지. 유일한 경계는 *쓰기 폭주 한 곳*(모집 신청 버스트), 그것도 동아리 규모면 SQLite가 가볍게 처리. "규모 작으니까"가 아니라 "성격이 읽기 위주니까"가 진짜 이유다.
- **SQLite의 "한 번에 한 쓰기"는 직렬이지 저속이 아니다 — 그리고 앱 인스턴스 1개를 전제한다.** 병목은 창구 수가 아니라 트랜잭션 길이(작은 INSERT는 1ms라 직렬이어도 초당 수백). 단 한 `.db`를 여러 프로세스가 공유하면 깨지므로 *BE 앱 인스턴스는 1개*가 불변식 — 수평 확장(replica)이 필요해지는 순간이 곧 Postgres로 옮길 신호.
- **무료 클라우드 티어는 예고 없이 깎인다.** 오라클이 2026-06-15 Always Free 한도를 공지 없이 절반(4 OCPU/24GB→2 OCPU/12GB)으로 줄인 게 실증 — "현재 무료"이지 "영구 무료"가 아니다. 무료티어 위에 올릴 땐 한도의 절반만 써 여유를 두고, 깨졌을 때의 fallback(유료 전환 비용)을 미리 숫자로 알아둔다.
- **DB를 앱과 같은 노드에 올리면 백업이 필수가 된다.** 단일 노드 동거는 단순하지만 앱과 DB가 운명을 공유 — 노드가 죽으면 데이터도 같이 멈춘다. "있으면 좋은" 백업이 "없으면 안 되는" 백업으로 바뀐다.

## 디자인
- **디자인은 코드 홈이 없어 가장 애매하다.** "done"을 *PM 승인*으로 정의해 주관적 모호함을 죽인다.
- **디자인은 크리티컬 패스 — "거칠게 빨리"가 병목을 막는다.** 다른 팀을 여는 산출물(IA·와이어)을 먼저, 완성도는 나중.
- **모르는 영역은 지시하지 말고 결과만 판단한다.** PM이 디자인을 art-direct하면 망친다.
- **와이어프레임 평가는 카피가 아니라 구조·흐름·상태로 한다.** 플레이스홀더("Sub text")는 와이어의 정상 상태 — 와이어가 답할 건 "메시지 들어갈 *자리*가 있나"지 "메시지 내용"이 아니다. 콘텐츠를 구조 결함으로 착각하면 디자인팀에 헛 피드백이 간다.

## 하네스 · 문서
- **의도적으로 깃에 안 올리는 운영 정보는 "찾을 단일 출처"가 사라진다.** 스테이징 URL을 obscurity 때문에 깃허브에 안 올리니, 정작 우리도 "어딜 봐야 알지?"가 됐다. 은닉과 발견가능성은 트레이드오프 — 둘 다 잡으려면 *레포 밖 비공개 단일 출처*(gitignore된 로컬·비밀번호 관리자)를 정해 둬야 한다. 안 그러면 카톡에만 흩어진다.
- **맥락 축적 ≠ 대화 로깅.** 결정에 이른 과정 말고 *배운 것*을 남긴다.
- **시스템 프롬프트(CLAUDE.md)엔 상태를 넣지 않는다.** 상태는 썩는다 — 진행은 GitHub로.
- **계약은 문서가 아니라 코드(`shared`)로.** 문서로 베끼면 drift한다.
- **하네스는 맥락을 잃지 않는 선에서 최대한 작게.** 안 쓰는 문서·중복 규칙은 노이즈로 죽는다.
- **진짜 AI-native = 운영 작업을 스킬로 만들어 에이전트가 직접 굴리게 하는 것.** "전원이 Claude를 쓴다"만으론 AI-assisted일 뿐 — 이슈·보드 관리 같은 걸 스킬화해야 근육이 붙는다.
- **공통 운영 루프(첫날·이슈사이클·막힘·PR)는 각 역할 문서에 흩지 말고 한 곳에 모아 링크한다.** 흩으면 중복·drift, 한 곳이면 단일 진실.
