## 근거 (시나리오별 상세)

### WRITE-001 — pass

`frontend/src/app/blog/write/page.tsx:4-7`: `metadata.robots = { index: false, follow: false }`가 라우트 전체(토큰 유무와 무관)에 고정 적용된다. `WriteForm.tsx:21`: `token`이 없으면 `tokenState` 초기값이 `'no-token'`이고, 121-128행에서 `<NoticeScreen title="접근할 수 없어요" description="글 작성 링크가 필요해요. 운영진에게 받은 링크로 다시 접속해 주세요." />`만 렌더 — 폼(제목·본문 input)은 이 분기에서 아예 렌더되지 않는다.

### WRITE-002 — pass

`WriteForm.tsx:34-72`의 `useEffect`가 `getTokenStatus(token)`(`feedApi.ts:70-75`, `GET /api/feed/tokens/{token}`)을 호출한다. 상태는 `MagicLinkTokenService.checkStatus()`(`MagicLinkTokenService.java:37-47`)가 결정: `found.isUsed()`면 `invalid(...,"USED")`, `found.isExpired()`면 `invalid(...,"EXPIRED")`, 아니면 `valid(...)`. 토큰 자체가 없는 경우엔 `findByToken()`이 `MagicLinkTokenNotFoundException`을 던지고, FE는 `catch`(42-45행)에서 `tokenState='error'`로 처리한다.

화면 분기(`WriteForm.tsx:117-155`): `checking`→"확인하고 있어요…", `expired`→"링크가 만료됐어요…", `used`→"이미 사용된 링크예요…", `error`→"확인하지 못했어요…", `valid`→"{authorName} 님으로 작성해요." + 폼(168-169행). then의 다섯 분기 모두 정확히 일치한다.

### WRITE-003 — pass

`WriteForm.tsx:88-91`: `if (!title.trim() || !content.trim()) { setSubmitError('제목과 본문을 모두 입력해 주세요.'); return; }`. 174-178행: `preview` state 토글 버튼(`미리보기` ↔ `편집으로 돌아가기`), 183행: `{title || '(제목 없음)'}`. 임시저장은 75-83행 `useEffect`가 `title`/`content`/`thumbnailUrl`이 바뀔 때마다 `localStorage.setItem(draftKey(token), ...)`, 34-65행 마운트 시 `localStorage.getItem(draftKey(token))`으로 복원, 제출 성공 후 105-108행 `localStorage.removeItem(draftKey(token))`으로 삭제 — then의 세 조건 모두 코드에 그대로 있다.

### WRITE-004 — pass

프론트: `ImageUploader.tsx:6-7` `ALLOWED_TYPES`(jpeg/png/webp/gif), `MAX_SIZE = 5*1024*1024`. `validate()`(34-42행)가 형식·용량을 클라이언트에서 먼저 거른다. 업로드 중엔 108-112행 진행률 바(`style={{width:`${progress}%`}}`)와 상위 `WriteForm.tsx:236` `disabled={submitting || imageUploading}` + 239행 버튼 텍스트 `'이미지 업로드 중…'`. 실패 시 148행 `{state==='error' && <p className="text-red-400">{error}</p>}`, 113-119행 `삭제` 버튼(`handleRemove`).

서버: `backend/src/main/resources/application.yml:29-32`에 `spring.servlet.multipart.max-file-size: 5MB`/`max-request-size: 5MB`가 명시돼 있고, `GlobalExceptionHandler.handleMaxUploadSizeExceeded()`(66-71행)가 `MaxUploadSizeExceededException`을 `413 PAYLOAD_TOO_LARGE`로 응답한다 — "서버도 413"이 코드로 확인된다.

### WRITE-005 — pass

`PostService.createPost()`(`PostService.java:26-33`): `Post.create(...)`가 `p.status = PostStatus.DRAFT`(`Post.java:56`)로 고정 생성한다 — 공개 목록·상세는 PUBLISHED만 조회하므로(BLOG-L01·BLOG-D03) DRAFT는 노출 안 됨. `MagicLinkTokenService.consume()`(`MagicLinkTokenService.java:52-62`)이 `found.markUsed()`로 토큰을 1회 소모한다. FE `WriteForm.tsx:157-163`: 제출 성공(`submitted=true`) 시 "글이 등록됐어요 / 이 링크는 1회용이라…" 안내.

### WRITE-006 — pass

`PostController.create()`(`PostController.java:23-28`)는 `@RequestHeader("X-Magic-Token") String token`이 필수라, 헤더 자체가 없으면 Spring이 `MissingRequestHeaderException`을 던지고 `GlobalExceptionHandler`에 대응 핸들러가 없어 Spring 기본 400 에러로 응답한다. 토큰이 있지만 무효한 경우는 `MagicLinkTokenService.consume()`이: 없는 토큰 → `MagicLinkTokenNotFoundException` → `GlobalExceptionHandler.handleTokenNotFound()`(45-50행) → `404 {success:false,message}`. 사용됨/만료 → `MagicLinkTokenAlreadyUsedException`/`MagicLinkTokenExpiredException` → `handleTokenInvalid()`(52-57행) → `410 GONE {success:false,message}`. then의 세 조건 모두 일치한다.

### WRITE-007 — fail

`MagicLinkTokenController.issue()`(`MagicLinkTokenController.java:23-27`): `@PostMapping` + `@Valid @RequestBody MagicLinkTokenIssueRequest` — 인증 관련 애노테이션이 전혀 없다. `MagicLinkTokenIssueRequest`(`dto/MagicLinkTokenIssueRequest.java`)는 `@NotBlank private String authorName;` 필드 하나뿐이다. `SecurityConfig.java:49-51`: `.requestMatchers("/api/feed/tokens/**").permitAll()` 위에 명시적 주석 — "매직링크 토큰 — 운영진 인증 붙기 전까지 임시 공개 / TODO: 운영진 인증 도입되면 발급(POST)은 운영진 전용으로 좁히기". 즉 익명이 authorName 문자열 하나만 보내면 누구 이름으로든 토큰을 무제한 발급받을 수 있다는 게 코드·주석 양쪽에서 확정된다. 다만 발급받은 토큰으로 만든 글은 WRITE-005에서 확인했듯 DRAFT로만 생성돼 공개 노출은 어드민 게시 게이트 뒤에 있다.

### WRITE-008 — fail

`FeedImageController`(경로 확인용으로 이름만 봤고 실제 검증은 `FeedImageService`) + `SecurityConfig.java:52-54`: `.requestMatchers("/api/feed/images/**").permitAll()` — 주석 "피드 이미지 업로드 — 매직링크 글쓰기 흐름 자체가 비인증이라 임시 공개 / TODO: 남용 방지용 인증/레이트리밋 필요해지면 여기에 추가". `FeedImageService.validate()`(`FeedImageService.java:30-38`)는 `file.getContentType()`이 `ALLOWED_CONTENT_TYPES`(jpeg/png/webp/gif) 안에 있는지만 검사한다 — 이건 클라이언트가 요청에 실어 보내는 `Content-Type` 헤더 문자열을 그대로 믿는 것이라, 실제 파일 바이트(매직넘버)를 열어 이미지인지 확인하는 코드가 없다. rate limit 코드도 이 서비스·컨트롤러·`SecurityConfig` 어디에도 없다. then의 두 조건(토큰 없이 업로드 가능, Content-Type 선언만 검사) 모두 확정.
