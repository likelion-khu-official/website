# R (research) — 고치기 전에 이해한다

목표: RP 게이트에서 "이렇게 하겠다"를 자신 있게 말할 만큼 파악. 아직 코드를 바꾸지 않는다.

## 렌즈
- **Done부터 거꾸로.** 미션의 `Done` 각 항목이 참이 되려면 무엇을 만져야 하나. 그게 조사 범위를 정한다(그 밖은 안 판다).
- **계약을 먼저 본다.** FE↔BE가 걸리면 `shared/`의 타입이 진실이다(문서 아님). 내 변경이 계약을 건드리나?
- **주변 코드의 관례를 읽는다.** 이 레포/이 디렉터리는 이미 어떤 패턴을 쓰나. 새 패턴을 들이기 전에 기존 걸 따를 수 있는지 본다(mindset: 추상화는 2번째 중복에서).
- **Notes의 경계·접점.** 다른 팀 산출물에 의존하나? 하지 말라는 게 있나? 블록 요인은 여기서 발견해 일찍 ask-pm.
- **가정을 표시한다.** 확실치 않은 건 "가정"으로 명시 — RP 게이트에서 사람이 바로잡게.

## 산출
RP 게이트에 낼 3줄: (1) 알아낸 핵심, (2) 접근 + Done 충족 방식, (3) 가정/불확실.

## 함정
- 미션과 무관한 코드까지 훑느라 시간 태우기 → Done이 가리키는 데만.
- 계약을 문서에서 유추 → `shared/` 코드를 봐라.
