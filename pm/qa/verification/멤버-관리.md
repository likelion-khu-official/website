## 근거

### ADMMEM-001 — fail — 트리 상태("미구현")보다 실제로는 진전돼 있음

`pm/features/멤버-관리.md`는 "멤버 계정 관리 흐름은 미구현"이라고 되어 있다. 하지만 `MemberController.java`를 확인한 결과 이미 `POST /api/admin/members`(등록, 28-36행)·`PATCH /api/admin/members/{id}`(수정, 38-46행)가 존재하고 둘 다 `@PreAuthorize("hasRole('SUPER_ADMIN')")`로 보호돼 있다. `gh pr diff 121`도 확인했는데 `POST /api/admin/members/{id}/password/reset`(비밀번호 초기화, `@PreAuthorize("hasAnyRole('ADMIN','SUPER_ADMIN')")`)까지 추가한다 — 스펙의 "등록·정보수정·비밀번호 초기화" 세 가지 중 등록·수정은 dev에 이미 있고 비번 초기화는 PR #121에 있다. 오프보딩(소프트 딜리트)만 코드에서 못 찾았다.

**MYPOST-001과 반대 방향의 문서-코드 불일치** — 저긴 트리가 진도를 과장했고, 여긴 트리가 진도를 과소평가하고 있다. fail로 분류한 건 "given(문서 서술)이 실제와 다르다"는 같은 기준을 적용한 것 — `pm/features/멤버-관리.md`의 "진행 상태" 텍스트와 상태색(현재 🔵 계획) 갱신이 필요해 보인다. 익명 경계(인증 없이 두 API 호출 시 401) 자체는 정상 동작한다.
