# feed — 멋사 경희 진행 피드를 한 편 쓴다

부정기 발행. 우진님이 꼴릴 때 돌린다. **"지난 호 이후 그동안 바뀐 것"** 을 diff해서, 외부에 보여줄 **자극용 쇼케이스** 한 편으로 응축한다.

## 이 글의 정체 — 헷갈리면 다 틀어진다
- **독자는 바깥사람.** 신입 지원자·타 동아리·기업. 우리 맥락 모르고 봐도 읽혀야 한다.
- **독자가 궁금한 건 "무슨 일이 있었나"지 "누가 했나"가 아니다.** → 그래서 **글은 팀이 아니라 주제로 짠다**(이게 이 스킬의 심장 — §4).
- **목적은 자극.** "이 조직 일하네"를 보여준다. 회고·배운 것 정리가 **아니다**.
- **보여줄 인상 = 전문 제품팀.** AI-native하고 agile한 팀. *학생 학습 동아리 냄새가 나면 실패다.*
- **주인공은 제품 만드는 6명.** **PM(김우진)은 무대 뒤 — 안 보이는 게 정상.**

> **한 문장 원칙:** 데이터는 *팀으로 모으고*, 글은 *주제로 쓴다*. 팀은 주제 속에 녹는다.

---

# 1. 커서 — "그동안"이 어디부터인가

부정기라 "이번 주"라는 창이 없다. **마지막 발행 지점부터 지금까지**를 본다.

1. `pm/feed/` 에서 가장 큰 번호로 시작하는 파일(`NNN-*.md`)을 읽는다.
   - 있으면 → frontmatter `cursor`(ISO 시각)가 **이번 호의 시작선**. 다음 번호 = 최신+1.
   - 없으면(첫 호) → 시작선 = 프로젝트 시작 무렵(직전 ~14일 또는 첫 커밋). 번호 = 1.
2. 끝선 = 지금 시각.
3. 이번 호가 보는 창 = **(이전 cursor, 지금]**. 이 창 안의 변화만 긁는다.

> 각 발행물이 다음 발행의 시작선을 남긴다 — 그래서 부정기여도 빠짐·중복이 없다.

# 2. 무엇을 긁나 — *팀으로* 모은다 (탐색 대상)

org = `likelion-khu-official/website`. 인증 prefix는 프로젝트 CLAUDE.md(레포 밖)의 `GH_HOST=… GH_TOKEN=…`. **토큰을 명령에 하드코딩하지 말고** prefix 변수로 부른다.

> 모으는 단위는 팀이다 — 빠짐없이 긁고 PM을 걸러내려면 팀·사람으로 정리해야 한다. **단, 이건 재료 수집일 뿐. 글의 구조가 아니다(§4에서 주제로 다시 짠다).**

긁는 순서 — **이슈 제목만 보면 허접해진다. 살은 PR·커밋·코멘트·미션 result에 있다.**

- **A. 머지된 PR (가장 강한 신호 = 실제로 출시된 것).** 창 안에 머지된 것.
  `gh pr list --repo … --state merged --json number,title,author,mergedAt,labels` → `mergedAt`로 필터.
- **B. 커밋 (세부 작업).** `gh api 'repos/likelion-khu-official/website/commits?since=<ISO>&until=<ISO>&per_page=100'`.
- **C. 이슈 상태 변화.** `--state all`, `updatedAt`이 창 안. 무엇이 열리고·움직이고·닫혔나.
- **D. 이슈·PR 코멘트 (결정·핑퐁).** 드물지만 있으면 금. 창 안 코멘트만.
- **E. 디자인 (GitHub 흔적 0).** `pm/.claude/skills/figma-read/designs/*/digest.md`의 `figma_last_modified`가 창 안에 갱신됐으면 그게 디자인 진척. digest 본문이 "무엇이 나왔나"를 말해준다. 신호 부족하면 PM이 한 줄 채울 자리로 비워 표시.
- **F. 번역·맥락 사전 (본문 아님 — 읽기 전용).** 커밋 `fix(cd): arm64`를 사람 말로 옮기고, "이게 왜 한 일인지"를 알려면:
  - `pm/missions/<n>-<slug>/result.md`·`proposal.md` — 닫힌 미션의 산출물·결정·왜. **두께의 진짜 원천**(스택을 왜 골랐나 등은 여기 다 있다).
  - `pm/docs/brief.md` — 사이트가 뭔지(고정 안내 줄·orient용).
  - ⚠️ 미션 박스의 PM 결정·log는 *번역에만* 쓰고 본문에 옮기지 않는다.

# 4. ★ 모으기는 팀별, 쓰기는 주제별 — 팀을 녹인다 (이 스킬의 심장)

§2에서 팀으로 모은 재료를, 여기서 **독자가 궁금한 "무슨 일이 있었나"로 다시 짠다.** 팀 이름은 헤더에서 사라지고, 일은 주제 안으로 녹는다.

## 4a. 주제를 뽑는다
창 안 변화 전부를 읽고, 독자 관점의 **사건/결과 2~4개**로 묶는다. "팀"이 아니라 "이번에 무슨 일이 일어났나".
- ❌ 헤더가 팀: `## 백엔드` `## 인프라`
- ✅ 헤더가 사건: `## 무엇으로 만들지 정했어요` `## 코드를 올리면 바로 인터넷에 떠요`
- 주제 헤더도 제목과 같은 톤 — **~해요체, 구체 명사, 독자 언어.** 약어·내부표현 금지.

## 4b. 팀을 녹인다
한 주제에 여러 팀이 기여했으면 **한 이야기로 엮어** 경계를 지운다.
- 예: '무엇으로 만들지' = 프론트(Next.js) + 백엔드(Java/Spring) + 둘이 맞춘 데이터 계약 → *"화면 쪽은 …, 서버 쪽은 …, 둘이 주고받을 형태도 미리 맞췄어요"* 한 문단.
- **테스트:** 어느 문장이 어느 팀인지 *티가 나면 실패.* "프론트엔드팀이 …" "백엔드팀은 …" 식으로 팀을 주어로 세우지 마라. 일·결과를 주어로.
- 주제 하나에 한 팀만 기여했으면 그 팀만 녹여도 된다(예: 화면 설계는 디자인뿐). **억지 혼합 금지** — 핵심은 *헤더가 팀이 아니라 사건*이라는 것.

## 4c. 제목을 응축한다
H1 제목은 톱기사 하나가 아니다. **주제 2~4개를 관통하는 한 줄로 재구성.** ~해요체, **구체 명사**(추상·은유 금지 — "'보는 것'에서 '쓰는 것'으로" ❌, "화면 설계가 나오고, 서버가 떴어요" ✅).

## 4d. 두께 — 허접 금지
주제 한 칸은 **3~5문장**, 구체로 채운다. 미션 result(§2-F)에서 *왜 그렇게 정했는지*까지 길어 올린다. 이슈 제목만 베끼면 허접해진다. (참고 두께는 §8 골든 예시.)

# 5. 글의 모양 — 우아한 마크다운 (절제된 편집 레이아웃)

예쁘게 = 장식이 아니라 **절제·여백·일관.** 이모지·과한 기호 금지. 아래 골격을 그대로 따른다.

```markdown
---
edition: <N>
since: <YYYY-MM-DD>
until: <YYYY-MM-DD>
cursor: <ISO8601>
generated: <YYYY-MM-DD>
source: likelion-khu-official/website
---

> 멋쟁이사자처럼 경희대가 만드는 첫 공식 홈페이지        ← 고정 안내(매 호 동일, 인용구로 조용히)

**#<N>** · <YYYY.M.D – M.D>                            ← 호수·기간 kicker (작게)

---

# <응축 제목>                                           ← H1, §4c

<리드 한 문단>                                          ← 주제들을 한 번에 꿰는 도입. 구체 명사. 2~4문장.

## <주제 1>                                             ← §4 사건 헤더
<3~5문장. 팀 녹임.>

## <주제 2>
<…>

## <주제 3>
<…>

---

**다음** — <곧 나올 것 한 줄. 설렘 유발.>
```

레이아웃 규칙:
- **인용구(`>`)로 고정 안내** — 회색 톤으로 조용히 깔린다. 매 호 동일.
- **`**#N** · 날짜`** kicker — 작은 식별 줄. 그 아래 `---`로 머리글과 본문을 가른다.
- 본문 끝에 다시 `---`, 그 아래 **`다음`** 을 한 줄 푸터로. 큰 H2 안 쓴다(꼬리는 가볍게).
- 주제는 `##`. 각 주제 본문은 **흐르는 문단**으로(단일 줄바꿈은 GitHub이 한 문단으로 합치니, 문장을 이어 자연스러운 산문으로). 불릿 나열은 피한다 — 피드는 읽히는 글이지 목록이 아니다.
- 이모지·굵게 도배 금지. 강조는 `다음` 라벨 정도만.

# 6. 문체 — 안 지키면 학생 동아리 글이 된다

- **토스피드 톤.** 짧고 담백, 과장 없이, **~해요체**. 신문체("~했다") 금지.
- **팀을 주어로 세우지 마라(§4b).** "백엔드팀이 ~했어요" ❌ → 일·결과를 주어로.
- **제품팀처럼 — 헤지를 걷어라.** "~하려고요"·"알아보는 중"·"고민 중" ❌ → **결정·완료·범위**로. 진짜 확정만 "정했어요/확정". 진행 중은 망설임 대신 **시한**으로("곧 하나로 정해요").
- **학습 냄새 금지.** "배웠어요"·"여정"·"성장" ❌ → "정했어요"·"붙었어요". *AI-native라 적지 마라 — 한 호의 분량·속도가 증명한다.*
- **친절하게.** 추상·은유 금지, 실제 명사. 약어 풀어쓰기. **이슈번호(#16)·링크·이미지 없음** — 그냥 읽히게.
- **쉽게.** "엔드포인트 구현" ❌ → "메일을 저장하는 API". 안 보는 사람도 알게.
- **솔직하게.** 부풀리지 마라. 느슨한 호를 "엄청난 진전"으로 포장하면 다음부터 아무도 안 믿는다.

## 발행을 말리는 판단
긁고 나서 **창 안 변화가 빈약하면(제품 PR·커밋 거의 없음) 발행을 권하지 않는다.** "변화가 적어요, 며칠 더 쌓고 돌리죠"라고 PM에게 말한다. 거짓 발행이 제일 나쁘다.

# 화면 설계가 나오고, 서버가 떴어요

이번엔 홈페이지를 만들 바탕을 깔았어요. 어떤 화면이 될지 전부 그려졌고, 그걸 무엇으로 만들지 정했고, 만든 걸 띄울 서버까지 준비됐어요. 아이디어만 있던 사이트가 이제 진짜로 만들기 시작할 준비를 끝냈어요.

## 어떤 홈페이지가 될지 그려졌어요
홈페이지의 모든 화면을 설계했어요. 소개·프로젝트·운영진·활동·블로그·모집까지 6개 페이지를 다 그렸어요. … 운영진 페이지는 두 가지 안을 그려두고 곧 하나로 정해요. 모집 알림 버튼은 평소엔 작은 종으로 떠 있다가 갖다 대면 '알림 받기'로 펼쳐지게 움직임까지 정했어요.

## 무엇으로 만들지 정했어요
이 화면들을 실제로 만들 도구를 정했어요. 화면 쪽은 Next.js로, 서버 쪽은 Java와 Spring으로 — 팀이 가장 빠르게 움직일 수 있는 선택이에요. 화면과 서버가 주고받을 데이터 형태도 미리 한곳에 맞춰뒀어요.

## 코드를 올리면 바로 인터넷에 떠요
그렇게 만든 걸 띄울 자리까지 마련했어요. 24시간 도는 실제 서버를 올리고, 코드를 올리기만 하면 자동으로 배포되는 길을 깔았어요. 이 서버는 월 0원이에요.

# 품질 루브릭 (저장 전 자가검증)
- **팀 경계 테스트 (가장 중요)** — 헤더가 팀 이름인가? 어느 문단이 어느 팀인지 티 나나? 그러면 §4로 주제화·녹이기 다시.
