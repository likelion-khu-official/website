## 2026-07-05 — 멤버 카드 스펙 확정 후 발주

멤버 탭에 뜰 "멤버 한 명"의 정보를 확정하고 API로 발주. 결정과 왜:

- **설계 원칙 = "안 썩는 필드만."** 대표작·기술스택·외부링크는 뺐다. 왜 — 사람은 크는데 박제된 대표작은 1년 뒤 짐덩이고, 나간 사람은 갱신도 안 함. 남긴 건 시간이 지나도 참인 것(실명·역할·기수)뿐. 덕분에 멤버 탭은 기수가 쌓일수록 좋아지고 유지보수는 0.
- **사진 대신/겸용 랜덤 이모지.** 생성 시 서버가 무난한 이모지 하나를 랜덤 배정(고정), 본인이 사진이나 다른 이모지로 교체 가능. 왜 — 사진 수집이 콘텐츠 수급의 최대 병목인데, 이모지 기본값이면 **카드가 절대 안 비고** 수집 마찰이 0. 랜덤이라 "바꿔야지" 하는 자발적 커스터마이즈도 유도.
- **입부 계기 한 줄**만 실제 수집 대상. 나머지(실명·역할·기수)는 이미 아는 정보.
- **역할 겸직 허용**(배열). 왜 — PM겸조직장처럼 겸직이 실제로 있음.
- **숨김/상태·email·display_order는 뺌**(YAGNI). 왜 — 지금 8명 전원 공개라 숨김 상태가 필요 없고, email·정렬값도 현 단계엔 불필요. 나중에 필요해지면 그때.
- **audit는 유지**(created_by/updated_by). 왜 — 누가 등록·수정했는지는 운영상 필요. 단, 전체 변경이력 로그 테이블까진 지금 과함.
- **접근 = 최고관리자만**(관리 API). #74(어드민 인증)의 최고관리자 role에 의존 → #74 선행.
- **FE(멤버 탭 화면)는 별도.** 이번은 API까지.
