## 근거 (시나리오별 상세)

### MEM-005 — 운영진 소개 실데이터(BE #87)가 붙으면 랜딩에 실명·직책·학과학번·사진·소개글이 관리자 정렬순으로

**무엇을 확인했나**: 운영진 카드 전용 백엔드 코드(엔티티·컨트롤러·서비스)가 dev 브랜치에 이미 존재하는지, 그리고 랜딩 프론트에 운영진 섹션이 실데이터를 붙일 준비가 됐는지.
**어느 파일·코드를 봤나**:
- `find backend/src/main/java -iname "*staff*" -o -iname "*officer*" -o -iname "*executive*"` → 결과 없음.
- `find backend/src/main/java -iname "*board*" -o -iname "*intro*"` (운영진/소개 관련 다른 네이밍 가능성까지 확인) → 결과 없음.
- `grep -rl "운영진" frontend/src/app` → 결과 없음. 랜딩 프론트에 "운영진" 문자열을 포함한 파일 자체가 없다.
- 참고로 `member` 패키지는 존재하지만(`backend/src/main/java/likelion/khu/website/member/`), 기능 노드(`pm/features/운영진-소개.md`) 설명대로 이는 **멤버 로스터와 별개 데이터**여야 하고, `Member`/`MemberResponse`에는 직책·학과·소개글·정렬순서 같은 운영진 카드 전용 필드가 없다(로스터-탭 검증에서 이미 `Member.java`/`MemberResponse.java` 전체를 확인함).

**왜 그렇게 판단했나**: then이 요구하는 "실명·직책·학과/학번·사진·소개글이 관리자 정렬 순서대로" 노출되려면 최소한 운영진 카드를 저장할 엔티티와 이를 내려주는 API가 있어야 하는데, 두 가지 다 backend 코드베이스에 존재하지 않는다. 이는 기능 노드에 적힌 진행 상태("#87로 착수됐으나 7/09 발주 이후 아직 구현 PR은 없다")와 정확히 일치한다. given 자체("실데이터가 붙으면")가 재현 불가능한 상태이므로 pass/fail 판정 대상이 아니라 `not-run`이 맞다. PR #97(어드민 인증 파운데이션)의 "운영진 관리"는 어드민 *계정* 관리(초대·목록·삭제·역할변경)이지 운영진 소개 카드 데이터가 아니므로 이 시나리오의 근거가 될 수 없다는 점도 코드 검색으로 확인했다(관련 엔티티·컨트롤러가 `admin` 패키지에서도 "운영진 카드" 스키마로는 발견되지 않음).
