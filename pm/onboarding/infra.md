# 인프라 — 업무 방식

이 문서는 이 프로젝트에서 **인프라(장찬욱)가 일하는 방식**만 다룹니다.
(이 프로젝트가 무엇이고 왜 하는지는 [`about.md`](about.md), 첫 미션은 GitHub 이슈로 전달됩니다.)

> [!NOTE]
> 저장소 구조와 Claude로 일하는 법은 [**개발자 온보딩 — 모노레포 & AI-native**](dev-monorepo-ai.md)를 함께 읽어주세요.

---

## 1. 책임과 권한

인프라는 **배포(CI/CD) · 백업 · 운영 · 사이트 생존**을 담당합니다.

서비스가 안정적으로 떠 있고 유지되는 것이 목표이며, 플랫폼·파이프라인 선택은 인프라가 제안합니다 — 단 **추가 예산 0 · 기존 인프라 재사용**이라는 제약 안에서. CI 워크플로는 저장소 루트 `.github/workflows/`에 둡니다.

## 2. 일하는 방식

| 원칙 | 내용 |
|---|---|
| **Day-1 배포** | Walking Skeleton 단계에서 빈 페이지라도 실서버에 올려, 통합·배포 리스크를 프로젝트 맨 앞에서 제거합니다. 이것이 인프라의 핵심 책임입니다. |
| **병렬 (릴레이 아님)** | 디자인·기능의 완성을 기다리지 않습니다. 배포 파이프라인은 코드가 거의 없어도 먼저 뚫어 둡니다. |
| **제약 안에서** | 추가 예산 0, 현 인프라 재사용이 전제입니다. 새 비용이 드는 선택은 PM과 먼저 상의합니다. |
| **완료(Done)의 기준** | "스크립트를 짰다"가 아니라 **"FE·BE가 push하면 실제로 배포되고, 떠 있고, 죽으면 복구된다"**입니다. |

## 3. 소통 방식

| 상황 | 채널 |
|---|---|
| **미션 전달** | PM이 GitHub 이슈로 전달합니다: `[인프라] 미션 (~목표일)`. 이슈는 PM의 로드맵이므로 팀이 열고 닫지 않습니다. |
| **배포 조율 (FE·BE)** | 배포·환경 관련 요구는 **PR·카카오톡에서 FE·BE와 직접** 진행합니다. PM을 거치지 않습니다. |

## 4. PM(김우진)의 역할

| 한다 | 하지 않는다 |
|---|---|
| 무엇·왜·우선순위 결정 | "어떻게"에 대한 지시 (구현은 팀의 몫) |
| 미션 전달, 막힌 것 해소 | 팀 간 기술 인터페이스 중계 (팀끼리 직접) |
| 콘텐츠 수급, 팀 간 통합 | 코드 마이크로매니징 |

> PM은 방향과 우선순위를 제시하는 사람이며, 팀의 작업을 어깨너머로 관리하지 않습니다.
