## 근거 (시나리오별 상세)

### NOTI-001 — pill 클릭 → 이메일 폼 교체
`frontend/src/components/sections/Recruit.tsx`를 봤다. `useState`로 `open` 플래그를 갖고, `open`이 `false`면 "지원기간 신청 알림 받기" 버튼(pill)을, `true`면 `<NotificationForm />`을 렌더링한다(21~46행). 버튼 `onClick`은 `setOpen(true)`만 하고 되돌리는 핸들러가 어디에도 없다 — pill로 돌아가는 UI가 구조적으로 없다. 스펙대로다.

### NOTI-002 — 빈값/형식오류 검증
프론트: `frontend/src/components/NotificationForm.tsx` 51행, `<input type="email" ... required>` — 브라우저 네이티브 검증이 빈 값·형식 오류를 막는다.
백엔드: `backend/src/main/java/likelion/khu/website/notification/dto/NotificationSubscribeRequest.java`에 `@NotBlank(message="이메일을 입력해주세요.")`, `@Email(message="유효하지 않은 이메일 형식이에요.")`가 걸려 있다. `NotificationController.subscribe()`가 `@Valid`로 이 검증을 태우면 `MethodArgumentNotValidException`이 던져지고, `backend/src/main/java/likelion/khu/website/common/GlobalExceptionHandler.java` 34~43행의 핸들러가 첫 필드 에러 메시지를 골라 `HTTP 400 {success:false, message:...}`로 응답한다. 스펙 문구와 정확히 일치한다.

### NOTI-003 — 신규 이메일 제출 성공
`NotificationForm.tsx` 16~45행: 제출 시작 시 `setState('loading')` → 버튼 텍스트가 '신청 중…'으로 바뀌고(64행) `disabled={state==='loading'||state==='success'}`(56·61행)로 입력·버튼이 잠긴다. 성공하면 `setState('success')` + `setMessage(data.message)` + `setEmail('')`(34~36행) — `success===true`인 동안 위 disabled 조건이 계속 참이라 재신청이 막힌다.
백엔드 `NotificationSubscriptionService.subscribe()`(16~22행): 미등록 이메일이면 `repository.save(new NotificationSubscription(...))` 후 `{success:true, message:'모집 알림을 신청했어요!'}` 반환 — 스펙 메시지와 정확히 일치. 이 서비스는 `NotificationSubscriptionRepository`만 의존하고 `EmailService`를 import도 안 한다 — 확인(더블옵트인) 메일을 보낼 코드 경로 자체가 없다.

### NOTI-004 — 이미 구독된 이메일 재신청
`NotificationSubscriptionService.subscribe()` 17~19행: `repository.existsByEmail(...)`이 참이면 `repository.save()`를 타지 않고(=멱등, 신규 저장 없음) `{success:true, message:'이미 등록된 이메일이에요.'}`를 반환한다. 신규 성공 메시지('모집 알림을 신청했어요!')와 문자열이 다르다 — 스펙(#70)이 요구하는 "가입 여부와 무관하게 동일 응답"을 어긴다. `pm/features/모집-알림.md`에도 #70이 OPEN으로 기록돼 있어 알려진 이슈고 fail이 정당하다.

### NOTI-005 — 서버 오류/네트워크 예외
`NotificationForm.tsx` 37~44행: `res.ok`가 거짓이거나 `data.success`가 거짓이면 `catch` 없이 `else` 분기로 `setMessage(data.message ?? '오류가 발생했어요. 다시 시도해 주세요.')`, `fetch` 자체가 던지면(네트워크 예외) `catch` 블록에서 `setMessage('서버에 연결할 수 없어요. 잠시 후 다시 시도해 주세요.')` — 두 경로가 서로 다른 문구를 쓴다. 스펙과 일치.

### NOTI-006 — 동의 체크박스·봇 방지·해지 경로
`NotificationForm.tsx` 전체(47~74행)를 봤다 — `<input type="email">`과 submit 버튼 하나뿐이고 체크박스 요소가 없다. rate limit·캡차 관련 코드도 프론트·`NotificationController`·`NotificationSubscriptionService` 어디에도 없다(백엔드는 `@Valid` 형식 검증만 하고 요청 빈도 제한 로직이 없음). 구독 해지용 엔드포인트도 `notification` 패키지 안에 없다(컨트롤러 메서드는 `POST /subscribe` 하나뿐). `pm/features/모집-알림.md`가 #68·#69를 OPEN으로 기록하고 있어 정합.

### NOTI-007~010 — PR #126 코드 리뷰
`gh pr view 126 --repo likelion-khu-official/website`로 상태를 확인했다 — **state: OPEN(머지 전)**. `gh pr diff 126`으로 전체 diff(611줄)를 읽었다.

- **NOTI-007(전이 시 발송·제목·링크·수신거부)**: `RecruitmentManagementService.open()`(PR diff, `RecruitmentManagementService.java` 133~143행)은 `!status.isOpen()`일 때만(=닫힘→열림 전이) `status.markOpened()` 저장 후 `sendToAllSubscribers()`를 호출한다. `sendToAllSubscribers()`(156~164행)는 `subscriptionRepository.findAll()` — 그 호출 시점(=전이 시점)의 구독자 스냅샷 — 전원에게 `emailService.sendRecruitmentOpenEmail(email, frontendBaseUrl)`을 보낸다. 제목은 `EmailType.java`에 추가된 `RECRUITMENT_OPEN("email/recruitment-open", "[멋쟁이사자처럼 경희대] 모집이 시작됐어요!")` — 스펙 문구와 완전히 일치. 템플릿 `backend/src/main/resources/templates/email/recruitment-open.html`을 보면 `<a th:href="${siteUrl}" th:text="${siteUrl}">지원하러 가기</a>` — Thymeleaf의 `th:text`가 엘리먼트의 텍스트 노드를 `${siteUrl}` 값으로 덮어쓰므로, 실제 렌더링된 메일에서는 "지원하러 가기"라는 글자가 아니라 URL 원문 문자열이 그대로 노출된다(버그성 갭이지만 스펙 시나리오가 정확히 이 현상을 요구하고 있어 이 시나리오는 pass). `siteUrl` 값 자체는 `RecruitmentManagementService`의 `@Value("${app.frontend-base-url}")` — 지원폼이 아니라 랜딩 기준 URL이다. 템플릿 마지막 줄 "더 이상 모집 알림을 받고 싶지 않다면 이 메일을 무시해주세요."가 전부고 수신거부 링크(`<a>` 등)는 없다.
- **NOTI-008(멱등성)**: 위 `if (!status.isOpen())` 한 줄이 "이미 열림 상태에서 재요청 시 무발송"의 전부다. `close()`(145~150행)는 `sendToAllSubscribers()`를 호출하지 않는다. 코드만으로도 논리가 서지만, PR이 직접 추가한 `RecruitmentManagementControllerTest.java`의 `open_AlreadyOpen_DoesNotResend`(364~377행, `verify(emailService, times(1))`로 두 번 열어도 1번만 발송됨을 확인)와 `close_OpenToClosed_DoesNotSendAnything`(381~395행)이 이 로직을 MockMvc로 실제 검증하고 있어 근거가 이중으로 확인된다.
- **NOTI-009(전이 시점 스냅샷)**: `sendToAllSubscribers()`가 `subscriptionRepository.findAll()`을 그 호출 시점에 한 번만 평가한다 — 이후 새로 구독한 사람은 이번 발송 루프에 없으므로 이번 전이 메일을 못 받는다. `NotificationSubscriptionService.subscribe()`(구독 시점 코드)에는 애초에 이메일 발송 코드가 전혀 없으므로, 다음 닫힘→열림 전이가 일어나 `open()`이 다시 `findAll()`을 부를 때 비로소 포함된다 — 스펙과 일치하는 구조.
- **NOTI-010(실패 격리 + email_log)**: `sendToAllSubscribers()`의 `forEach` 안에서 `try { emailService.sendRecruitmentOpenEmail(...) } catch (EmailSendException e) { /* 계속 진행 */ }`(157~163행) — 한 명 실패가 루프를 멈추지 않는다. 코드 주석(152~155행)이 인용한 `EmailService.send()` 쪽 계약("실패 순간 email_log에 FAILURE를 먼저 저장한 뒤 예외를 던진다")은 이번 diff에 포함되지 않은 기존 `EmailService` 코드라 이 PR diff만으로 100% 재확인은 못 했지만, PR이 새로 추가한 `RecruitmentOpenEmailHttpEndToEndIntegrationTest.java`의 `open_OneMalformedAddressAmongSubscribers_OthersStillReceiveMail`(538~556행)이 실제 Mailpit SMTP + 실제 HTTP 경로로 정상 주소는 `EmailStatus.SUCCESS`, 깨진 주소는 `EmailStatus.FAILURE`로 `email_log`에 각각 기록되고 나머지 발송이 막히지 않는 걸 직접 검증한다 — 테스트 코드 자체가 실행되는 걸 이 세션에서 재현하진 않았지만(그럴 경우 "실제 요청·응답 관찰"이라 표기했을 것), 코드 구조와 PR이 직접 짠 E2E 테스트 둘 다 시나리오를 뒷받침한다.

### RECR-001 — /recruit 플레이스홀더
`frontend/src/app/recruit/page.tsx`: `<p>모집 안내 — 준비 중이에요.</p>` 한 줄이 전부(1~7행), 문구가 스펙과 글자 그대로 일치한다. ### RECR-002 — 모집 켜졌을 때 익명 화면
먼저 dev 브랜치 현재 상태를 확인했다: `find backend/src/main/java -iname "*recruitment*"` 결과가 비어 있다 — 모집 on/off라는 개념 자체가 아직 백엔드에 없다(PR #126이 그 개념을 처음 도입한다). 즉 "관리자가 모집을 켰다"는 상황 자체가 dev 브랜치에서는 아직 발생할 수 없다.
PR #126(머지 전) diff를 봐도, 새로 추가되는 엔드포인트는 `RecruitmentManagementController`의 `GET/PATCH /api/admin/recruitment/status` 하나뿐이고 `@PreAuthorize("hasAnyRole('ADMIN','SUPER_ADMIN')")`가 클래스에 걸려 있다 — **익명이 조회할 수 있는 공개 상태 API가 diff에 없다**. PR이 직접 추가한 테스트 `RecruitmentManagementControllerTest.status_NoCookie_Returns401`이 인증 없이 호출하면 401임을 확인한다 — 시나리오가 요구하는 "GET .../status 를 익명이 치면 401"과 일치.
프론트 쪽도 확인했다 — PR #126의 diff에는 프론트 변경이 전혀 없고(파일 목록에 `frontend/` 경로가 없음), dev 브랜치의 `frontend/src/components/sections/Recruit.tsx`·`frontend/src/app/recruit/page.tsx`에도 모집 상태를 fetch하거나 분기하는 코드가 없다(`Recruit.tsx`는 로컬 `open` state로 pill↔폼만 토글할 뿐 서버 상태를 안 봄). 즉 PR #126이 머지되더라도 **익명 방문자 화면은 아무것도 안 바뀐다** — 관리자가 모집을 켜도 랜딩·`/recruit` 어디에도 그 사실을 반영할 공개 조회 경로·FE 분기가 없다. 스펙(모집기간엔 '지원하기'로 전환)과 다르므로 fail — RECR-002는 별도 이슈로 등록돼 있지 않아 이슈화 검토가 필요하다.
