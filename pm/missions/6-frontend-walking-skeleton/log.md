# 진행 로그 — #6 프론트 · 모집 알림 수직 관통 (walking skeleton)

> PM(김우진)이 **판단·결정한 것만** 시간순으로 추린다. GitHub 코멘트 복붙이 아니라, "왜 이렇게 됐나"의 정제된 흔적.
> 형식: `## YYYY-MM-DD` 아래 일어난 일 / 결정 / 왜.

## 2026-06-27 — 미션 설계 (output→outcome 전환)

- **결정:** 초안(6라우트+배포, "협업 잘해" 당부)을 폐기하고 **outcome 중심**으로 재설계.
  - Done = "모집 알림이 디자인→프론트→백엔드를 관통해 실 URL에서 동작"(통합 증명). 라우트 나열(output) 아님.
  - 협업을 **구조로**: 프론트가 `shared/` 모집알림 타입을 *먼저 PR로 연다* → 백엔드·디자인이 거기 모임.
- **왜:** walking skeleton의 핵심은 "통합이 되는가". 라우트 6개는 프론트 단독이라 통합을 증명 못 함.
  "협업 잘해라"는 당부는 PM이 사람에게 떠넘기는 것 — 계약이 협업을 강제해야 한다(레제 리뷰).
