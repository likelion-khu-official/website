## 근거 (시나리오별 상세)

### MEM-001 — 익명이 `/members` 직접 접속 시 플레이스홀더

**무엇을 확인했나**: `/members` 라우트가 실제로 렌더하는 내용.
**어느 파일을 봤나**: `frontend/src/app/members/page.tsx` (전체 8줄).
```tsx
export default function MembersPage() {
  return (
    <main className="min-h-screen flex items-center justify-center">
      <p className="text-gray-400 text-sm">멤버 — 준비 중이에요.</p>
    </main>
  );
}
```
**왜 그렇게 판단했나**: Next.js App Router에서 `app/members/page.tsx`는 `/members` 경로 자체다. 이 컴포넌트는 데이터 페칭도, 조건 분기도 없이 고정 문자열 `"멤버 — 준비 중이에요."`만 렌더한다. then이 요구하는 문구와 글자 그대로 일치한다. 로그인 여부와 무관하게 이 텍스트만 나오므로 "익명이 URL 직접입력"이라는 given도 별도 분기가 없어 그대로 성립한다.

### MEM-002 — 익명이 `GET /api/members`를 치면 200 + 필드 제한 배열

**무엇을 확인했나**: 컨트롤러의 인증 요구 여부, 서비스의 정렬·페이지네이션 로직, 응답 DTO의 필드 구성.
**어느 파일·코드를 봤나**:
- `backend/src/main/java/likelion/khu/website/member/MemberController.java:23-26` — `@GetMapping("/api/members")`에 `@PreAuthorize`나 다른 인증 어노테이션이 없다. 같은 클래스의 `create`/`update`에는 `@PreAuthorize("hasRole('SUPER_ADMIN')")`가 붙어 있어 대비된다. 즉 `list()`만 익명 허용.
- `backend/src/main/java/likelion/khu/website/member/MemberService.java:37-42` — `getAll()`이 `memberRepository.findAllByOrderByCreatedAtAsc()`를 호출한다. 정렬 기준이 `createdAt` 오름차순 = 등록순. `Pageable`이나 검색 파라미터를 받는 오버로드가 없다 — 페이지네이션·필터 없음.
- `backend/src/main/java/likelion/khu/website/member/dto/MemberResponse.java:12-19` — 필드가 `id, name, roles, cohort, emoji, photoUrl, joinReason` 7개뿐이고, `from(Member member)` 정적 팩토리가 이 7개만 옮겨 담는다. then에서 나열한 필드 목록과 정확히 일치(순서 무관).

**왜 그렇게 판단했나**: 컨트롤러가 인증 없이 열려 있고, 서비스가 등록순 정렬·페이지네이션 없는 전체 조회를 하고, 응답 DTO가 정확히 7개 필드만 노출한다는 세 가지가 코드로 모두 확인됐다. `List<MemberResponse>`를 그대로 반환하므로 Spring이 200 OK로 JSON 배열 직렬화한다(예외 처리 분기 없음).

### MEM-003 — PR #121(멤버 계정 체계) 반영 후에도 응답에 학번·전화·비번해시 없음

**무엇을 확인했나**: PR #121이 `member` 패키지에서 실제로 어떤 파일을 바꿨는지, 그중 `MemberResponse.java`가 포함되는지.
**어느 파일·코드를 봤나**: `gh pr diff 121 --repo likelion-khu-official/website` 전체 diff(4013줄)를 받아 `member` 관련 변경 파일 목록을 추렸다.
- 바뀐 파일: `Member.java`, `MemberController.java`, `MemberRepository.java`, `MemberService.java`, `MemberCreateRequest.java`, 그리고 새 `member/auth/` 하위 패키지(로그인 전용 컨트롤러·서비스·DTO들).
- **`MemberResponse.java`는 diff 목록에 아예 등장하지 않는다** — 이 PR에서 손대지 않았다는 뜻.
- `Member.java` diff(파일 내 473번째 줄부터)에서 `studentId`(unique, not null), `passwordHash`, `phone`, `mustChangePassword`, `failedLoginAttempts`, `lockedUntil` 필드가 엔티티에 추가된 것을 확인했다.
- `MemberService.java` diff에서 `create()`가 `passwordEncoder.encode(request.getPhone())`로 초기 비밀번호를 해시해 저장하지만, 반환값은 여전히 `MemberResponse.from(member)` — 변경 전과 동일한 7필드 DTO다.
- `MemberController.java` diff에서 `GET /api/members`(`list()`) 메서드 자체는 수정되지 않았다(새로 추가된 건 `POST /api/admin/members/{id}/password/reset`뿐).

**왜 그렇게 판단했나**: `Member` 엔티티는 민감 필드를 갖게 됐지만, 외부 노출 경계인 `MemberResponse`와 그걸 만드는 `list()`/`getAll()` 경로가 이 PR에서 전혀 수정되지 않았으므로, `GET /api/members` 응답에 학번·전화·비밀번호 해시가 새어나갈 코드 경로가 없다. PR은 아직 머지 전(open, CI는 통과 — build-and-test/gitleaks pass)이라 "PR #121 코드 리뷰(머지 전)"로 확인 방법을 표기했다. 머지 후 코드가 바뀔 가능성은 낮지만, 머지되면 dev 기준으로 재확인 대상으로 표시해야 한다.

### MEM-004 — 로스터 탭 그리드 화면(디자인 #77 진행 중)

**무엇을 확인했나**: `/members` 경로 아래 실제로 존재하는 프론트엔드 파일.
**어느 파일을 봤나**: `find frontend/src/app/members -type f` → `frontend/src/app/members/page.tsx` 단 하나. 카드 컴포넌트, 그리드 레이아웃, `/api/members` fetch 호출 어느 것도 없다(MEM-001에서 본 것처럼 고정 플레이스홀더 텍스트뿐).
**왜 그렇게 판단했나**: then이 요구하는 "멤버 카드 그리드(이름·역할·기수·비주얼·입부계기)" 자체가 코드에 없다. 디자인(#77)이 아직 진행 중이라는 기능 노드 설명과 일치한다. `pass`/`fail` 판정 대상이 아니라 구현 자체가 없어 `not-run`이 맞다.
