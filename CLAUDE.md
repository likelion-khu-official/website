## Claude에게 — 작업 전·후

1. **시작 전** `pm/docs/learnings.md`(배운 것) → `brief.md`(프로젝트 정의)를 읽는다. 작업 디렉터리의 `CLAUDE.md`도 자동으로 함께 로드된다.
   - **코드 작업(FE·BE·인프라)이면 `pm/onboarding/dev-monorepo-ai.md`도 읽는다** — 저장소 구조·AI-native 워크플로·운영 루프(첫날·이슈사이클·막힘·PR)·용어 미니사전. *항상 로드하지 않고 개발할 때만 — 시스템 프롬프트는 가볍게(lazy load).*
   - **"사이트가 어떻게 동작해야 하나"(정책·권한·기능 명세·서버냐 코드냐)가 걸리면 서비스 문서 위키를 WebFetch 한다** → https://github.com/likelion-khu-official/website/wiki — 결정된 스펙의 정본(개요와 원칙 · 아키텍처 결정 · 정보구조와 권한 · 기능 명세). 논의 원문은 `pm/discussions/`, 판단 갈리면 위키가 이긴다. *이것도 lazy load — 관련될 때만.*
   - **보안이 걸리면(시크릿·인증·PII·취약점 등) `SECURITY.md`를 읽는다** — 공통 규칙(이 레포가 public이라는 것 등)이 있고, 거기서 다시 작업 중인 도메인의 `SECURITY.md`로 연결된다. *이것도 lazy load — 관련될 때만.*
2. **작업 끝에 *배운 것*이 생기면** `pm/docs/learnings.md`에 한 줄 남긴다(왜가 드러나게). 일회성 결정·진행 상태·기술 스택 선택 같은 건 굳이 쌓지 않는다 — 진행은 GitHub 이슈/보드로.

## 하네스 지도 — 어디에 뭐가 사는가

모노레포. 전원 Claude 사용(한 레포 = AI가 전체 맥락을 봄). 작업 디렉터리에서 위로 올라가며 `CLAUDE.md`가 자동 로드된다.

- **코드** → `frontend/`·`backend/`·`infra/` (각자 코드 + 역할 `CLAUDE.md`)
- **배운 것·정의** → `pm/docs/` (`learnings.md`·`brief.md`)
- **계약** → `shared/` (코드. FE↔BE API 타입. 문서로 베끼지 않는다 — drift 방지)
- **디자인** → 레포 밖 **Figma**(단일 진실). 구현은 FE가. (디자인 디렉터리 없음)
- **운영** → `pm/`
- **미션** → `pm/missions/<번호>-<슬러그>/` (제안·진행·결과 한 상자. PM쪽 정제 아카이브)
- **스펙(정본)** → GitHub **위키**(서비스 문서, `…/website/wiki`). "무엇이어야 하나". 구현·논의 전 확인. 원 논의는 `pm/discussions/` 아카이브.
- **제품 현황(기능 트리)** → `pm/features/<노드>.md` + 루트 `README.md` 트리. 사이트 전 기능을 노드로, 상태 색(🟢 됨 / 🟡 진행 / 🔵 계획 / ⚪ 미착수)으로. 위키 스펙에서 개요, 이슈·PR에서 진행 파생.

> **불변식:** 코드는 포지션 디렉터리, 배운 것은 `pm/docs`, 계약은 `shared`, 미션 맥락은 `pm/missions` 박스, 팀의 실시간 협업은 GitHub. 박스는 GitHub 복붙이 아니라 PM이 정제한 맥락 — 안 섞는다.
>
> **미션 ↔ 기능 트리는 한 몸:** 위키(스펙) → 트리 노드(현황) → 미션(전진). 미션은 반드시 `pm/features/`의 한 노드를 `Target`으로 걸고, 닫히면 그 노드의 상태 색·`진행 상태`와 README 트리가 갱신된다. **트리의 ⚪·🔵 노드 = 미션 백로그** — 새 일은 여기서 고른다. (운영: `mission`·`claim-mission` 스킬)
