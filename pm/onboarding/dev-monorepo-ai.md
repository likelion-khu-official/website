# 개발자 온보딩 — 모노레포 & AI-native

이 문서는 **저장소 구조**와 **이 프로젝트에서 Claude와 함께 일하는 방식**을 다룹니다.
(팀별 업무 방식은 각 팀 문서를, 프로젝트 배경은 `pm/docs/brief.md`를 보세요.)

> [!NOTE]
> 학습이 이 프로젝트의 1순위 가치예요. 여기 적힌 기준은 *최종* 모습이고, 처음부터 다 못 해도 됩니다. 모르면 묻고, 틀리면 고치며 배우는 게 이 팀의 방식이에요.

---

## 처음 시작하기 (Day 1)

1. GitHub 조직(`likelion-khu-official`) 초대를 수락하고 레포를 클론합니다. *(초대·레포 주소는 PM이 카톡으로 보내줍니다.)*
2. 자기 팀 디렉터리(`frontend/`·`backend/`·`infra/`)로 들어가 Claude Code를 실행합니다. 그러면 그 역할의 `CLAUDE.md`가 자동으로 떠요.
3. **로드맵 보드**에서 나에게 온 이슈를 확인합니다. (아래 *GitHub Projects* 참고)
4. 이슈가 아직 없으면? Claude에게 `pm/docs/brief.md`와 `shared/`를 같이 읽자고 해서 프로젝트와 코드베이스를 익히면 됩니다.

---

## 1. 모노레포 구조

하나의 저장소에 모든 코드가 있습니다. 자기 팀 디렉터리에서 일하되, 전체를 볼 수 있습니다.

| 위치 | 내용 |
|---|---|
| `frontend/` · `backend/` · `infra/` | 각 팀의 **코드 + 역할 `CLAUDE.md`** |
| `shared/` | **FE↔BE API 계약**(타입). 단일 진실 — 문서로 베끼지 않습니다. |
| `pm/docs/` | 배운 것·프로젝트 정의 (`learnings.md`·`brief.md`). PM 소유. |
| 루트 `CLAUDE.md` | 프로젝트 헌법 + 구조 지도. |

> 원칙: **코드는 팀 디렉터리, 계약은 `shared`, 맥락은 `pm/docs`.** 세 축을 섞지 않습니다.

## 2. 왜 모노레포인가

전원이 Claude를 쓰기 때문입니다. 한 저장소면 Claude가 **전체 맥락**(FE·BE·계약)을 한 번에 보고 레이어를 가로지르는 작업을 처리합니다. 저장소가 나뉘면 그때마다 Claude의 맥락이 끊깁니다.

---

## 3. AI-native로 일한다는 것

이 저장소는 **Claude와 함께 일하도록** 설계됐어요. 핵심은 Claude를 *단순 조수*가 아니라 *운영 파트너*로 쓰는 것입니다.

### CLAUDE.md는 자동으로 읽힙니다
`backend/`에서 Claude를 실행하면 `backend/CLAUDE.md`와 루트 `CLAUDE.md`가 **자동으로 함께 로드**됩니다. Claude는 이미 당신의 역할과 프로젝트 맥락을 알고 시작합니다 — 매번 설명할 필요가 없어요.

### 일은 이렇게 한 바퀴 돕니다
1. **이슈를 받는다** — 미션은 GitHub 이슈로 와요. 무엇·왜·완료 기준이 적혀 있고, *어떻게*는 당신이 정합니다.
2. **Claude와 함께 한다** — 이슈를 연 채로 Claude에게 "이 이슈 같이 보자"라고 하면, Claude가 이슈·`brief.md`·`shared/` 계약을 읽고 함께 작업해요. **당신이 결정하고, Claude가 손을 움직입니다.**
3. **PR을 올린다** — 작업이 되면 브랜치에서 PR을 올리고, 본문에 `Closes #이슈번호`를 적어 이슈와 연결합니다.
4. **완료를 알린다** — 이슈에 "끝났습니다" 코멘트를 달고 카톡으로 PM에게 알려요. **이슈를 닫는 건 PM입니다**(확인 후). 당신은 안 닫아요.

### Claude를 어떻게 쓰는가

| Claude에게 맡겨도 되는 것 | 당신이 해야 하는 것 |
|---|---|
| 보일러플레이트 · 문법 · 디버깅 | **설계·구조 결정** |
| "이거 어떻게 하지" 리서치 | "이게 맞나" 판단 |
| 코드베이스·계약 읽고 이해 돕기 | *왜* 이 방식인지 이해 |

> [!IMPORTANT]
> Claude가 전부 짜주게 두면 *코드는 나오지만 당신은 배우지 못합니다.* 1순위 가치는 **당신의 학습**이에요. 처음엔 Claude에게 "왜 이렇게 했는지 설명해줘"라고 하며 따라가도 괜찮습니다 — 결정 근육은 천천히 붙어요. 핵심은 **결정은 당신이, 타이핑은 Claude가.**

### 계약은 코드에서 봅니다
FE↔BE API는 `shared/`에 타입으로 있어요. Claude에게 "`shared` 보고 이 API 연동해줘"가 됩니다 — 계약이 코드라서 Claude가 정확히 압니다. *(mock = 백엔드가 아직 없어도 가짜 데이터로 프론트를 먼저 만드는 것.)*

---

## 막히면 — 두 종류

- **기술적으로 막힘** (어떻게 구현? 버그? 에러?) → 먼저 Claude에게 물어봐요. 그래도 30분 넘게 혼자 막히면 팀·카톡으로. **묻는 건 민폐가 아니에요** — 혼자 끙끙대는 게 더 손해입니다.
- **PM이 정할 문제에 막힘** (이거 범위에 있나? 뭐 먼저? 브리프가 애매?) → `ask-pm` 스킬. Claude가 그걸 PM 결정 이슈로 올려줘요. 당신이 PM 영역을 멋대로 정하지 않게 도와줍니다.

> [!TIP]
> `main`은 보호돼 있어요. 당신 브랜치/PR에서 **맘껏 실험하세요** — 머지 전에 검토되고, 깨져도 되돌릴 수 있습니다. 실서버는 인프라가 지켜요.

---

## PR · 머지 규칙

- `main`에 직접 push하지 않습니다. 항상 **브랜치 → PR**.
- PR은 **최소 1명 리뷰** 후 머지합니다. (작은 팀이라 가볍게 — 같은 팀원이나 PM)
- PR 본문에 *무엇을·왜* 한두 줄이면 충분해요. (화면 작업이면 스크린샷 환영)

---

## 배포 — 깊이 몰라도 돼요

브랜치가 환경입니다. 아래 두 줄이 전부예요.

| 내가 한 것 | 자동으로 일어나는 일 |
|---|---|
| `dev`에 push (또는 PR 머지) | **stage** 서버에 자동 배포 |
| `main`에 PR 머지 | **prod** 서버에 자동 배포 |

- **stage** — 개발 중 프론트가 붙는 서버. 맘껏 실험해도 돼요. 데이터가 지저분해지면 인프라에 리셋 요청하거나, GitHub Actions → "Reset Stage DB" 워크플로를 직접 실행해도 됩니다 (`RESET` 입력 후 Run).
- **prod** — 실제 사용자 서버. main에 머지됐을 때만 바뀌고, CI를 통과한 것만 올라가요.

> 서버 주소는 PM이 카톡으로 공유합니다. (보안상 레포엔 없어요)

---

## GitHub Projects — 로드맵 보기

PM이 모든 이슈를 **로드맵 보드**(GitHub Projects)로 관리해요. 팀원은 *보기*만 하면 됩니다. (여닫고 상태 바꾸는 건 PM)

- **내 일 찾기** — 보드에서 `Team` 또는 담당자(assignee)로 필터.
- **전체 그림** — Roadmap 뷰에서 팀별로 누가 언제 뭘 하는지 시간축으로 보여요.
- **터미널에서도** — `gh issue list --assignee @me` 로 내 이슈, `gh project item-list <번호> --owner likelion-khu-official` 로 보드.
- 보드 링크는 PM이 공유합니다.

## 용어 미니사전

- **스캐폴딩 / 보일러플레이트** — 기능 없는 *빈 프로젝트 골격*. (Next.js면 `create-next-app`의 결과물)
- **계약(contract)** — FE·BE가 주고받는 API의 *모양*. `shared/`에 타입(코드)으로 둡니다.
- **mock** — 진짜 API가 없어도 *가짜 응답*으로 먼저 개발하는 것.
- **IA** — 사이트의 페이지 구조·내비게이션.
- **RFC** — "이렇게 하자"는 짧은 제안 문서.
- **Walking Skeleton** — 기능은 비어도 *전 레이어를 한 번 관통해 띄워보는* 최소 뼈대.
- **크리티컬 패스** — 늦어지면 다른 일까지 다 막는 길목.
- **통합 드라이버** — 한 페이지를 끝까지 책임지고 디자인·BE를 당겨오는 사람(보통 FE).
