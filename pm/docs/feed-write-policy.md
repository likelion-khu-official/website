## 한 줄 결정

**매직링크를 폐기하고, "누구나 작성 → 등록된 멤버 리스트에서 본인 선택 → 제출 → 관리자 검수 후 게시"로 간다.**
로그인·OTP·비밀번호·봇가드 전부 지금은 안 붙인다. 방어선은 **관리자 검수 하나**. 작성자는 자유입력이 아니라 **멤버 명단에서 고른다**(오타·비멤버 방지, 일관된 바이라인).

## 왜 이렇게 가나

- **핵심 동기:** 부원이 수고스러우면 글을 안 쓴다. 마찰을 최대한 없애야 한다.
- **매직링크 폐기:** "의사표명 → 운영진이 링크 발급 → 작성" 핸드셰이크가 가장 큰 마찰. 게다가 링크 발급 게이트는 **관리자 검수 게이트와 중복**이다. 월 예상 2건 볼륨에 게이트가 둘일 이유가 없다.
- **학교메일 OTP 검토 후 폐기:** 경희대 메일이 **핸들 기반**(`hjdd0309@khu.ac.kr`)이라 주소에서 이름·학번을 못 뽑는다. 그래서 OTP가 남기는 값은 "경희대 소속 증명 + 봇 차단"뿐인데, 그건 관리자 검수와 겹친다. 이름은 어차피 자가입력 → OTP를 붙여도 신원이 안 채워짐 → YAGNI로 제외.
- **자유입력이 아니라 멤버 선택:** 이름을 타이핑시키면 오타·비멤버·표기 불일치가 생긴다. **등록된 멤버 명단에서 본인을 고르게** 하면 작성자가 실제 멤버로 제약되고 바이라인이 일관된다(유한솔 제안).
- **사칭 리스크 수용:** 명단 선택엔 인증이 없어 남이 본인인 척 고를 수 있으나, **말이 안 되면 검수에서 컷**한다. 소규모·검수 있음이라 수용.

## 검수의 의미 — 왜 유지하나 (그리고 왜 못 빼나)

검수를 "없앨 부담"으로 보기 쉽지만, 이건 이 방식이 성립하는 **전제**다.

**핵심 — 검수가 있어서 "누구나 쓰기"를 열 수 있다.**
앞문(링크·인증 없는 공개 글쓰기)을 활짝 열 수 있는 유일한 이유가 뒤에 검수라는 백스톱이 있어서다. **앞문 개방과 검수는 한 쌍**이다. 검수를 빼면 앞문이 무방비가 되고, 다시 앞문에 자물쇠(링크·인증)를 달아야 하는데 그건 방금 없앤 그 마찰이다. 즉 검수 = 앞문을 열어둔 채 안전할 수 있게 하는 장치.

**검수가 지키는 것**
- **공식 얼굴 통제(가장 깊은 의미):** 이 블로그는 브리프상 "외부에 보여줄 멋사의 얼굴". 올라간 글은 전부 **동아리 이름을 달고** 나간다. 검수 = "우리 이름으로 뭐가 나가는지 우리가 정한다"는 편집 게이트. 개인 블로그엔 불필요하지만 **공식 채널이라 필요한** 것.
- **법적·평판 방어선:** 명예훼손·개인정보 노출·부적절 콘텐츠가 공식 채널로 나가면 동아리 책임. 대학 동아리라 **학교까지 엮일 수 있다.** 마지막 컷.
- **사칭·악의 차단:** 인증이 없으니 이게 유일한 방어.
- **억제 효과(안 보이는 값):** "사람이 게시 전에 읽는다"는 사실만으로 쓰레기가 준다.

**검수가 아닌 것**
- 무거운 편집 아님 — "말도 안 되는 것만 컷", 낮은 바.
- 자격 심사 아님 — "쓸 자격 있나"가 아니라 "우리 이름으로 나가도 되나".
- 병목 아님 — 월 2건에 1클릭. 부담으로 느껴지면 그건 검수 자체가 아니라 **어드민 화면이 없어서**(아래 "알려진 갭"). 화면 만들면 풀린다.

**왜 못 빼나:** 우려의 핵심인 *사칭*은 오직 **인증(로그인)만** 막는다(드롭다운·AI필터·봇가드로는 못 막음). 그런데 멤버 로그인이 없고, #74는 어드민 인증이지 부원 인증이 아니며, 월 2건 쓰자고 부원 로그인을 만드는 건 과하다. **검수를 그냥 빼면 우려하던 사칭이 오히려 즉시 게시된다** — 방향이 반대다.

**열린 질문(검수 강도를 가른다):** 이 블로그를 **공식 얼굴로 볼지, 자유 게시판으로 볼지**에 따라 검수의 의미가 달라진다. 공식 얼굴 → 사전검수(지금처럼 승인해야 게시)가 격을 지키는 최소 장치. 자유 게시판 → 사후검수(바로 게시 후 나쁘면 `HIDDEN`)로 늦출 수 있다. **현재 정책은 "공식 얼굴" 전제 → 사전검수 유지.** 정체성이 바뀌면 이 결정도 다시 본다.

## 제품 관점 — 글쓰기가 이렇게 바뀐다

**지금 (매직링크)**
1. 부원이 운영진에게 "글 쓰고 싶다"고 말한다
2. 운영진이 1회용 링크를 발급한다 (누구 이름으로 쓸지도 이때 지정)
3. 부원이 그 링크로 들어가 글을 쓴다
4. 제출 → 운영진 검수 후 게시. 링크는 1회용이라 다음 글은 또 발급받아야 함

**바뀌면 (자유 작성 + 멤버 선택)**
1. 부원이 글쓰기 버튼을 그냥 누른다 (링크 불필요)
2. 멤버 명단에서 본인을 고른다 (이름·파트 자동)
3. 제목·본문 쓰고 이미지 넣고 제출
4. "검수 후 게시" 안내 → 운영진이 확인하면 게시

| | 지금 | 바뀌면 |
|---|---|---|
| 글쓰기 시작 | 운영진에 링크 요청 → 대기 | 버튼 누르면 바로 |
| 작성자 이름 | 링크에 지정됨(운영진) | 멤버 명단에서 본인 선택 |
| 운영진 할 일 | 링크 발급 + 검수 | 검수만 |
| 게시 전 검수 | 있음 | **그대로 있음** |

**안 바뀌는 것:** 게시 전 관리자 검수, 이미지 업로드, 임시저장, 미리보기.

## 새 흐름 (기술 요약)

1. 누구에게나 보이는 **글쓰기 버튼** (링크 불필요)
2. 작성 시 **멤버 명단에서 본인 선택** (이름·파트가 멤버 레코드에서 따라옴 — 로그인이 없어 작성자 식별용)
3. 작성 후 **제출** → 글은 `DRAFT`(검수 대기)로 저장
4. **관리자가 검수** 후 승인 → `PUBLISHED`로 게시

## 신원 스펙 — 멤버 선택 (결합: 갈래 1, FE만 멤버 API 의존)

작성자는 `GET /api/members`(멤버 API #76) 로스터에서 고른다. **FE가 목록을 받아 드롭다운 → 선택된 멤버의 이름·파트를 문자열로 POST.** BE는 멤버 엔티티와 결합하지 않고 문자열만 받는다(아래 표는 기존 계약과 동일 — 값의 출처만 "자유입력"→"멤버 선택").

| 필드 | 필수 | 타입 | 출처 |
|---|---|---|---|
| `authorName` | ✅ | string (≤50) | 선택한 멤버의 이름. 공개 바이라인에 표시 |
| `authorPart` | ✅ | enum | 선택한 멤버의 역할. `PLANNING`·`DESIGN`·`FRONTEND`·`BACKEND`·`INFRA` |
| ~~학번~~ | ❌ | — | 수집 안 함 |

> **왜 갈래 1(FE만 의존)인가:** BE에 `memberId` FK를 걸면(갈래 2) 작성자↔멤버가 서버에서 진짜 연결되지만, BE가 #76 멤버 모듈 구현 완료까지 통째로 블록된다. FK 정합성의 값은 어차피 검수가 방어선이라 낮고, memberId를 나중에 얹는 건 Type 2(되돌리기 쉬움). 그래서 지금은 BE를 멤버에서 떼어 **`GET /api/members` 하나에만** 의존을 좁힌다.

> **의존:** 이 흐름은 **멤버 API #76의 `GET /api/members`가 나와야** 동작한다(현재 미구현·OPEN). 아래 "알려진 갭" 참조.

## 검수는 이미 있다 — 새로 만들지 않는다

BE에 `DRAFT → PUBLISHED → HIDDEN → PUBLISHED` 상태기계와 `PATCH /api/admin/posts/{id}/status`가 이미 구현돼 있다. **새 글이 `DRAFT`로 저장되는 것도 그대로.** 검수 = 이 상태 전이의 재사용이지 신규 개발이 아니다.

## 무엇이 바뀌나 (요약)

| 영역 | 변경 |
|---|---|
| **shared** | `PostCreateRequest`에 `authorName`·`authorPart` 추가. 매직링크 타입 3종 삭제. `PostSummary`/`PostDetail`에 `authorPart` 추가 |
| **BE** | `POST /api/posts`에서 `X-Magic-Token` 헤더 제거 → 본문에서 작성자 식별. 매직링크 전체(엔티티·컨트롤러·서비스·DTO·예외·테스트) 삭제. `Post`에 `authorPart` 컬럼 추가 |
| **FE** | 작성 폼에서 토큰 분기 전부 제거 → **`GET /api/members` 로 멤버 드롭다운**(본인 선택). 선택 멤버의 이름·파트를 POST. 제출 후 "검수 후 게시" 안내. `/blog/write`는 링크 없이 접근 가능 |

## 의존 — 멤버 API(#76)가 선행이다

멤버 선택 흐름은 **`GET /api/members`(#76) + 채워진 로스터**를 전제로 한다. 현재 #76은 OPEN·미구현이고, 멤버 생성/수정은 #74(어드민 인증) 선행이다.

- **코드 의존:** FE 작성 폼은 `GET /api/members`가 나와야 드롭다운을 채운다 → 블로그 글쓰기 FE는 #76의 멤버 읽기 엔드포인트 뒤에 줄 선다.
- **데이터 의존:** 고를 명단이 있어야 한다. #74 어드민 멤버관리 UI를 기다리지 말고, **멤버 몇 명을 수동 시드**(SQL/API)해 블로그 글쓰기를 언블록한다.
- **시퀀싱:** 블로그 글쓰기 BE 미션은 멤버와 결합하지 않으므로 **지금 나갈 수 있다**(문자열 수신). FE 미션만 `GET /api/members` 완료까지 대기.

## 알려진 갭 — 어드민 검수 화면이 없다

정책은 "관리자 검수 후 게시"인데 **현재 어드민 화면이 없다**(`/blog` 공개 페이지만 존재). `PATCH /api/admin/posts/{id}/status` API는 있으나 UI가 없고, 이 어드민 API는 `#74 인증`이 붙기 전까지 `permitAll`이라 **지금 검수 UI를 만들면 무인증 공개 어드민 페이지**가 된다.

- **당분간:** 관리자가 Swagger/API로 직접 승인(`DRAFT→PUBLISHED`).
- **정식:** #74 어드민 인증 완료 후 "검수 대기 목록 + 승인/숨김" 어드민 화면을 별도 미션으로.
- 이 갭을 방치하면 "제출은 되는데 아무도 게시 못 하는" 상태가 되므로, #74 result에 검수 UI 승계를 박아야 한다.
