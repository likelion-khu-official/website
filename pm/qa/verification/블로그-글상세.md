### BLOG-D01 — pass

`frontend/src/app/blog/[slug]/page.tsx:47-73`: `<h1>{post.title}</h1>`, `<p>{post.authorName} · {formatDate(...)}</p>`, 썸네일은 `post.thumbnailUrl`이 있을 때만 조건부 렌더(56-63행), 본문은 65-67행 `<div className="whitespace-pre-wrap break-words ...">{post.content}</div>` — `whitespace-pre-wrap`은 줄바꿈만 살리는 CSS일 뿐 마크다운·HTML을 파싱하지 않는다(`dangerouslySetInnerHTML` 같은 게 없다). then의 조건과 정확히 일치한다.

### BLOG-D03 — pass

`PostService.getPublishedPost()`(`PostService.java:42-48`): `postRepository.findBySlugAndStatus(slug, PostStatus.PUBLISHED)` — 조회 조건에 `PostStatus.PUBLISHED`가 고정으로 걸려 있어 HIDDEN·DRAFT 글은 슬러그가 맞아도 안 걸린다. 없는 slug와 동일하게 `ResponseStatusException(NOT_FOUND, "글을 찾을 수 없어요.")`를 던진다 — 세 경우(없음·숨김·초안) 모두 리포지토리 쿼리 단계에서부터 구분 없이 같은 예외로 합쳐진다.

프론트는 `frontend/src/lib/feedApi.ts:45-51`의 `getPostBySlug()`가 `res.status===404`면 `null`을 반환하고, `frontend/src/app/blog/[slug]/page.tsx:45`의 `if (!post) notFound();`가 Next.js의 `not-found.tsx`(`frontend/src/app/blog/[slug]/not-found.tsx`)를 렌더한다 — "글을 찾을 수 없어요." + "목록으로 돌아가기" 링크(`href="/blog"`)가 정확히 이 파일에 있다.

### BLOG-D04 — pass

Next.js `error.tsx` 컨벤션: `frontend/src/app/blog/[slug]/error.tsx`가 404 외의 실패(예: 백엔드 500)를 캐치해 표시한다. `getPostBySlug()`는 404일 때만 `null`을 반환하고 그 외 실패는 `parseJsonOrThrow()`가 그대로 예외를 던지므로(`feedApi.ts:26-38`), Next가 이를 `error.tsx` 바운더리로 넘긴다. (파일 내용은 `blog/error.tsx`와 동일 패턴 — "불러오지 못했어요…" + 다시 시도 버튼.)

### BLOG-D05 — pass

PR #120(open, 머지 전)을 `gh pr diff 120`으로 읽었다. **백엔드는 이 PR에서 전혀 바뀌지 않는다** — PUBLISHED→HIDDEN 전환 API(`PATCH /api/admin/posts/{id}/status`)는 이미 dev 브랜치의 `PostController.java:53-58`·`PostService.updateStatus()`(`PostService.java:55-61`)에 존재한다. `Post.transitionTo()`(`Post.java:63-72`)가 상태를 바꿀 뿐 slug나 다른 필드는 안 건드린다.

핵심은 조회 쪽이 **캐시 없이 매번 새로 상태를 검사**한다는 것 — `getPublishedPosts()`/`getPublishedPost()`가 매 호출마다 `status=PUBLISHED` 조건으로 DB를 다시 쿼리하고, 프론트 `feedApi.ts`도 목록·상세 둘 다 `cache:'no-store'`다(BLOG-L01·BLOG-D01 참고). 따라서 어드민이 상태를 HIDDEN으로 바꾸는 순간부터, 그다음 요청(새로고침)은 목록에서 빠지고 상세는 404(BLOG-D03의 동일 로직)가 된다. "이미 열어둔 탭은 새로고침 전까지 그대로" 부분은 Next.js 서버 컴포넌트가 최초 렌더 시점의 HTML을 클라이언트에 유지하는 표준 동작이라 별도 코드 없이도 성립한다(재요청 전엔 재검증이 없으므로).

PR #120 diff(`BlogManager.tsx`)는 이 API를 호출하는 **어드민 UI**를 새로 추가할 뿐이다 — `changeStatus(post, 'HIDDEN', ...)`가 `updatePostStatus(post.id, 'HIDDEN')`(`adminApi.ts` 신규 함수, PATCH `/posts/{id}/status`)을 호출한다. 즉 이 시나리오의 **동작 자체는 이미 dev에 있고, PR #120은 그걸 어드민이 클릭 몇 번으로 쓸 수 있게 하는 화면일 뿐**이다.

### BLOG-D06 — pass

`PostStatus`(`PostStatus.java`, 파일명만 확인 — enum 값 DRAFT/PUBLISHED/HIDDEN)와 `Post.transitionTo()`가 `next==PUBLISHED && publishedAt==null`일 때만 `publishedAt`을 새로 찍고(67-69행), 이미 발행 이력이 있으면(재게시) 기존 `publishedAt`을 건드리지 않는다 — 재게시해도 같은 slug로 그대로 다시 노출된다는 것이 이 필드 로직만으로 설명된다. BlogManager.tsx(PR #120 diff)의 UI 라벨도 `post.status==='DRAFT' ? '게시' : '다시 게시'`(148행 부근)로 재게시 케이스를 별도 구분해 같은 `changeStatus(post,'PUBLISHED',...)` 호출로 처리한다.
