## 인프라 · CI/CD
- **바이트코드로 컴파일되는 언어(Java 등)는 멀티스테이지 Dockerfile에서 컴파일 스테이지를 크로스 플랫폼 타깃에 묶을 필요가 없다 — `--platform=$BUILDPLATFORM`으로 빌드 머신 네이티브에 고정하면 QEMU 에뮬레이션을 완전히 피한다.** CD의 "이미지 빌드 & 푸시" 스텝이 11분 44초로 배포 시간 대부분을 차지해서 조사해보니, `backend/Dockerfile`의 Gradle 빌더 스테이지가 워크플로의 `platforms: linux/arm64` 전체에 묶여서 **컴파일 자체가 QEMU arm64 에뮬레이션 위에서** 돌고 있었다(OCI 서버가 arm64라 크로스 빌드가 필요했던 것). Java는 네이티브 기계어가 아니라 JVM 바이트코드로 컴파일되므로 어떤 CPU에서 컴파일하든 결과물이 동일하다 — `FROM --platform=$BUILDPLATFORM gradle:... AS builder`로 빌더 스테이지만 빌드 머신 아키텍처(GH 러너=amd64)에 고정하고, 최종 런타임 스테이지는 플랫폼 지정을 그대로 둬서 buildx의 타깃(arm64)을 따르게 하면 된다. 로컬에서 재현 검증: 최종 이미지 아키텍처는 여전히 arm64/linux, `compileJava`는 에뮬레이션 없이 정상 속도(~72초)로 완료. sqlite-jdbc처럼 네이티브(JNI) 의존성이 있어도 실행 시점 JVM 기준으로 라이브러리를 고르므로 크로스 컴파일에 영향 없음(#158). → "왜 arm64로 빌드해야 하지?"라고 물을 때, 정답은 "런타임 이미지만" — 컴파일이 CPU-bound인 컴파일형 언어(Java/Go/Rust 등, 단 Go/Rust는 네이티브 기계어라 타깃 아키텍처 크로스컴파일 플래그가 별도로 필요함)를 도커로 크로스 빌드할 땐 항상 스테이지 분리를 먼저 의심할 것.
- **git이 파일을 한 번도 실행권한(755)으로 추적한 적이 없으면, 서버의 수동 `chmod +x`는 다음 재체크아웃 때 반드시 벗겨진다 — CRLF 사고와 증상은 같지만 원인은 다르다.** 07-13 18:04 KST stage/prod DB 백업 부재 알람 재발(FIRING) 조사 결과: `infra/backup-db.sh`가 `git ls-tree`상 처음부터 줄곧 `100644`였다(755로 커밋된 적이 아예 없음). 07-04~07-12 정상 작동은 서버에서 언젠가 수동으로 걸어둔 `chmod +x` 덕분이었을 뿐 — 이 실행권한은 git 추적 대상이 아니라 서버 로컬 상태에만 존재했다. 07-13 14:46 KST PR #110 머지로 서버가 이 파일을 다시 pull/체크아웃하면서 git이 기억하는 644로 조용히 되돌아갔고, 그날 18:00 UTC cron이 `Permission denied`로 실패했다. 이전 CRLF 사고(줄바꿈)는 "블롭 내용은 정상인데 로컬 체크아웃 설정이 변형"이었던 반면, 이번은 "블롭의 파일 모드 자체가 원래 틀렸음"이라 증상(조용한 cron 실패 → 26시간 뒤 알람)은 똑같아도 고치는 지점이 다르다(`.gitattributes` eol 강제가 아니라 `git update-index --chmod=+x` 커밋). → cron이 직접 실행하는 서버 스크립트를 새로 추가하거나 만질 때는 `git ls-tree HEAD -- <path>`로 실제 커밋된 모드가 `100755`인지 반드시 확인할 것 — 서버에서 지금 실행되고 있다는 사실이 git에 그렇게 기록돼 있다는 뜻은 아니다.
- **"방금 고친 버그"가 "지금 겪고 있는 증상의 원인"이라는 보장은 없다 — 인과관계는 반드시 별도로 검증해야 한다.** CD 3연속 실패(07-11 12:33/14:35/15:16 UTC)를 조사하다 git pull 무음 실패를 찾아 고쳤는데(아래 항목), 정작 그 3번의 헬스체크 실패는 이것과 무관했다 — `docker-compose.yml`이 그 몇 주간 내용 자체가 안 바뀌었어서(diff 0) 낡은 채로 있어도 실질적 영향이 없었다. 진짜 원인은 완전히 별개로, admin-auth 파운데이션(`84d7528`, 07-11 01:00 KST, 3번의 실패보다 먼저 머지됨)이 들여온 `JwtProvider`가 stage의 `JWT_SECRET`(184비트, HS256 최소 256비트 미달)으로 크래시 루프를 돌고 있었던 것. 두 버그를 같은 날 같이 발견하면 "먼저 고친 게 원인이었겠지"라고 넘겨짚기 쉬운데, 실제로 확인(commit 시각 비교, 파일 diff)해야 한다.
  - 타임라인: CD 실패 3회(07-11) → git pull 원인 규명·수정 PR #104 머지(07-12 01:36) → 서버 동기화 완료 → 드리프트 감지 PR #105 머지(02:04) → 헬스체크 로그 가시화 PR #106 머지(02:28) → 그 로그로 JWT_SECRET 184비트 발견 → stage JWT_SECRET 교체(512비트) → 재배포 성공(run 29176978544, 02:34~02:36) → 배포 마커 파일이 드리프트 오탐 유발한 것 발견·수정 PR #107 머지(02:40) → 알람 포맷 통일(ONS_OPTIMIZED)·pending-duration 5→3분 → 테스트 드리프트로 왕복 검증(파일 생성 02:42:06 → FIRING 02:50:00 → 파일 제거 → OK 복귀 03:01:00) 중 중복 알람 발견·정리.
- **OCI Monitoring 알람의 쿼리 윈도우(`[Xm].max()`)를 줄여도 "순간적 blip이 알람을 못 띄우게" 막을 수 없다 — 윈도우가 pending-duration보다 크기만 하면 단 한 번의 데이터포인트로도 파이어된다.** 커스텀 메트릭을 cron 5분 주기로 찍으면, 데이터 공백을 안 만들려고 윈도우를 최소 그 이상(5~6분)으로 잡아야 하는데, 그러면 윈도우가 pending-duration(3분)보다 커져서 그 안에 있는 단 하나의 값도 pending-duration을 만족시키기 충분하다. 즉 저빈도 폴링 메트릭에서는 윈도우 튜닝으로 오탐을 못 막는다 — 윈도우를 줄여서 실제로 얻는 건 "문제 해소 후 OK 복귀 속도"뿐. 오탐 방지는 애초에 "정상인데 순간적으로 잡히는 상태"를 메트릭에서 빼는 것(예: 배포 스크립트의 롤백 마커 파일을 `.gitignore`에 추가)으로 해야 한다.
- **OCI CLI(Windows)가 한글이 섞인 `--body`로 알람을 생성하면, API 호출 자체는 성공한 뒤 응답을 콘솔에 출력하다가 `UnicodeEncodeError`(cp949)로 죽는다** — 이걸 "명령 실패"로 착각하고 재시도하면 동일 조건의 알람이 중복 생성된다(포맷 등 후속 설정은 재시도로 얻은 ID에만 적용되고, 먼저 만들어진 유령 알람은 기본값인 RAW 포맷으로 계속 따로 알림을 보낸다). 클라이언트가 에러를 던져도 서버 쪽은 이미 처리됐을 수 있다는 걸 의심하고, 재시도 전에 `list`로 실제 생성 여부부터 확인할 것. 실행 전 `PYTHONIOENCODING=utf-8` 설정도 예방책.
- **Windows에서 `core.autocrlf=true`로 체크아웃한 셸/파이썬 스크립트를, git이 아닌 경로(수동 scp 등)로 Linux 서버에 올리면 shebang이 깨진다.** git 저장소의 blob 자체는 LF로 정상이어도(`git show`로 확인 가능), 로컬 작업 트리 체크아웃은 `autocrlf=true`가 checkout 시 LF→CRLF로 되돌린다 — 이 상태에서 파일을 직접 서버로 복사하면 `#!/usr/bin/env bash\r`처럼 shebang에 `\r`이 섞여 `bash\r: No such file or directory`로 즉시 실패한다. `infra/`는 CD 파이프라인 대상이 아니라(`backend/**`·`shared/**`만 트리거) 수동 배포 경로라 이 문제가 특히 잘 숨는다 — git이 중간에 안 끼면 blob이 정상이어도 아무 의미가 없다. 실제로 `backup-db.sh`가 이렇게 깨져서 **이틀간 DB 백업이 조용히 실패**했고, 마침 그걸 감시하려고 만든 백업 부재 Alarm이 실제로 계속 FIRING 중이었던 걸 나중에 발견했다(#83 실발동 검증 중 우연히 포착 — 감시 체계를 만든 목적 자체가 "조용한 실패를 먼저 알아채기"였는데, 그 감시 체계를 테스트하러 갔다가 진짜 실패를 잡은 것). → 재발 방지는 `.gitattributes`에 `*.sh text eol=lf` / `*.py text eol=lf`처럼 경로별로 줄바꿈을 강제하는 것 — 이건 `core.autocrlf` 설정보다 우선 적용돼 그 사람 로컬 설정과 무관하게 작업 트리 체크아웃 자체를 LF로 고정한다.
- **원격 SSH 배포 스크립트에 `set -e`가 없으면, 중간 명령(`git pull` 등)이 실패해도 조용히 다음 줄로 넘어가서 진짜 원인이 아니라 몇 분 뒤 헬스체크 타임아웃으로만 실패가 드러난다.** 실제 사고: 서버 워킹트리에 untracked 파일이 있는 상태에서 그 경로가 나중에 git에도 커밋되면 `git pull`이 "untracked working tree files would be overwritten" 로 매번 조용히 실패 — `set -e`가 없어 스크립트는 계속 진행되고, 그 결과 서버의 infra 체크아웃이 몇 주째 옛날 커밋에 고정된 채 매 배포가 롤백만 반복했다. 원격 배포 스크립트는 무조건 `set -euo pipefail`로 시작할 것.
- **미머지 브랜치 작업을 검증 목적으로 서버에 직접 반영해두면, 그 사실이 git에는 전혀 안 보인 채로 운영에 계속 남는다.** 관측·알림(#83) 브랜치 작업 중 만든 백업 성공 메트릭 스크립트가 dev 머지 전에 이미 서버에서 실제로 돌고 있었고, 이후 dev의 다른 커밋들과 계속 충돌만 냈다. 서버에서 급하게/검증차 고친 게 있으면 그날 안에 최소 단위로라도 git에 반영해둬야 다음 정기 배포가 그 상태를 깨뜨리지 않는다.
- **git의 untracked-overwrite 보호는 내용이 완전히 같아도 무조건 막는다(mode만 달라도 add/add 충돌로 뜬다).** 실행권한(755 vs 644) 차이만으로도 병합이 막힐 수 있음 — 이럴 땐 `git add` 후 커밋하면 끝, 내용 자체는 건드릴 게 없다.
- **권한 없는 API의 위험도는 "얼마나 깊은 정보를 보여주냐"가 아니라 "내용(토큰 등)을 포함하냐"로 갈린다 — 두 축은 독립적이다.** 메일 발송 이력 조회 기능 논의 중 나온 프레임: 배달 확인은 ①우리→릴레이 접수 ②릴레이→수신서버 relay/bounce ③받은편지함 안착 ④열람 확인, 이렇게 계층이 있지만 "몇 차까지 보여주냐"(깊이)는 "본문을 담냐"(내용)와 별개 문제다. 메타데이터만이면 permit-all이어도 위험도가 낮고(이메일 수집 정도), 본문(특히 매직링크·재설정 토큰)을 담으면 몇 차든 어떤 인원 제한을 걸든 위험하다 — "개발자만 조회 가능"도 본문 문제는 못 푼다(제3자가 토큰을 쓸 수 있는 힘을 갖게 되는 구조 자체가 원인이라 인원을 좁혀도 안 없어짐).
- **"아직 서비스 공개 전이라 괜찮다"는 보안 판단 근거가 아니다.** 서비스 공개는 마케팅 개념이지 네트워크 접근 통제가 아니다 — 백엔드는 이미 공개 인터넷에 떠 있고, 어드민 초대 대상이 실제 팀원 본인들이면 "공개 전"에도 실피해가 가능하다. "미공개니까 임시로"는 그대로 영구화되기 쉬운 패턴이기도 하다.
- **OCI Email Delivery는 자체 배달 로그(Accepted/Relayed/Bounce/Complaint)를 제공하지만 본문·제목 필드는 아예 없다** — 스키마 어디에도 content 관련 필드가 없어서, "배달 이력엔 본문을 안 남긴다"는 게 우리 판단이 아니라 이 서비스 자체의 설계 원칙임을 실측(공식 문서 JSON 스키마)으로 확인했다. 백엔드가 Instance Principal로 OCI Logging Search를 직접 호출해 이 메타데이터를 가져올 수 있어(새 자격증명 불필요), 별도 웹훅 파이프라인 없이도 온디맨드 조회로 충분하다.
- **Deliverability Dashboard(반송률·평판 추이)는 Email Delivery에 기본 포함된 무료 기능이라 새로 만들 게 없다** — Administrator 계정이면 콘솔에서 바로 보임(최근 90일). 개별 메일 단위 확인보다 "시스템 전체가 건강한가"를 보는 더 중요한 1차 지표인데 이미 있는 걸 안 보고 새 기능부터 만들려는 경우를 주의할 것.
- **커밋에 개인정보·내부식별자가 이미 들어간 뒤 발견했다면, dev(스쿼시 금지 브랜치) 정책을 지키면서도 정리할 수 있다.** 방법: 문제 커밋이 있는 기능 브랜치를 로컬에서 `git reset --soft <merge-base>` + 재커밋으로 깨끗한 커밋 하나로 압축 → `force-push`로 그 브랜치 자체를 덮어씀 → dev엔 평소처럼 일반 머지(스쿼시 아님, dev 정책 그대로 준수) → 머지 직후 브랜치 삭제. 결과: dev 히스토리엔 마스킹된 최종 상태 커밋 하나만 남고, 문제됐던 중간 커밋들은 어떤 브랜치·태그에서도 도달 불가능해져 일반적인 열람·검색·clone으론 안 보인다. 완전 물리적 삭제(`git filter-repo`+강제푸시)는 SHA가 다 바뀌고 기존 clone이 깨지는 파괴적 작업이라, 새어나간 게 자격증명이 아니라 개인 이메일·OCID 같은 낮은 심각도라면 이 정도가 비용 대비 적정선.
- **`gitleaks-action@v2`는 조직(organization) 소유 레포에 유료 라이선스를 요구한다** — 개인 계정 공개 레포만 무료. 조직 레포에선 액션 대신 오픈소스 CLI를 Docker로 직접 돌리면 라이선스 없이 동일 기능(`docker run zricethezav/gitleaks:latest detect ...`). 스캔 범위도 전체 히스토리(`git log -p ALL`)가 아니라 `--log-opts="origin/<base>..<head-sha>"`로 **PR이 새로 추가하는 커밋만** 좁혀야 한다 — 안 그러면 이미 병합된 과거 히스토리의 기존 항목 때문에 완전히 무관한 미래 PR까지 영원히 막힐 수 있다.
- **`docker-compose.yml`에서 `image: ...${STAGE_TAG:-stage-latest}` 패턴을 쓰면, 태그 없이 수동으로 `docker compose up -d`를 실행하는 순간 조용히 `stage-latest`로 떨어진다.** CD는 매번 커밋 SHA 태그(`STAGE_TAG=<sha>`)로 정확히 배포하고, 레지스트리의 `stage-latest` 태그 자체도 매 배포마다 함께 갱신한다 — 문제는 태그가 아니라 **서버 로컬 이미지 캐시**다. CD가 SHA 태그로만 pull/기동하기 때문에 서버에 로컬로 캐시된 `stage-latest` 이미지는 그동안 최신으로 pull된 적이 없다. 이 상태에서 사람이 SSH로 들어가 태그 없이 재기동하면 레지스트리의 최신 이미지가 아니라 로컬에 얼어붙은 옛날 이미지를 재사용하게 되며, 헬스체크가 실패할 수도 있다(예: 스키마 마이그레이션 이전 이미지). 수동 재기동 시 반드시 `STAGE_TAG=<원하는 태그>`/`PROD_TAG=<원하는 태그>`를 명시할 것.
- **certbot standalone 발급 시 Docker nginx를 먼저 내려야 한다.** certbot이 80 포트로 인증하는데 nginx 컨테이너가 잡고 있으면 실패 — `docker compose stop nginx` 후 발급, 완료 후 재기동. 두 도메인을 한 cert에 묶으려면 `-d domain1 -d domain2`로 한 번에 발급하면 첫 번째 도메인 경로에 저장된다.
- **서버 로컬 DNS 캐시가 느려도 글로벌 전파는 이미 됐을 수 있다.** `nslookup`이 실패해도 `dig @8.8.8.8`로 확인하면 실제 전파 여부를 알 수 있다 — certbot은 글로벌 DNS 기준으로 검증하므로 로컬 캐시와 무관하게 발급 가능.
- **compose `depends_on`이 있으면 named service 기동 시 dependency 이미지 태그도 재평가된다.** `docker compose up -d nginx`가 nginx의 `depends_on`인 backend-stage/prod의 이미지 태그가 바뀌었는지 확인하고, 달라졌으면 재생성한다. 동적 리졸버를 쓰는 nginx처럼 실제로 의존성이 없는 경우엔 `depends_on`을 아예 빼는 게 낫다.
- **compose 파일의 `restart:` 정책을 고쳐도 이미 떠 있는 컨테이너엔 소급 적용 안 된다 — 재생성해야 반영된다.** prod 컨테이너가 compose 파일엔 `unless-stopped`로 선언돼 있는데도 실제로는 `on-failure`로 떠 있었다(9일 전, 정책이 바뀌기 전에 생성된 채 그 뒤로 재배포가 없었던 것) — 재부팅 복구력을 실측 점검하던 중 `docker inspect`로 발견(#83). 고치려고 태그 없이 `docker compose up -d backend-prod`를 돌렸다가, `${PROD_TAG:-prod-latest}`의 fallback인 `prod-latest`가 **amd64 전용 이미지**라 arm64 서버에서 `exec format error` 크래시 루프로 실제 장애(약 2~3분)가 발생 — fallback 태그는 "오래된 버전"일 뿐 아니라 "아예 실행 자체가 안 되는" 이미지일 수도 있다는 걸 보여준 사례. 수동 재생성 시 `PROD_TAG=<정확한 태그>`를 반드시 명시해야 하는 이유가 하나 더 늘었다(기존 알려진 위험: 구버전 스키마 / 이번에 새로 확인: 아키텍처 불일치로 완전 크래시). 덤으로 확인된 것: 이 정도(2~3분) 장애는 UptimeRobot 무료 플랜(5분 간격)의 체크 타이밍과 안 맞으면 알림 자체가 안 온다 — 5분보다 짧은 장애는 순전히 운으로 놓칠 수 있다는 걸 실측으로 확인(`infra/uptime-monitoring.md` 참고). **같은 날 stage에서도 재현됨** — `STAGE_TAG` 없이 재생성했더니 `stage-latest`로 조용히 바뀌어 있었다(이번엔 arm64 호환이라 무중단이었지만 의도한 커밋과는 다른 이미지). 태그 미지정 재생성 위험은 prod 한정이 아니라 이 fallback 패턴을 쓰는 모든 서비스에 동일하게 적용된다.
- **여러 서비스를 동시에 stop해도 실제로 죽는 시점은 갈릴 수 있다.** prod+stage를 같은 명령(`docker compose stop backend-stage backend-prod`)으로 동시에 내렸는데, nginx 로그상 prod가 stage보다 약 1분 먼저 502로 넘어갔다(#83 모니터링 테스트, 2026-07-08) — Spring Boot가 SIGTERM을 받으면 즉시 죽지 않고 graceful shutdown(진행 중 요청을 정리하느라 포트를 잠깐 더 붙드는 유예시간)에 들어가기 때문. 장애 시나리오나 모니터링 테스트를 설계할 때 "동시에 내렸으니 감지·알림도 동시에 올 것"이라고 가정하면 안 된다 — 서비스별 종료 로직에 따라 실제 다운 시점이 갈린다.
- **알림 채널마다 실제 도착 시각이 다르다 — 웹훅은 초 단위, 이메일은 몇 분 지연될 수 있다.** 같은 장애 이벤트에 대해 Discord 웹훅은 거의 즉시 왔는데 이메일은 SMTP 릴레이·수신서버 처리 때문에 몇 분 늦게 도착했다(#83 실측, 2026-07-08). 두 채널의 도착 시각 차이만 보고 "하나가 안 왔다/뭔가 이상하다"고 오판하지 말 것 — 채널별 전달 지연 특성을 먼저 감안해야 한다.
- **compose에서 여러 서비스가 같은 변수를 공유하면 하나의 배포가 다른 환경을 오염시킨다.** `${BACKEND_TAG:-stage-latest}` / `${BACKEND_TAG:-prod-latest}` 처럼 fallback만 다르고 변수는 같으면, BACKEND_TAG가 세팅된 순간 fallback이 무시되고 두 서비스 모두 같은 값을 쓴다. 서비스마다 독립 변수(`STAGE_TAG` / `PROD_TAG`)를 써야 한 배포가 다른 쪽 컨테이너를 건드리지 않는다.
- **nginx를 compose에 넣으면 동시 운영 시 포트 충돌로 결국 앞단 레이어가 필요해진다.** 두 스택을 동시에 띄우는 게 기본이라면 native nginx가 더 단순하다.
- **롤백용 이전 태그를 `/tmp`에 두면 재부팅 시 유실된다.** deploy path 안에 두어야 영속성이 보장된다.
- **이미지 태그에 환경 prefix(`test-`/`prod-`)를 붙이면 하나의 레지스트리에서 두 환경이 독립적으로 관리된다.** `latest`만 쓰면 롤백 포인트가 없다.
- **브랜치를 환경과 1:1로 대응시키면(`dev`→test, `main`→prod) CI/CD 분기 로직이 브랜치 이름 하나로 결정된다.** 환경별 워크플로를 별도 파일로 나누는 것보다 단순하다.

## 팀 · 운영
- **학습은 페어여야 성립한다.** 1명은 "일하는 자리", 2명은 "배우는 자리" — 혼자는 막혀도 가르칠 짝이 없다.
- **일은 단계가 아니라 오너십+목표로 준다.** 결정을 멤버에게 남기는 게 곧 학습을 남기는 것. 단계까지 주면 받아쓰기가 되고, 그건 Claude가 더 잘한다.
- **핸드오프(던지기)는 안티패턴.** 계약을 먼저 정하고 병렬로 가면 통합이 싸다. "디자인 끝→FE→BE" 릴레이는 틈에서 깨진다.
- **PM은 팀 사이 유일한 채널이 되면 안 된다.** 인터페이스는 팀끼리 직접, PM은 우선순위·통합·막힘 해소. 중계자가 되면 전화게임·병목.
- **발산은 sync, 수렴은 텍스트.** 열린 결정·신생팀 정렬은 통화로, 정해진 것(계약·이슈)은 텍스트로.
- **PM이 "어떻게"까지 주면 학습이 죽는다 — 결정엔 "왜"를 붙여라.** "X 써"(받아쓰기)가 아니라 "이 제약이라 X"(근거→학습). 프로젝트 1순위가 멤버 학습이면 미션은 *무엇을·왜·제약*까지만 주고 *어떻게*는 비운다. 답을 주지 말고 제약을 줘라.
- **Walking skeleton도 전 팀 동시는 아니다.** 남의 산출물에 의존하는 팀(예: 배포는 띄울 앱이 있어야)은 그게 무르익은 *다음* 사이클이 제 타이밍 — 억지로 같은 줄에 넣으면 대기·빈작업이 된다. 한 줄은 "되는 곳까지"만 솔직히 긋는다.

## 제품
- **"정적 서빙 vs DB"는 변경 빈도가 아니라 ①미디어(바이너리) 유무 ②비개발자가 고칠 편집 surface의 필요/수렴으로 가른다.** 운영진 소개와 연간 활동 계획을 저울질하다 나온 프레임 — 둘 다 "연 1회 변경·운영진이 갱신"으로 빈도·주체가 같아서, 빈도로 판단하면 둘을 못 가르고 틀린 결론이 나온다. 진짜 갈림길은 (1) **사진 같은 바이너리가 붙나** — 텍스트는 GitHub 웹 에디터로 비개발자도 고치지만 이미지 리사이즈·최적화·커밋은 고통이라 업로드(DB+스토리지)가 이김, (2) **"운영진이 고치는 공개 콘텐츠"를 한 편집 surface(어드민)로 수렴시킬 것인가** — 어떤 건 어드민, 어떤 건 코드 파일로 두면 진짜 비용은 기술이 아니라 이 이원화된 멘탈 모델이다. 그래서 운영진 소개(사진O)는 어드민(DB)으로, 활동 계획(텍스트뿐)은 static 후보로 갈렸다. 단 멋사가 *코딩 동아리*라 "static + GitHub 웹 편집"이 비개발 조직보다 현실성이 있다는 전제가 이 판단에 깔린다.
- **권한 체계는 "역할 정의가 확정될 때까지 하나로 통일"이 낫다 — 이른 세분화는 빈 그릇만 만든다.** 운영진 vs 최고관리자 권한을 나누려 했는데, 운영진의 역할 범위 자체가 아직 애매했다(핸드오프 구조라 "어느 데이터까지 수정 가능한가"가 미정). 역할 정의 없이 권한 계층만 먼저 만들면 기능은 그대로인데 분류만 늘어나고, 나중에 실제 역할이 잡히면 다시 재설계해야 한다. 지금은 최고관리자 하나로 통일하고, 기능별 접근권한 세분화는 역할이 구체화된 뒤에 한다. 운영진은 물갈이되는 사람들이라 그 위에 소수의 영속적 관리자("왕")가 필요하다는 구조적 이유가 있고, 최소한 현재 운영진의 역할은 "멤버 관리하는 사람" 정도로 잡는다.
- **읽기중심 사이트의 콘텐츠 병목(PM 1인)의 구조적 해법은 "멤버 셀프서비스 표면" — 로그인은 서비스에 녹이지 말고 격리된 3번째 표면으로 본다.** 블로그 글쓰기 신원 문제로 시작했지만 진짜 병목은 랜딩·프로필·프로젝트 콘텐츠가 전부 PM 한 명에게 쏠려 사이트가 플레이스홀더로 멈춘 것(브리프의 "콘텐츠는 운영진이 채운다"가 현실에선 1인 쏠림). 해법은 인증을 각 기능에 급조하는 게 아니라 **부원이 자기 것을 직접 관리하는 표면 하나(`/member/*`)를 여는 것** — 그러면 작성자 신원·프로필·프로젝트가 한 번에 풀린다. 리드온리 사이트에서 로그인이 "묵직"하게 느껴지는 이유는 인증을 공개 화면에 녹이려 하기 때문인데, **공개/멤버/어드민 3면으로 격리**하면 공개 화면의 보안 핸들링이 0이 되고 로그인은 URL 뒤 별도 표면이 된다(방문자는 존재조차 모름). 전제: 인증 기반(#74)이 이미 생기면 멤버는 role 티어 하나 추가일 뿐이라 증분 비용이 작다. 결정 기록: `product-map.md`.
- **정책을 조각조각 바꾸면 개발자가 같은 기능을 N번 갈아엎는다 — 방아쇠 결정이 상위 구조에 딸렸으면 상위부터 확정하고 하위 미션을 홀드한다.** 블로그 글쓰기가 매직링크 폐기(1차)→멤버 드롭다운(2차)까지 갔는데, 팀 논의가 "멤버 로그인" 상위 구조로 번지면서 드롭다운도 세션 신원으로 대체될 운명이 됐다. 여기서 미션을 그대로 진행시켰으면 devs가 세 번째로 같은 화면을 갈아엎어야 했다. 방아쇠(블로그 신원)가 상위 결정(사이트 표면 구조)의 함수임을 알아챈 순간, 하위 미션(#94·#95)을 **홀드 코멘트만 보내고** 상위를 지도로 확정한 뒤 재설계하는 게 옳다. 신호: "자꾸 정책 바꿔서 죄송"이 나오면 그건 하위를 급하게 미는 게 아니라 상위가 안 잡힌 것 — 상위부터.
- **도구를 서비스 성격에 맞춰라.** 읽기 중심 콘텐츠/홍보 사이트엔 무거운 백엔드가 오버킬.
- **상시 모집이 아니면 사이트는 두 모드로 생각한다.** 평상시=인지·관심, 모집기간=전환. 한 모드만 만들면 평상시 방문이 휘발.
- **타겟은 좁힐수록 날카로워진다.** 카피·디자인·IA가 선명해진다.
- **BE는 비주얼을 안 보지만 거친 흐름은 본다.** 데이터는 브리프에서, 세부 기능은 와이어/IA에서.

## 기술 · 인프라
- **OCI Absence 알람의 대괄호(리샘플링 격자)와 소괄호(부재 허용 시간)는 별개 값이고, 대괄호는 실제 신호 주기 이상이어야 한다.** 하루 1번(24h) 오는 신호에 `[1h].absent(26h)`를 쓰면 24개 버킷 중 23개가 항상 비어 보여 방금 신호가 와도 FIRING이 안 풀린다 — `[24h].absent(26h)`처럼 대괄호를 신호 주기에 맞춰야 한다. 소괄호의 데드라인은 달력이 아니라 마지막 실제 신호 시각에 붙어 움직이는 rolling window라 하루 지연엔 안전하지만, 진짜 연속 무신호(예: 이틀 연속 실패)는 정확히 잡아낸다.
- **`@Transactional` 메서드가 실패를 알리려 unchecked 예외를 던지면, 그 예외가 남기려던 "실패 기록" 자체도 같이 롤백된다.** 어드민 로그인 잠금(#90)에서 실측: 비밀번호 오답 시 `admin.recordFailedLogin()`으로 실패 횟수를 늘린 뒤 `InvalidCredentialsException`/`AccountLockedException`을 던지게 짰는데, 5번 연속 오답을 줘도 끝까지 401만 나고 423(잠금)이 안 떴다 — 원인은 Spring의 기본 롤백 정책이 "unchecked 예외 발생 시 해당 트랜잭션 전체 롤백"이라, 예외를 던지는 순간 그 직전에 늘려둔 실패 카운터까지 함께 사라진 것. "실패했다는 사실을 던지면서(예외) 그 실패의 흔적도 남기고 싶다"는 요구는 기본 정책과 정면으로 충돌한다 — `@Transactional(noRollbackFor = {해당 예외들})`을 명시해야 카운터 증가가 커밋되면서 예외도 정상 전파된다. 무차별 대입 방어·재시도 카운터처럼 "실패 자체를 기록해야 하는" 로직엔 항상 이 패턴을 의심할 것.
- **`@WithMockUser`의 기본 principal 타입은 커스텀 `@AuthenticationPrincipal` 타입과 다르다 — 조용히 null이 된다.** Spring Security Test의 `@WithMockUser`는 `org.springframework.security.core.userdetails.User`를 principal로 채우는데, 컨트롤러가 `@AuthenticationPrincipal MyCustomPrincipal principal`처럼 자체 타입을 기대하면 타입이 안 맞아 `principal`이 그냥 null로 들어온다(에러도 안 남). JWT 쿠키 기반 인증(#90)처럼 필터가 커스텀 principal 타입을 채우는 구조라면, `WithSecurityContextFactory`를 구현한 전용 `@WithMockXxxUser` 애노테이션을 만들어 그 필터가 실제로 채우는 것과 동일한 타입으로 테스트 `SecurityContext`를 구성해야 한다 — 기본 `@WithMockUser`로 role 체크는 통과해도 principal을 쓰는 순간 NPE로 터진다.
- **커넥션 풀이 1개(SQLite)일 때 "호출자 트랜잭션과 무관하게 독립 커밋"하려면 REQUIRES_NEW가 아니라 `@Async` + `@TransactionalEventListener(AFTER_COMPLETION)`을 써야 한다 — 핵심은 "같은 스레드냐 다른 스레드냐".** `@Transactional(REQUIRES_NEW)`는 같은 스레드에서 새 커넥션을 요청하는데, 풀에 1개뿐이면 바깥 트랜잭션이 쥔 그 커넥션을 기다리다 타임아웃(email_log 트랜잭션 격리 시도 중 CI에서 실측, #85). `@TransactionalEventListener(AFTER_COMPLETION)`만 얹어도 안 풀린다 — 콜백이 트랜잭션을 실행한 그 스레드에서 동기 호출되고, 커넥션 반납(`cleanupAfterCompletion`)은 그 콜백이 끝난 *뒤*라 REQUIRES_NEW와 똑같은 순환 대기가 재현된다. `@Async`를 추가로 얹어야 콜백이 별도 스레드에서 돌아서, 원래 스레드가 콜백을 안 기다리고 바로 커넥션을 반납할 수 있다 — 그래야 순환 대기가 끊긴다. 풀을 키워도 근본 해결 안 됨 — SQLite 자체가 파일 레벨 단일 쓰기라 커넥션 2개가 동시에 쓰려 하면 서로를 기다리다 셀프 데드락. → **"트랜잭션 있으면 이벤트 발행(비동기 저장), 없으면 그 자리에서 즉시 저장"으로 분기**하면 기존 동작(트랜잭션 없는 경로)은 그대로 두면서 이 문제만 해결된다(#85, `EmailLogEventListener`). 대안(활성 트랜잭션 감지 시 즉시 실패하는 가드)은 "로그가 남는다"는 원래 목표를 회피하는 것일 뿐이라 진짜 해법이 아니었다.
- **프록시(nginx) 앞단 제한이 앱 자체 제한과 같은 값이면, 앱의 친절한 에러 로직이 죽은 코드가 된다.** 백엔드가 이미지 업로드 5MB 초과 시 친절한 JSON 에러를 만들어놨는데, nginx `client_max_body_size`를 설정 안 해서 기본값(1MB)에서 먼저 막혀 못생긴 nginx HTML 413이 나가고 있었다(이슈 #57 curl 테스트 중 발견). 값을 앱과 똑같이 5MB로 맞춰도 여전히 문제 — 두 한도가 같으면 nginx가 항상 동시에 또는 먼저 끊어서 앱 코드가 실행될 기회가 없다. **앞단 한도를 앱 한도보다 살짝 크게(예: 6MB vs 5MB)** 잡아야 그 사이 구간에서 앱이 요청을 받아 자기 검증 로직·메시지를 실제로 실행할 수 있다 — 프록시는 바깥 방어선, 앱 메시지가 실제 사용자 경험이라는 걸 분리해서 생각해야 한다.
- **Docker 파일 바인드 마운트는 경로가 아니라 inode를 참조한다 — `sed -i`처럼 임시파일+rename으로 "in-place 편집"하는 도구는 호스트에서 새 inode를 만들어 컨테이너의 마운트를 갈아치우지 못한다.** `infra/nginx.conf`를 `sed -i`로 고치고 `nginx -s reload`까지 했는데 컨테이너 안 내용이 그대로였다 — `docker inspect`의 Mount `Source` 경로는 똑같은데 호스트/컨테이너 파일의 inode 번호와 mtime이 서로 달랐다(컨테이너는 예전 inode를 계속 붙들고 있음). reload는 nginx 프로세스 재시작일 뿐 마운트를 다시 맺어주지 않는다 — 이런 경우엔 `docker compose up -d --force-recreate <service>`로 컨테이너 자체를 다시 만들어야 새 inode(=새 마운트)를 잡는다. 단일 파일을 통째로 바꿔치기하는 편집기·스크립트를 서버의 바인드 마운트 대상 파일에 쓸 땐 항상 의심할 것.
- **제한 계정 이름은 실제 권한 범위를 정확히 반영해야 한다 — 아니면 이름이 거짓 안전감을 준다.** DB 접근용 제한 계정을 `dbviewer`로 만들었는데, 실제로는 stage에서 `INSERT`/`UPDATE`/`DELETE`(DML)까지 되고 `no-pty` 같은 SSH 강제 명령 옵션은 셸 탈출만 막지 SQL 실행 범위는 안 막는다 — "viewer"라는 이름만 보면 읽기 전용인 줄 알고 방심하게 된다. `dbclient`처럼 범위를 특정하지 않는 이름으로 바꾸거나, 반대로 이름을 지키려면 DB 연결 자체를 `?mode=ro`로 강제해 진짜 읽기 전용으로 좁혀야 한다 — 이름과 실제 권한 중 하나를 실제에 맞춰야지 이름만 그럴듯하면 안 된다.
- **1인 SSH 계정만 있을 땐 안 보이던 권한 구멍이, 두 번째 계정을 추가하는 순간 실제 위험이 된다.** `.env.prod`/`.env.stage`가 world-readable(644)이어도, DB 파일이 `root:root`라 쓰기에 `sudo`가 필요해도, sudoer 계정 하나뿐일 땐 그 자체가 이미 root라 문제가 안 드러난다. 팀원용 계정을 새로 만드는 순간 world-readable secret은 실제 유출 경로가 되고, "sudo 필요"는 원래 sudoer였던 사람에게만 무의미한 장벽이었다는 게 드러난다. → 접근 범위를 넓히기(팀원 계정 추가 등) *전에* 기존 permissive 설정(world-readable secret, 소유권)부터 점검. 넓은 계정(sudo) 공유 대신 목적 전용 제한 계정(그룹 기반 파일 권한 + SSH `authorized_keys`의 `command=` 강제 명령)으로 블라스트 반경을 좁히는 패턴이 이런 상황에 맞는다.
- **SSH `authorized_keys`의 `command=` 강제 명령은 "그 프로그램이 셸을 안 준다"는 것만 보장하지, "그 프로그램 자체가 셸을 실행할 수 있는지"는 안 막는다.** `dbclient` 계정을 `command="sqlite3 /path/db",no-pty,...`로 제한해두고 "셸을 못 얻으므로 안전하다"고 문서화했는데, sqlite3 CLI는 `.shell`/`.system` dot-command로 임의 OS 명령을 실행할 수 있고 `no-pty`는 pty 할당만 막을 뿐 그 입력 자체는 안 막는다 — 등록된 키를 가진 사람이면 누구나 그대로 셸 탈출이 가능했다(다행히 아직 등록된 키가 없어 실제 악용 전에 발견). → forced command로 REPL/인터프리터(sqlite3, python, 각종 shell-out 지원 CLI)를 직접 노출하지 말고, 위험한 입력(dot-command, `ATTACH` 등)을 걸러내는 wrapper를 한 겹 끼워야 한다. "셸을 안 준다"와 "셸을 실행 못 한다"는 다른 말이다.
- **"other에게만 열자"는 디렉터리 하나만 보고 판단하면 안 된다 — 그 안의 형제 디렉터리까지 다 확인해야 한다.** 제한 계정(`dbclient`)이 필요한 파일에 닿게 하려고 상위 디렉터리(`/home/ubuntu`)에 `chmod o+x`를 하려다가, 그 밑에 이미 group/other로 열려있던 다른 디렉터리(`backups/` — cron이 만든 로컬 백업 스냅샷, `-rw-r--r--`)가 같이 뚫릴 뻔했다. 지금은 상위 디렉터리가 막혀 있어 안 보이던 느슨한 하위 권한이, 상위를 열면 한꺼번에 다 드러난다. → (1) 열기 전에 그 디렉터리 밑 형제 항목을 전부 나열해서 각각의 권한을 확인하고, (2) `chmod o+x`(전체 other에게)보다 `setfacl -m u:계정:x`(딱 그 계정에게만) 로 블라스트 반경을 계정 단위로 좁히는 게 안전하다. 자동화 스크립트(cron 등)가 만드는 파일도 기본 umask에 맡기지 말고 명시적으로 권한을 좁혀야 한다(민감 데이터일수록).
- **정책 문서("이건 하지 마세요")와 기술적 강제는 다르고, 후자가 없으면 전자는 신뢰에만 의존한다.** `db-access.md`에 "스키마 변경(ALTER/CREATE/DROP)은 ❌ 절대 금지"라고 표로 적어놨는데, `dbclient` forced command 스크립트는 실제로는 이 SQL문들을 전혀 막고 있지 않았다 — 문서와 실제 동작이 따로 놀고 있었던 것. → 제한 계정에 "이러이러한 건 하지 마세요"라고 룰만 적어두는 걸로 끝내지 말고, 정말 막아야 하는 것이면 스크립트 레벨에서 걸러야 한다(신뢰 기반 정책 vs 기술적 가드레일).
- **한 줄에 세미콜론으로 여러 SQL문을 이어 붙이면, 문장 시작 키워드만 보는 필터는 쉽게 우회된다.** 첫 필터는 각 입력 줄의 맨 앞 키워드만 검사했는데, `SELECT 1; DROP TABLE x;`처럼 허용된 문장 뒤에 금지된 문장을 세미콜론으로 붙이면 그대로 통과할 뻔했다. → 줄 단위가 아니라 세미콜론으로 나눈 각 문장 단위로 검사해야 한다(완벽한 SQL 파서는 아니어도, 최소한 "한 줄에 여러 문장" 우회는 막아야 함).
- **`/tmp`에서 "다른 사용자 소유 파일에 666으로 쓰기"를 테스트하면 커널이 막아서(`fs.protected_regular`) 가짜 실패가 난다 — 실제 배포 환경으로 착각하지 말 것.** sticky-bit + world-writable 디렉터리(`/tmp`)에서 소유자가 다른 파일은, 권한 비트가 666이어도 최신 커널의 `fs.protected_regular=2` 하드닝이 다른 uid의 쓰기를 막는다(고전적 `/tmp` 레이스 공격 방지 기능, 리눅스 기본 하드닝이지 버그 아님). → 다른 계정 권한을 테스트할 땐 실제 운영 경로와 같은 소유권·권한 패턴(`root:그룹`, `2770`/`660`, world-writable 아님)으로 스크래치 디렉터리를 만들어서 검증해야 한다 — `/tmp`에 대충 만들어 테스트하면 있지도 않은 버그를 쫓게 된다.
- **OpenSSH `authorized_keys`에 같은 공개키를 여러 줄(옵션만 다르게) 등록해도, 처음 매치되는 한 줄만 적용되고 나머지는 완전히 무시된다.** 한 사람에게 stage.db용 `command=` 한 줄 + prod.db용 `command=` 한 줄, 이렇게 같은 키를 두 번 등록해서 "둘 다 접근 가능"하게 하려 했는데, 실제로 테스트 키를 만들어 두 줄 등록하고 접속해보니 첫 번째 줄의 `command=`만 실행되고 두 번째 줄은 죽은 코드였다(2026-07-04 실측, 이론만으로 안 믿고 실제 테스트 키로 검증). → 한 키로 여러 옵션(환경 등)을 선택하게 하려면, `command=`에 값을 고정하지 말고 스크립트 안에서 `$SSH_ORIGINAL_COMMAND`(클라이언트가 실제로 요청한 값)를 읽어서 분기해야 한다 — "같은 키를 여러 줄" 방식은 SSH 인증 매칭 로직상 작동하지 않는다.
- **AWS CLI v2.23+의 기본 체크섬(Default Integrity Protections)이 OCI S3 호환 API의 고수준(`aws s3`) 명령을 깨뜨린다.** `aws s3 cp`/`aws s3 ls`가 자격증명·리전 다 멀쩡해도 `SignatureDoesNotMatch`("secret key required... could not be found")를 낸다 — 원인은 권한이 아니라, AWS CLI 2.23+가 Put/Get에 기본으로 체크섬 헤더를 얹는데 OCI가 이걸 지원 안 해서 서명이 깨지는 것. `AWS_REQUEST_CHECKSUM_CALCULATION=when_required` 환경변수로 끄면 되는데, 저수준 `aws s3api`(put-object/list-objects-v2)엔 바로 먹히지만 고수준 `aws s3 cp`/`aws s3 ls`엔 안 먹힐 수 있다(내부 transfer 경로가 달라서). → OCI 같은 S3 호환 스토리지를 로컬에서 권한 검증할 땐 처음부터 `s3api`로 하면 이 삽질을 건너뛴다.
  **후속(2026-07-04, DB 백업 자동화 중 실측):** 체크섬 환경변수를 꺼도 `aws s3api`(v2.35, awscrt 서명기) 자체가 OCI 앞에서 *간헐적으로* `SignatureDoesNotMatch`를 낸다 — 같은 자격증명·같은 명령이 방금 성공하고 바로 다음 호출에 실패하는 식이라 재시도로 숨겨질 뿐 근본 원인이 아니다. 반면 `boto3`(classic SigV4, awscrt 안 씀)로 같은 요청을 짜면 반복 호출(5연속 put/delete)에서 한 번도 안 깨졌다. → 자동화 스크립트(cron 등 사람이 매번 결과를 안 보는 곳)에서 OCI S3 호환 API를 칠 땐 aws-cli 대신 boto3를 쓴다 — 사람이 지켜보는 1회성 디버깅에는 `s3api`도 충분하지만, 무인 반복 실행엔 간헐적 실패가 곧 무결성 사고다.
- **Docker 파일 마운트는 대상 경로가 컨테이너에 없으면 디렉터리를 만든다.** `./data/app.db:/app/data/app.db`처럼 파일 단위 마운트 시, 컨테이너 이미지에 `/app/data/` 디렉터리가 없으면 Docker가 마운트 포인트를 파일이 아닌 디렉터리로 생성한다 — SQLite가 열려는 경로가 디렉터리가 돼 CANTOPEN. 파일 하나를 마운트하려면 디렉터리 단위로 올리고(`./data:/app/data`) 파일명은 앱에서 결정하게 하는 게 안전하다.
- **Hibernate 6.x에서 `spring.jpa.database-platform`은 dialect로 전달되지 않는다.** `spring.jpa.properties.hibernate.dialect`로 직접 설정해야 한다. `database-platform`은 Spring Boot 자동 구성이 처리하던 방식인데 Hibernate 6.6.x에서 동작이 바뀌었다.
- **Walking skeleton의 버그는 순서대로 숨는다.** 앞 레이어 에러가 뒤 레이어를 가린다 — 플랫폼 불일치(arm64), 엔드포인트 없음(actuator), 앱 기동 실패(DB env), 볼륨 버그, Dialect 설정 순으로 5개가 차례로 드러났다. 파이프라인 첫 성공까지는 각 단계가 독립 실패 포인트다.
- **오래 사는 브랜치(main·dev)엔 squash·rebase 머지 금지.** 계속 사는 브랜치는 공통 조상이 함께 전진해야 다음 머지가 깔끔하고, 브랜치 간 진행 추적·롤백이 된다. squash/rebase는 새 SHA를 만들어 히스토리를 갈라놓고 공통 조상이 못 움직여 → 머지 충돌이 반복된다(실제로 dev→main에서 터짐). 버려지는 브랜치(feat·doc)는 squash해도 무방하나 설정이 레포 단위라, "히스토리가 지저분해짐 vs 추적 가능"의 트레이드오프에서 *추적*을 택했다(일반 머지만 허용).
- **프론트↔백엔드는 서버사이드 rewrite 프록시로 한 도메인에 통합한다.** 브라우저는 항상 한 도메인으로만 요청하고, 프론트 서버(Next.js)가 그걸 백엔드로 프록시. 효과 둘 — ① CORS 설정이 아예 불필요(서버↔서버 통신이라 브라우저 CORS 정책이 적용 안 됨), ② 실제 백엔드 주소가 외부에 노출 안 됨(은닉). 서브도메인 분리(api.…)보다 단순·안전 → 그래서 이 방식을 택함.
- **stage/prod를 나누는 진짜 목적은 "환경이 둘"이 아니라 데이터 격리다.** 테스트 요청이 운영 DB를 오염시키지 않게 dev=테스트·main=운영으로 가른다. 분리 자체가 목적이 아니라 *데이터 혼동 방지*가 목적 — 그래서 프론트도 테스트는 테스트 백엔드로 보낸다.
- **이미 떠 있는 환경엔 로컬에 복제하지 말고 직접 붙어라.** 프론트는 로컬에 백엔드를 띄우지 않고 staging 서버에 바로 연결해 개발 — 셋업 마찰을 없애고 통합을 늘 실물로 검증. 환경이 상시 떠 있다는 전제가 깔리면 "각자 로컬에 남의 서버 재현"이 사라진다.
- **왜 SQLite인가 — DB 선택은 트래픽 규모가 아니라 읽기/쓰기 비율로 먼저 가른다.** 우리 홈피는 정적 콘텐츠(소개·멤버·활동·블로그) 위주라 *읽기가 대부분, 쓰기는 드물다*. 그래서 DB 역할 자체가 작고, SQLite의 single-writer 제한이 안 아프다 — 거기에 단일 무료 노드에 별도 DB를 안 늘려 운영이 단순해지는 이득까지. 유일한 경계는 *쓰기 폭주 한 곳*(모집 신청 버스트), 그것도 동아리 규모면 SQLite가 가볍게 처리. "규모 작으니까"가 아니라 "성격이 읽기 위주니까"가 진짜 이유다.
- **SQLite의 "한 번에 한 쓰기"는 직렬이지 저속이 아니다 — 그리고 앱 인스턴스 1개를 전제한다.** 병목은 창구 수가 아니라 트랜잭션 길이(작은 INSERT는 1ms라 직렬이어도 초당 수백). 단 한 `.db`를 여러 프로세스가 공유하면 깨지므로 *BE 앱 인스턴스는 1개*가 불변식 — 수평 확장(replica)이 필요해지는 순간이 곧 Postgres로 옮길 신호.
- **무료 클라우드 티어는 예고 없이 깎인다.** 오라클이 2026-06-15 Always Free 한도를 공지 없이 절반(4 OCPU/24GB→2 OCPU/12GB)으로 줄인 게 실증 — "현재 무료"이지 "영구 무료"가 아니다. 무료티어 위에 올릴 땐 한도의 절반만 써 여유를 두고, 깨졌을 때의 fallback(유료 전환 비용)을 미리 숫자로 알아둔다.
- **DB를 앱과 같은 노드에 올리면 백업이 필수가 된다.** 단일 노드 동거는 단순하지만 앱과 DB가 운명을 공유 — 노드가 죽으면 데이터도 같이 멈춘다. "있으면 좋은" 백업이 "없으면 안 되는" 백업으로 바뀐다.
- **메일 발송 같은 "대행" 서비스는 우리 서버가 아니라 대행사가 실제 발신자다.** BE는 SMTP AUTH로 릴레이(OCI Email Delivery)에 인증만 하고, 실제 인터넷 발송·발신 IP 평판·SPF/DKIM 서명은 전부 대행사 쪽에서 일어난다 — 그래서 SPF 레코드도 우리 서버 IP가 아니라 대행사의 IP 대역(`include:`)을 가리키게 등록한다. 직접 발송(Postfix 등)이 기술적으로 가능해도 IP 평판·포트25 차단·워밍업 부담 때문에 실전에서 밀리는 것도 이 구조 때문 — 새 외부발신 연동을 볼 때 "누가 실제로 인터넷에 나가는 주체인가"부터 확인하면 나머지(등록·인증 절차)가 왜 그런 모양인지 풀린다.
- **OCI Email Delivery는 SPF와 DKIM의 DNS 재확인 주기가 다르다.** DNS에 SPF(TXT)·DKIM(CNAME)을 똑같이 정확하게 등록해도, SPF는 거의 즉시 `is-spf: true`로 반영되는 반면 DKIM은 한동안 `NEEDS_ATTENTION(NEED_DNS)`로 남아있었다(같은 시점에 `nslookup`으로 DNS 자체는 이미 정확히 전파된 걸 확인했는데도). 두 검증기가 서로 다른 주기로 도는 것으로 보임(이번 케이스는 생성 후 약 65분) — DNS를 다시 점검하며 헤매기 전에, "SPF는 됐는데 DKIM만 안 됐다"는 시차 자체가 정상일 수 있다는 걸 먼저 의심한다.
- **DMARC 레코드가 아예 없으면 "FAIL"로 뜬다 — 정렬 실패가 아니라 정책 부재다.** SPF·DKIM이 둘 다 PASS여도 `_dmarc.<도메인>` TXT가 없으면 수신 측(Gmail)이 DMARC를 FAIL로 표시한다. 이걸 "설정이 잘못됐다"로 오독하기 쉬운데, 실제로는 "적용할 정책 자체가 없다"는 뜻 — `_dmarc` TXT(`v=DMARC1; p=none; rua=mailto:...`)만 추가하면 대개 바로 PASS로 바뀐다(우리 케이스는 DKIM 정렬만으로 통과). `p=none`으로 시작해 관찰만 하다가 안정되면 `quarantine`→`reject`로 단계적으로 올리는 게 안전한 순서.
- **인증을 뒤로 미루면 `permitAll` 부채가 SecurityConfig 한곳에 쌓인다 — 그게 오히려 안전하다.** 피드 백엔드(#66)는 어드민 인증을 #74로 미루면서 어드민·매직링크 엔드포인트를 전부 `permitAll`+`TODO: 인증 도입 시 좁히기`로 임시 개방했다. 도메인마다 인증을 급조하지 않고 개방 지점을 한 파일에 모아두면, 나중에 인증이 들어올 때 *좁혀야 할 목록이 이미 한 곳에 다 보인다* — 흩어진 임시 인증보다 회수가 쉽다. 단 불변식: **이 상태로 prod에 나가면 무방비**라, 개방을 만든 미션의 result에 "어느 미션이 언제까지 좁힌다"를 명시하고 그 미션의 완료조건에 박아 승계를 강제해야 한다(안 그러면 TODO가 영구화).
- **계약(`shared/types`)을 먼저 열면 병렬이 핸드오프 릴레이를 이긴다 — 실측.** #66 백엔드가 `shared/types/feed.ts`(Post·Comment·매직링크 + `SpringPage<T>` 래퍼)를 먼저 머지해두니, 프론트(#78)가 백엔드 구현 완료를 안 기다리고 타입만 보고 화면을 붙일 수 있게 됐다. 계약을 문서가 아니라 코드로 두는 규칙(`shared/`)이 이 병렬을 가능케 한 전제 — 문서였으면 drift로 못 믿어 결국 기다렸을 것.
- **서버 컴포넌트에서도 "`/api/*`만 호출" 규칙을 지키려면 `next/headers`의 `headers()`로 자기 자신을 향한 절대 URL을 만들어 fetch해야 한다.** `fetch()`는 상대경로를 못 받아서 서버 컴포넌트에서 `fetch('/api/posts')`는 그냥 에러난다 — 그렇다고 `BACKEND_URL`로 직접 백엔드를 호출하면 "API는 `/api/*`로만, 절대경로 금지" 규칙을 어기고 프록시를 우회하게 된다. `headers().get('host')` + 프로토콜로 `http(s)://이-서버-자신` 절대 URL을 만들고 그 위에 `/api/*`를 붙이면, 여전히 Next 서버 자신의 rewrite를 타면서(=백엔드 주소를 앱코드에 안 노출) fetch의 절대 URL 요구사항도 만족한다. SEO용 SSR·OG 메타(`generateMetadata`)가 필요한 페이지(피드 목록/상세, #78)에서 이 패턴이 필요했다 — 앞으로 SSR이 필요한 페이지(활동·멤버·프로젝트 등)에도 그대로 재사용 가능.
- **`eslint-plugin-react-hooks`의 `set-state-in-effect` 규칙은 "effect가 이름 있는 함수를 호출"하면 그 함수 내부 setState가 await 이후에 있어도 걸리지만, effect 안에 async IIFE를 바로 인라인하면 안 걸린다.** "마운트 시 fetch, 버튼으로 재시도"를 만들며 `useCallback`으로 뺀 fetch 함수를 effect와 버튼 onClick 양쪽에서 재사용하려다 계속 lint 에러가 났다 — 원인은 await 위치가 아니라 "effect가 함수 참조를 호출하는 형태" 자체였다. 해결: effect 안엔 `(async () => {...})()` 인라인만 두고, 재시도는 `reloadIndex` 같은 상태를 증가시켜 effect의 dependency로 재실행시키는 방식으로 우회(#78, 댓글 목록 재조회 구현 중 실측).

## 디자인
- **디자인은 코드 홈이 없어 가장 애매하다.** "done"을 *PM 승인*으로 정의해 주관적 모호함을 죽인다.
- **디자인은 크리티컬 패스 — "거칠게 빨리"가 병목을 막는다.** 다른 팀을 여는 산출물(IA·와이어)을 먼저, 완성도는 나중.
- **모르는 영역은 지시하지 말고 결과만 판단한다.** PM이 디자인을 art-direct하면 망친다.
- **와이어프레임 평가는 카피가 아니라 구조·흐름·상태로 한다.** 플레이스홀더("Sub text")는 와이어의 정상 상태 — 와이어가 답할 건 "메시지 들어갈 *자리*가 있나"지 "메시지 내용"이 아니다. 콘텐츠를 구조 결함으로 착각하면 디자인팀에 헛 피드백이 간다.

- **아키텍처 분석을 코드 grep만으로 하면, 이미 `pm/docs`에 있는 PM 결정을 "새로 발견한 리스크"로 잘못 보고하게 된다.** Reze 논의록(계정 정책 등)을 원자 문제로 분해하며 "Admin/Member 계정 통합 여부가 안 정해졌다", "#94가 이 설계와 충돌한다"를 미해결 리스크로 정리했는데, 실제로는 `pm/docs/product-map.md`에 통합 방식(멤버 로그인 = #74 위 role 티어, 별도 시스템 아님)이 이미 확정돼 있었고, #94도 이미 2026-07-10에 PM이 코멘트로 명시적 홀드를 걸어둔 상태였다 — 코드와 이슈 목록만 훑고 `pm/docs/*.md`를 안 읽어서 생긴 오분석. 역으로 발견한 진짜 갭: **그 홀드가 이슈 라벨·마감일엔 전혀 반영이 안 돼 있어서**, 코멘트를 안 읽으면(예: 보드만 보면) 마감 임박한 정상 진행 작업처럼 보인다 — 코멘트에만 있는 홀드는 라벨화하지 않으면 조용히 무시될 위험이 있다. → 백엔드/제품 구조를 분석할 땐 코드 전에 `pm/docs/`(특히 `product-map.md`류 "구조의 단일 기준" 문서)부터 확인할 것.
- **의도적으로 깃에 안 올리는 운영 정보는 "찾을 단일 출처"가 사라진다.** 스테이징 URL을 obscurity 때문에 깃허브에 안 올리니, 정작 우리도 "어딜 봐야 알지?"가 됐다. 은닉과 발견가능성은 트레이드오프 — 둘 다 잡으려면 *레포 밖 비공개 단일 출처*(gitignore된 로컬·비밀번호 관리자)를 정해 둬야 한다. 안 그러면 카톡에만 흩어진다.
- **맥락 축적 ≠ 대화 로깅.** 결정에 이른 과정 말고 *배운 것*을 남긴다.
- **시스템 프롬프트(CLAUDE.md)엔 상태를 넣지 않는다.** 상태는 썩는다 — 진행은 GitHub로.

- **하네스는 맥락을 잃지 않는 선에서 최대한 작게.** 안 쓰는 문서·중복 규칙은 노이즈로 죽는다.

- **공통 운영 루프(첫날·이슈사이클·막힘·PR)는 각 역할 문서에 흩지 말고 한 곳에 모아 링크한다.** 흩으면 중복·drift, 한 곳이면 단일 진실.
