# P (plan) — 승인된 방향을 구체화한다

RP 게이트를 통과한 방향을, 손으로 옮길 수 있는 계획으로 좁힌다. 새 방향을 여기서 만들지 마라 — 그건 R로 되돌아갈 일이다.

## 렌즈
- **파일 단위로.** 어떤 파일을 새로/고치나(가능한 기존 파일 수정 우선 — mindset). 각 변경이 어떤 Done 항목에 대응하는지 매핑.
- **가장 작은 경로.** Done을 충족하는 최소 변경. 미래 확장·설정·훅 넣지 않는다(YAGNI).
- **계약 변경이면 표시.** `shared/` 타입을 바꿔야 하면 그건 다른 팀에 영향 → PM 게이트 사안일 수 있다(ask-pm 고려).
- **검증 방법을 미리 정한다.** 각 Done을 Q에서 무엇으로 확인할지(테스트/실행/화면)를 계획에 적어둔다 — 나중에 셀프 QA가 쉬워진다.

## 산출
변경 파일 목록 + 각 변경↔Done 매핑 + Q 검증 방법. (사람에게 매번 확인받지 않는다 — 자율 구간. I로 바로 넘어간다.)

## 함정
- 계획이 Done을 넘어 커짐(스코프 크리프) → 잘라낸다.
- "이 구조가 더 멋져서" 리팩터 끼워넣기 → mindset 위반. 미션 밖.
