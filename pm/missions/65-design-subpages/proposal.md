## [디자인] — 프로젝트·블로그 서브페이지 시안 (목록 + 개별)

메인 랜딩 시안(#5)에서 여섯 섹션을 다 그려주셔서, 소개·운영진·활동·모집은 랜딩 한 장으로 이미 완성됐어요. 이번엔 그중 **콘텐츠가 계속 쌓이는 두 곳 — 프로젝트와 블로그 — 의 전용 페이지**를 그립니다.

---

### Deliverable

Figma "Ll 공식 홈페이지" 파일 안 새 섹션에, 아래 네 화면을 그려주세요. **데스크탑·모바일 둘 다**, 메인에서 정한 룩앤필(다크 + 레드 그라디언트)과 컴포넌트를 그대로 이어서.

- **프로젝트 목록** — 동아리가 만든 프로젝트 전체를 훑는 페이지 (랜딩의 캐러셀 프리뷰보다 넓게).
- **프로젝트 개별 상세** — 프로젝트 하나를 깊게 보는 페이지 (무엇을 담을지 — 스크린샷·소개·팀·기술·링크 등 — 는 디자인팀이 판단).
- **블로그 목록** — 글을 훑는 페이지.
- **블로그 개별 글** — 글 하나를 읽는 페이지 (본문 + 이미지).

### Why it matters

> [!IMPORTANT]
> 소개·운영진·활동·모집은 정보가 고정이라 랜딩 섹션으로 충분하지만, **프로젝트와 블로그는 콘텐츠가 계속 늘어나요.** 랜딩의 미리보기만으론 다 못 담고, 방문자가 "전체를 훑고 → 하나를 깊게 보는" 흐름이 필요합니다. 특히 **프로젝트는 "멋사가 진짜 뭔가 만드는 동아리"라는 증거의 핵심** — 1순위 방문자(코딩을 시작하고 싶은 경희대생)를 설득하는 가장 강한 면이에요.

### Done

- [ ] 프로젝트 **목록**과 **개별 상세**가 데스크탑·모바일 둘 다 시안으로 나온다
- [ ] 블로그 **목록**과 **개별 글**이 데스크탑·모바일 둘 다 시안으로 나온다
- [ ] 랜딩의 프로젝트·블로그 섹션(프리뷰)에서 이 전용 페이지로 **넘어가는 연결**(더보기·카드 클릭)이 시안에 보인다
- [ ] 메인의 색·타이포·컴포넌트를 그대로 이어 써서, **랜딩과 서브페이지가 한 사이트로** 보인다

### Notes

> [!TIP]
> **프로젝트를 먼저** 공유해주세요 (얼굴 가치가 가장 높음). 완성해서 한 번에 주지 말고 나오는 대로 넘겨야 프론트가 빨리 붙어요. 부분 공유 환영.

- 나머지 네 기둥(소개·운영진·활동·모집)은 **이번 범위가 아니에요** — 랜딩 섹션으로 이미 완료. 여기 손대지 마세요.
- **블로그 개별 글은 곧 만들 "피드" 글 화면(#59)과 사실상 같은 형태**예요. 두 번 그리지 않게, 같은 글 화면으로 통일할 수 있는지 프론트팀과 먼저 맞춰주세요.
- 개별 프로젝트 상세에 들어갈 데이터 형태(필드)는 프론트(일하·현정)와 직접 상의. 색·폰트·컴포넌트 인계 형식도 프론트팀과 직접.
- 이번 범위는 **웹(데스크탑 + 모바일 웹)** 까지. 별도 네이티브 앱 화면은 범위 밖.
