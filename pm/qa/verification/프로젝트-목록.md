## 근거 (시나리오별 상세)

### PROJ-001 — 익명이 `/projects` 직접 접속 → 플레이스홀더

무엇을 확인했나: dev 브랜치의 실제 페이지 파일과, 랜딩에서 프로젝트를 보여주는 컴포넌트를 열어 실데이터를 그리는지 확인했다.

어느 파일을 봤나:
- `frontend/src/app/projects/page.tsx` — 전체가 이 7줄이다.
  ```tsx
  export default function ProjectsPage() {
    return (
      <main className="min-h-screen flex items-center justify-center">
        <p className="text-gray-400 text-sm">프로젝트 — 준비 중이에요.</p>
      </main>
    );
  }
  ```
  fetch·API 호출·데이터 매핑이 전혀 없다. 문자 그대로 "프로젝트 — 준비 중이에요." 한 줄짜리 플레이스홀더.
- `frontend/src/components/sections/Project.tsx` — 랜딩의 프로젝트 섹션. 카드 모양(가운데 큰 카드 + 양옆 사다리꼴 카드 6개)은 있지만, 가운데 카드는 `bg-[#d9d9d9]` 회색 배경 div 하나뿐이고 양옆 카드는 `/project/rect5826.svg`·`rect5830.svg` 장식 SVG를 씌운 것뿐이다. 제목·이미지·API 호출 어디에도 실제 프로젝트 데이터가 들어가지 않는다.
- `frontend/src` 전체를 `projects` 키워드로 검색해도 API 클라이언트나 fetch 로직을 참조하는 파일은 `page.tsx` 자신 말고는 없다(`grep -rl projects frontend/src` → `frontend/src/app/projects/page.tsx` 단 하나).

왜 그렇게 판단했나: 시나리오가 기대하는 "'프로젝트 — 준비 중이에요.' 플레이스홀더 + 랜딩 프로젝트 섹션도 빈 카드"가 코드 그대로 존재한다. 별도 추론 없이 파일 내용 자체가 증거다.

### PROJ-002 — 익명이 `GET /api/projects` 호출 → hidden=false만 최신순, FE 화면은 아직 없음

무엇을 확인했나: PR #121(open, `feat/117-member-auth` → `dev`, 아직 머지 안 됨)에 포함된 프로젝트 쇼케이스 API(#119) 컨트롤러·서비스·리포지토리 로직.

어느 파일을 봤나(전부 PR #121 diff):
- `backend/src/main/java/likelion/khu/website/project/ProjectController.java` (신규 파일) — `@GetMapping("/api/projects")`가 `projectService.getPublicList()`를 그대로 반환. `@PreAuthorize` 없음(공개).
- `backend/src/main/java/likelion/khu/website/project/ProjectService.java`의 `getPublicList()`:
  ```java
  public List<ProjectSummaryResponse> getPublicList() {
      return projectRepository.findAllByHiddenFalseOrderByCreatedAtDesc().stream()
              .map(project -> ProjectSummaryResponse.from(project, representativeImageUrl(project.getId())))
              .toList();
  }
  ```
  리포지토리 메서드 이름 자체가 `findAllByHiddenFalseOrderByCreatedAtDesc` — Spring Data 쿼리 메서드라 이름 그대로 "hidden=false만, createdAt 역순"이 SQL로 번역된다.
- `backend/src/test/java/likelion/khu/website/project/ProjectControllerTest.java`의 `list_Public_ExcludesHiddenProjects` 테스트가 숨긴 프로젝트를 만들고 `GET /api/projects` 호출 시 `$.length()`가 0임을 검증하고, `list_Public_ReturnsRepresentativeImageUrl`이 대표이미지 URL·제목이 응답에 들어가는지 검증한다.
- `backend/src/main/java/likelion/khu/website/config/SecurityConfig.java` diff에 `.requestMatchers(HttpMethod.GET, "/api/projects", "/api/projects/*").permitAll()`이 추가돼 있어 익명이 인증 없이 호출 가능함도 확인했다.
- FE 쪽은 PROJ-001에서 확인한 대로 `/api/projects`를 호출하는 코드가 frontend에 없다 — "FE 화면은 아직 없음"이 그대로 사실이다.

왜 그렇게 판단했나: 로직·필터링·공개 권한·테스트가 diff 안에 모두 갖춰져 있어 "PR #121 코드상으로는 pass"로 판단했다. 다만 **PR #121은 아직 dev에 머지되지 않은 open 상태**이므로 지금 dev 브랜치를 배포해도 이 엔드포인트는 존재하지 않는다 — 그 점을 결과에 괄호로 명시했다.

### PROJ-004 (목록 관점) — 숨김/복원/삭제 후 목록 API 재호출 시 반영

무엇을 확인했나: 어드민의 숨김·복원, 멤버의 삭제가 `getPublicList()`가 읽는 데이터에 실제로 영향을 주는지.

어느 파일을 봤나(전부 PR #121 diff):
- `ProjectController.setHidden()` — `@PreAuthorize("hasAnyRole('ADMIN','SUPER_ADMIN')")`, `PATCH /api/admin/projects/{id}/hidden` → `projectService.setHidden(id, request.getHidden())`.
- `ProjectService.setHidden()`:
  ```java
  public void setHidden(Long id, boolean hidden) {
      Project project = findProjectOrThrow(id);
      project.setHidden(hidden);
  }
  ```
  `hidden`을 그대로 엔티티에 세팅한다 — `true`(숨김)든 `false`(복원)든 대칭적으로 동작하는 하나의 로직이다. `getPublicList()`는 매 호출마다 `findAllByHiddenFalseOrderByCreatedAtDesc()`로 새로 조회하므로, `hidden`이 바뀌면 다음 목록 조회부터 즉시 반영된다(별도 캐시 없음).
- `ProjectController.delete()` → `ProjectService.delete()`:
  ```java
  public void delete(Long id, Long memberId) {
      findProjectOrThrow(id);
      requireParticipant(id, memberId);
      projectImageRepository.deleteAllByProjectId(id);
      projectParticipantRepository.deleteAllByProjectId(id);
      projectRepository.deleteById(id);
  }
  ```
  행 자체를 지우므로 목록에서도 당연히 빠진다. `requireParticipant()`가 참여자 본인인지 확인해 "멤버가 삭제"라는 시나리오 조건과 일치한다.
- 테스트 `hidden_ByAdmin_HidesFromPublicButKeptInStorage`가 숨김 처리 후 `GET /api/projects`가 `$.length()` 0을 반환함을 검증한다(DB 행은 안 지워지고 "숨김"만 됨 — 시나리오의 "존재는 하되 목록제외"와 일치).

왜 그렇게 판단했나: 숨김→제외, 복원(hidden=false로 재설정)→다시 노출, 삭제→행 자체 제거+목록 제외 세 경우 모두 `hidden` 플래그와 조회 쿼리 로직으로 대칭적·직접적으로 설명된다. 복원 케이스를 직접 검증하는 테스트 메서드는 diff에서 별도로 찾지 못했지만, `setHidden(id, hidden)`이 `true`/`false` 어느 값이든 동일 코드 경로로 처리하고 `getPublicList()`가 매번 재조회하는 구조라 "복원 시 다시 보임"은 숨김 로직의 대칭으로 코드상 자명하다고 판단했다.

### PROJ-005 (목록 관점) — 카드 구성(대표이미지1장+제목+한줄소개+기수+스택)

무엇을 확인했나: `find frontend/src/app/projects -type f`로 목록 화면 파일이 실제로 있는지, 있다면 카드 요소를 렌더링하는지.

어느 파일을 봤나: `frontend/src/app/projects/page.tsx` 한 파일뿐이고 PROJ-001에서 본 대로 플레이스홀더 문구 하나다. 카드 컴포넌트·리스트 렌더링·API 호출이 없다.

참고로 백엔드 쪽은 PR #121의 `ProjectSummaryResponse` DTO가 이미 카드에 필요한 필드를 전부 갖고 있다:
```java
public class ProjectSummaryResponse {
    private Long id;
    private String title;
    private String summary;
    private String representativeImageUrl;
    private Integer cohort;
    private Set<String> techStack;
    ...
}
```
`representativeImageUrl`은 서비스가 이미지 중 `representative=true`인 것 하나만 골라 조립해 넘긴다(대표이미지 1장 규칙).

왜 그렇게 판단했나: 시나리오는 "화면 구현되면"이 전제인 spec-only 항목이다. 화면이 없으므로 카드 렌더링 자체를 검증할 수 없어 not-run이 맞다. 다만 근거를 구체화하기 위해 "화면이 생기면 쓸 데이터는 이미 API가 제공한다"는 점까지 확인해 남겼다 — FE 미착수의 원인이 API 부재가 아니라 화면 작업 자체가 아직 없어서임을 분명히 하기 위해서다.

### PROJ-006 (목록 관점) — 익명이 쓰기 API 직접 호출 → 401 UNAUTHENTICATED

무엇을 확인했나: `/api/projects`에 대한 POST(생성)와 `/api/admin/projects/{id}/hidden`에 대한 PATCH(숨김)를 익명이 호출했을 때의 인증 경계.

어느 파일을 봤나(전부 PR #121 diff, 목록 관련 쓰기만):
- `SecurityConfig.java` diff — 이번 PR이 추가한 permitAll은 딱 이 한 줄뿐이다: `.requestMatchers(HttpMethod.GET, "/api/projects", "/api/projects/*").permitAll()`. GET만 열렸고, POST/PATCH/DELETE는 이 matcher에 안 걸리므로 파일 하단의 기존 규칙 `.anyRequest().authenticated()`(dev의 현재 `SecurityConfig.java` 75번째 줄, PR 대상 밖이라 변경 없음)로 떨어진다.
- 같은 파일의 `exceptionHandling`에서 `authenticationEntryPoint`가 미인증 요청에 `401 {"success":false,"code":"UNAUTHENTICATED","message":"로그인이 필요해요."}`를 직접 JSON으로 써서 반환한다(`@RestControllerAdvice`가 아니라 `ExceptionTranslationFilter` 레벨에서 처리 — 코드 주석에 이유가 적혀 있다).
- `ProjectController.create()`·`setHidden()`에 `@PreAuthorize`가 붙어 있지만, 인증 자체가 안 된 요청은 `@PreAuthorize` 평가 전에 `AuthenticationException`으로 걸러진다.
- `ProjectControllerTest.java`의 `create_Unauthenticated_Returns401`, `hidden_Unauthenticated_Returns401` 테스트가 각각 인증 쿠키 없이 호출 시 `status().isUnauthorized()`를 직접 검증한다.

왜 그렇게 판단했나: SecurityConfig의 permitAll 범위가 GET 두 개로 명시적으로 한정돼 있고, 나머지는 기존 anyRequest().authenticated() 규칙에 그대로 걸린다는 걸 diff로 확인했다. 테스트도 동일한 기대를 검증하고 있어 코드와 테스트가 일치한다. PR이 아직 dev에 머지되지 않았다는 점은 위와 동일하게 남긴다.
