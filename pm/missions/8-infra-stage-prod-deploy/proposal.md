### 먼저 — stage / prod가 뭐예요? (처음이어도 괜찮아요!)
이 개념이 처음일 수 있어요. 짧게:
- **prod(프로덕션)** = 실제 사용자가 보는 **라이브** 서버. 검증된 것만 올라가요.
- **stage(스테이징)** = ① prod에 올리기 전 **리허설** 서버이자, ② **(이번 핵심)** 프론트가 개발하면서 **연결해 쓰는 백엔드 서버**예요.
- **왜 나누냐면** — prod를 안 깨고 미리 검증하고, 개발 중인 프론트가 안전하게 붙을 곳을 주려고요.

감 잡는 글: [개발/스테이징/운영 서버 차이 (한국어)](https://understandkorea.com/개발-서버-스테이징-서버-운영-서버의-차이점/) · [dev·stage·prod 환경 (velog)](https://velog.io/@k7nsuy/개발환경에-대하여-devstageprod)

### 🎯 이번 핵심 목적 — 프론트가 붙는 stage 백엔드
프론트(일하·현정)는 로컬에서 화면을 만들면서, API 호출은 **stage 백엔드에 연결**해 개발해요 (mock 대신 진짜 서버). 그래서 stage 백엔드는:
- **항상 떠 있어야** 하고,
- 프론트 로컬에서 **외부로 호출이 닿아야** 해요 (공개 주소 + CORS 허용).

이게 stage의 **주목적**이에요. prod 리허설은 그 다음.

### + 추상화 (개발팀이 배포를 몰라도 되게)
개발팀이 도커·proxy·환경분리를 **몰라도** 일하게 만들어요. 그들에겐 "이렇게 하면 **stage**에 뜨고, 이렇게 하면 **prod**로 간다"만 보이면 돼요. 내부 복잡함은 **인프라가 숨겨주세요.**

### 정해진 제약
- 단일 무료 인스턴스에 **컨테이너로 stage/prod 격리** (각자 데이터·포트·자원제한).
- **prod는 보호**(검증된 것만) / **stage는 막 써도·리셋해도 OK** (CD 싸니 더러우면 다시 배포).
- 무료 경계(`infra/CLAUDE.md`) 유지.

### 협업 — 양 팀과
- **프론트와**: stage 백엔드 **주소·CORS**를 맞춰줘요 (프론트가 거기 연결해 개발).
- **백엔드와**: 앱을 컨테이너로 패키징하면 인프라 틀이 stage/prod에 올려요. 백엔드 초반이니 **지금은 더미(헬로월드)로 틀부터**, 크면 실앱 얹기.

### 어떻게는 장찬욱 몫 🙌
reverse proxy(nginx? Caddy?)·CD 도구(GitHub Actions?)·컴포즈 구조 — 다 본인이 정해요.
