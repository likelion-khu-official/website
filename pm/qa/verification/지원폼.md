## 근거 (시나리오별 상세)

### RECR-003 — 모집기간에 '지원하기' → /apply 전환
`find frontend/src/app -maxdepth 1 -type d`로 라우트 목록을 확인했다 — `recruit`·`projects`·`admin`·`blog`·`api`·`activities`·`members`뿐이고 `apply` 디렉터리가 없다. `grep -rli "apply" frontend/src`도 결과 없음(0건) — 코드 어디에도 "apply" 문자열이 없다. `frontend/src/components/sections/Recruit.tsx`(랜딩 모집 섹션)를 다시 봐도 pill 클릭 시 분기하는 건 `open` state 하나뿐이고 그 결과는 항상 `<NotificationForm />`이다 — "모집기간이면 지원폼, 아니면 알림폼"으로 갈리는 조건부 렌더링 자체가 없다(코드에 "모집기간 여부"를 나타내는 변수·API 호출이 전혀 없음 — 이는 `모집-알림.md` RECR-002 fail과도 같은 원인: 공개 모집 상태를 아는 방법이 FE에 없다). 따라서 '지원하기'로 전환되는 코드 경로가 존재하지 않는다.

### RECR-004 — 지원폼 작성·제출
프론트: 위와 동일하게 `apply` 관련 컴포넌트·페이지가 전혀 없다.
백엔드: `grep -rli "apply" backend/src/main`을 돌렸더니 `backend/src/main/java/likelion/khu/website/common/LocalDateTimeConverter.java` 1건만 걸렸는데, 열어 보니 JPA `AttributeConverter`의 `convertToDatabaseColumn`/`convertToEntityAttribute` 구현부에 있는 일반적인 코드 패턴(예: 날짜 변환 로직)에 "apply"라는 영어 단어가 우연히 포함된 것으로, 지원폼과 무관한 false positive다. 그 외 지원서 저장 엔티티·컨트롤러·서비스·질문 스키마 계약(`shared/types`) 어디에도 지원폼 관련 코드가 없다.
이슈 트래커도 확인했다 — `gh issue view 125`(지원폼 미션, `[FE·백엔드] 지원폼 — 모집기간 신규 부원 지원서`)는 **state: OPEN**이고 본문이 "완료기준(초안 — PM 확인 필요)"·"경계/확인 필요" 섹션으로 돼 있어 아직 계획 단계이지 착수된 상태가 아니다. `pm/features/모집-알림.md`가 인용하는 PR #126의 범위 밖 섹션도 "유저가 안내메일 링크를 눌러 지원서를 작성하는 부분(지원폼)은 별도 이슈 #125 — 이번 PR에 포함하지 않음"이라고 명시한다. 코드·이슈 상태 모두 미착수를 뒷받침해 `not-run`이 맞다.
