## 근거 (시나리오별 상세)

### BLOG-L01 — pass

`frontend/src/app/blog/page.tsx`(서버 컴포넌트)가 `frontend/src/lib/feedApi.ts:40-43`의 `getPosts(page, baseUrl)`을 호출한다. 여기서 `fetch(...,{ cache: 'no-store' })`를 명시적으로 쓰고 있어 매 요청마다 새로 가져온다(SSR + no-store).

백엔드 쪽은 `PostController.list()`(`backend/.../feed/post/PostController.java:31-35`)가 `postService.getPublishedPosts(pageable)`를 호출하고, `PostService.getPublishedPosts()`(`PostService.java:35-40`)는 `postRepository.findByStatusOrderByPublishedAtDesc(PostStatus.PUBLISHED, pageable)` — 리포지토리 메서드 이름 자체가 "PUBLISHED만, publishedAt 내림차순"이라 DRAFT·HIDDEN은 애초에 쿼리에 안 걸린다.

홈 복귀 링크는 `frontend/src/app/blog/layout.tsx:6-13` — `<Link href="/">← 멋쟁이사자처럼 경희대</Link>`가 블로그 영역 전체 레이아웃 헤더에 고정돼 있다(랜딩의 Nav와는 별개).

로딩 스켈레톤은 Next.js의 `loading.tsx` 컨벤션으로 구현돼 있다: `frontend/src/app/blog/loading.tsx`가 제목 바(`h-9 w-32`) + 카드 6개(`Array.from({length:6})`) 펄스 애니메이션을 렌더한다. then의 네 조건(SSR no-store, PUBLISHED만 내림차순, 홈 복귀 링크, 로딩 스켈레톤) 모두 코드로 확인됐다.

### BLOG-L04 — pass

`frontend/src/app/blog/page.tsx:18-19`:
```ts
const parsed = Number(pageParam);
const page = Number.isInteger(parsed) && parsed >= 0 ? parsed : 0;
```
`Number('-1')=-1`(정수지만 `>=0` 거짓) → 0, `Number('abc')=NaN`(`isInteger` 거짓) → 0, `Number('1.5')=1.5`(`isInteger` 거짓) → 0 — 셋 다 0으로 보정된다. 반면 `Number('999')=999`는 두 조건을 모두 만족해 **보정 없이 그대로 999가 백엔드에 전달**된다. Spring Data의 `Pageable`은 존재하지 않는 페이지 번호를 요청해도 예외 없이 빈 `content` 배열을 반환하는 게 표준 동작이라(별도 범위 검사 코드가 컨트롤러·서비스 어디에도 없음), 결과적으로 "글이 있어도 빈 상태가 보인다"가 성립한다. then의 두 조건 모두 확인됐다.

### BLOG-L07 — fail

`PostController.list()`(`PostController.java:32-33`)의 시그니처는 `@PageableDefault(size = 10) Pageable pageable`뿐이고, size 상한을 검사하는 코드가 컨트롤러·서비스 어디에도 없다. `backend/src/main/resources/application.yml`을 전체 확인했지만 `spring.data.web.pageable.max-page-size`를 설정하는 줄이 없다 — 즉 Spring Data 웹 지원의 **기본값 2000**이 그대로 적용된다. `?size=2000`은 그 상한 안이라 그대로 통과한다(더 큰 값은 자동으로 2000까지 잘림 — "무제한"은 아니지만 페이지당 2000개는 합리적 상한이라 보기 어렵다).

`?sort=없는필드`는 Spring Data가 `Sort` 객체로 바인딩하면서 실제 쿼리 실행 시점에 `PropertyReferenceException`을 던지는데, `backend/.../common/GlobalExceptionHandler.java` 전체를 읽어봐도 이 예외(또는 그 상위 타입)를 잡는 `@ExceptionHandler`가 없다. 즉 커스텀 `{success,message}` 응답이 아니라 Spring Boot 기본 에러 핸들링(`BasicErrorController`)으로 떨어진다 — then이 말하는 "500 Spring 기본 에러"와 일치한다(정확한 상태 코드가 400인지 500인지는 이 바인딩 예외가 어느 단계에서 던져지는지에 따라 갈릴 수 있어 실제 실행으로 재확인이 필요하지만, "커스텀 에러 핸들링이 없다"는 사실은 코드로 확정적으로 확인됨).

**두 조건(size 상한 없음, sort 필드 미검증) 모두 코드상 사실로 확인돼 fail.**
