# 디자인 팀 — 업무 방식

이 문서는 이 프로젝트에서 **디자인 팀이 일하는 방식**만 다룹니다.
(이 프로젝트가 무엇이고 왜 하는지는 [`about.md`](about.md), 구체 작업은 PM이 미션으로 전달합니다.)

> [!NOTE]
> 디자인은 저장소·코드 워크플로 **밖**이에요 — Figma가 전부입니다(GitHub 계정 필요 없음). 미션은 GitHub 이슈로 정리되지만 **PM이 카톡으로 링크를 전달**해 드려요. 막히거나 *무엇을·왜*에 대한 결정이 필요하면 편하게 **카톡으로 PM에게** 물어보세요 — 학습이 1순위라, 모르는 건 묻는 게 맞습니다.

---

## 1. 책임과 권한

디자인 팀은 **브랜드 · 디자인 시스템 · IA(사이트 구조) · 와이어프레임 · 전 화면 시안**을 담당합니다. 작업은 **Figma**에서 이루어지며, Figma가 디자인의 단일 진실입니다(저장소에는 디자인 폴더가 없습니다).

PM은 *무엇을 전달할지(메시지) · 우선순위 · IA의 방향*을 제시하고, **그것을 어떻게 표현할지 — 레이아웃 · 시각 · 흐름 — 는 디자인 팀의 영역**입니다. PM은 디자인을 지시하지 않으며, 결과를 목표에 비추어 판단합니다.

## 2. 일하는 방식

| 원칙 | 내용 |
|---|---|
| **크리티컬 패스** | 프론트는 디자인의 IA·시안 위에서 화면을 만듭니다. 디자인의 첫 산출물(IA·랜딩 와이어프레임)이 다른 팀을 여는 길목이므로, 가장 먼저 움직입니다. |
| **거칠게 먼저** | IA·와이어프레임처럼 *다른 팀을 여는 산출물*을 먼저 빠르게 내고, 시각 완성도는 그 뒤에 높입니다. 디자인이 병목이 되지 않도록 — "완벽하게 늦게"보다 **"거칠어도 빨리"**입니다. |
| **의도 vs 구현** | 디자인은 *의도*(위계·강조·구성·브랜드 느낌)를, 프론트는 *구현*(반응형·상태·미세조정)을 소유합니다. 프론트가 현실에 맞게 적응하는 것은 정상이며, **의도를 바꿔야 할 때만 함께 논의**합니다. |
| **결과 QA** | 시안을 넘기고 끝이 아니라, 프론트가 구현한 결과를 디자인이 확인하고 다듬는 **왕복 루프**로 일합니다. |
| **완료(Done)의 기준** | 코드처럼 자동 판정이 없으므로, **"PM이 목표에 부합한다고 승인 + 프론트가 쓸 수 있게 시안을 넘김"**을 완료로 봅니다. |

## 3. 소통 방식

| 상황 | 채널 |
|---|---|
| **미션 전달** | PM이 미션을 GitHub 이슈로 정리하고 **카톡으로 링크를 전달**합니다. (이슈는 PM의 로드맵이라 여닫는 건 PM) |
| **시안 공유·리뷰** | Figma 링크로 공유하고, 프론트·PM과 **Figma·카카오톡에서 직접** 리뷰합니다. |
| **에셋 전달** | 폰트·이미지·OG·파비콘 같은 정적 에셋은 **프론트(필요하면 인프라)와 직접** 주고받습니다. |

## 4. PM(김우진)의 역할

| 한다 | 하지 않는다 |
|---|---|
| 메시지·우선순위·IA 방향 결정 | 시각·레이아웃 지시 (디자인은 팀의 전문 영역) |
| 시안을 목표에 비추어 승인·방향 제시 | 어깨너머 아트디렉션 |
| 미션 전달, 팀 간 통합 | — |

> PM은 *무엇을 전달할지*를 정하고 결과를 판단하는 사람이며, *어떻게 보일지*는 디자인 팀이 결정합니다.
