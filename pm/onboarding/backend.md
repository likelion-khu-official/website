# 백엔드 팀 — 업무 방식

이 문서는 이 프로젝트에서 **백엔드 팀이 일하는 방식**만 다룹니다.
(프로젝트 배경·목표는 `pm/docs/brief.md`, 첫 미션은 GitHub 이슈로 전달됩니다.)

> [!NOTE]
> 저장소 구조와 Claude로 일하는 법은 [**개발자 온보딩 — 모노레포 & AI-native**](dev-monorepo-ai.md)를 함께 읽어주세요.

---

## 1. 책임과 권한

백엔드 팀은 **데이터 모델 · API · 인증 · admin · 지원폼 · 모집 상태**를 담당합니다.

**무엇을 만들지는 PM의 요구에 따릅니다.** 기능·우선순위·범위는 PM이 정하며, 백엔드는 그 요구에 맞춰 작업합니다.

## 2. 일하는 방식

| 원칙 | 내용 |
|---|---|
| **병렬 (릴레이 아님)** | 디자인·프론트의 완성을 기다리지 않습니다. IA와 데이터 요구만 있으면 곧바로 시작합니다. |
| **계약은 코드로** | 프론트와 주고받는 API는 `shared/`에 **타입으로** 정의합니다(문서로 옮겨 적지 않음). 프론트는 백엔드 완성 전에 그 타입으로 **mock**을 사용해 병렬로 진행합니다. |
| **완료(Done)의 기준** | "테스트 통과"가 아니라 **"프론트가 mock 없이 이 API를 연동할 수 있는가"**입니다. |

## 3. 소통 방식

| 상황 | 채널 |
|---|---|
| **미션 전달** | PM이 GitHub 이슈로 전달합니다: `[BE] 미션 (~목표일)`. 이슈는 PM의 로드맵이므로 팀이 열고 닫지 않습니다. |
| **기술 조율** | 프론트와의 API 협의 등은 **PR·카카오톡에서 직접** 진행합니다. PM을 거치지 않습니다. |
| **인프라** | 배포·인프라 관련은 **장찬욱(인프라)**과 직접 소통합니다. 해당 영역은 찬욱이 주로 담당합니다. |

## 4. PM(김우진)의 역할

| 한다 | 하지 않는다 |
|---|---|
| 무엇·왜·우선순위 결정 | "어떻게"에 대한 지시 (구현은 팀의 몫) |
| 미션 전달, 막힌 것 해소 | 팀 간 기술 인터페이스 중계 (팀끼리 직접) |
| 콘텐츠 수급, 팀 간 통합 | 코드 마이크로매니징 |

> PM은 방향과 우선순위를 제시하는 사람이며, 팀의 작업을 어깨너머로 관리하지 않습니다.
