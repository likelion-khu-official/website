# claim-mission — 내 미션을 받아 끝까지 민다

팀원의 표면은 한 마디다: **"나한테 할당된 미션 수행하자."** 그 뒤의 배관(신원·PAT·gh·라벨·보드 item-id)은 전부 `scripts/`가 처리한다. 너(Claude)는 스크립트를 부르고, 결과를 사람 말로 옮기고, 게이트에서 멈춰 사람에게 넘긴다.

> 대상은 초보일 수 있다. **gh 명령·토큰·필드 id를 사용자에게 보이지 마라.** 사용자가 볼 건 "누구세요? / 이 미션 맞아요? / 이대로 진행할까요?" 정도의 질문뿐이다.

---

## 계약 (어기면 QA에서 터진다)
불변식 전문은 `kb/contract.md`. 핵심만: 레포 `likelion-khu-official/website`, 보드 project #1. 라우팅 = **assignee**. 미션 뱃지 = 라벨 `roadmap` + 팀 라벨. 신원 = 레포 루트 `/.identity.local.yml`(키 `github_handle`·`pat`, gitignore됨). 이름·팀은 여기 안 적고 `pm/roster.yml`에서 handle로 파생.

---

## 흐름

### Step 0 — 신원 확인
`scripts/check-identity.sh` 를 돌린다.
- `OK: <login>` → 신원 확정. 바로 Step 1.
- `NO_IDENTITY: …` → **바로 묻지 마라.** 먼저 아래 "Step 0-a 로컬 토큰 발견"으로 이 머신에 이미 있는 토큰을 찾는다. 없을 때만 인터뷰.
- `PAT_INVALID` / `HANDLE_MISMATCH` / `BAD_IDENTITY` → 사유를 사람 말로 옮기고, 인터뷰의 해당 단계(PAT 재발급 또는 handle 재선택)로 고쳐준다. **PAT 값은 절대 되묻지도, 출력하지도 마라.**

### Step 0-a — 로컬 토큰 자동 발견 (NO_IDENTITY면 인터뷰 전에 먼저)
`scripts/discover-token.sh` 를 돌린다. 이 머신에 이미 쓸 수 있는 GitHub 토큰(대개 `gh` 로그인)이 있으면 찾아 신원 파일을 자동으로 만든다 — 초보에게 PAT를 만들라고 하기 전에 마찰을 없앤다.
- `FOUND:<login>` → 머신에 유효 토큰이 있었다. `<login>`이 `pm/roster.yml`에 있는지 확인하고, 사용자에게 **"이 계정(<login>)으로 진행할게요, 맞죠?"** 한 번만 확인. 맞으면 `check-identity.sh` 재확인 → Step 1. (roster에 없거나 "아니오"면 그 파일을 지우고 아래 인터뷰로.)
- `FOUND:<login>:MISSING_PROJECT_SCOPE` → 위와 같되, **"이 토큰엔 보드 이동 권한(project)이 안 보여요 — 나중에 착수(Step 2)나 마감(Step 4)이 막히면 토큰에 project 권한을 더해주세요"** 라고 미리 귀띔.
- `NONE` → 머신에 쓸 토큰이 없다. 아래 인터뷰로.

### Step 0-b 인터뷰 (로컬에 토큰이 없을 때만 — 아주 친절히)
1. **안심시킨다:** "처음이시네요. 딱 한 번만 신원을 잡아두면 다음부턴 '나한테 할당된 미션 수행하자' 한 마디면 됩니다."
2. **누구세요?** `pm/roster.yml`을 읽어 **handle이 있는 멤버만**(디자인·null 제외) 선택지로 제시하고 `AskUserQuestion`으로 고르게 한다. 고른 사람의 `handle`을 확정. **본인이 목록에 없으면**(디자인처럼 GitHub 없음, 또는 명부 밖) → "claim-mission은 GitHub 기반이라 이 방식의 대상이 아니에요. 디자인 작업은 카톡·Figma로 받습니다"라고 친절히 알리고 **여기서 멈춘다**(억지로 진행 금지).
3. **GitHub PAT 만드는 법** (단계별, 초보 기준):
   - 브라우저에서 https://github.com/settings/tokens/new 로 간다 (로그인 필요).
   - Note에 아무 이름(예: `likelion-claim`). Expiration은 적당히(90일 등).
   - **스코프 체크: `repo` 전체 + `project`** 두 개. (이슈를 읽고 보드 카드를 옮기려면 이 둘이 필요하다.)
   - 맨 아래 **Generate token** → 나오는 `ghp_…`(또는 `github_pat_…`) 문자열을 복사. 이 화면을 벗어나면 다시 못 보니 지금 복사.
4. **저장:** 레포 루트에 `/.identity.local.yml`을 만든다. 내용은 딱 두 줄:
   ```
   github_handle: <고른 handle>
   pat: <붙여넣은 토큰>
   ```
   이 파일은 `.gitignore`에 있어 **커밋되지 않는다**(개인 비밀). 이름·팀은 여기 안 적는다 — roster에서 파생.
5. **검증:** 다시 `scripts/check-identity.sh`. `OK`면 통과. `HANDLE_MISMATCH`면 2번(사람 선택)이 틀렸거나 다른 계정 토큰 → 바로잡는다. `PAT_INVALID`면 스코프 빠졌을 확률 → 3번 다시.
6. 통과하면 곧바로 Step 1.

> PAT를 대화에 다시 노출하지 마라. 파일에 쓸 때 외엔 값을 입에 담지 않는다.

### Step 1 — 내 미션 가져오기
`scripts/fetch-my-mission.sh` → `count=<N>` + 이슈 JSON.
- **0개:** 아직 배정된 미션이 없다. "PM(@xhae123)이 아직 안 던졌거나, 다른 사람에게 갔어요." — 끝. (초조하게 만들지 말 것.)
- **1개:** 그 미션으로 진행. 제목·`Deliverable`·`Done`을 사람 말로 요약해 "이 미션 맞죠?" 한 번 확인.
- **N개:** `AskUserQuestion`으로 어느 미션부터 할지 고르게 한다(제목으로).

### Step 2 — 착수 (보드 한 칸)
고른 미션 #n에 대해 `scripts/start-mission.sh <n>` → 보드 카드 Todo→In Progress. (착수 표시. 완료 마감은 Step 4에서 **미션을 연 이 사람**이 한다.)

### Step 3 — 수행: R → P → 【RP 게이트】 → I → Q → 【IQ 게이트】  (순서 고정 · 게이트는 하드 스톱)

먼저 `kb/loop.md`(루프 규약)·`kb/mindset.md`(우리 규모 개발 철학)를 적재. 각 페이즈에서 렌즈 kb를 **필요할 때만** 얕게 읽는다. **순서를 지켜라 — 게이트를 통과하기 전엔 다음 단계로 못 넘어간다.**

> ⚠️ **미션을 채간 사람은 이 미션의 맥락을 거의 모른다고 가정하라.** 매 단계 첫 줄에 *지금 뭘·왜* 하는지 쉬운 말로 한두 문장 설명하고 진행한다. 전문용어는 풀어서. **사람을 시험하지 마라 — 일은 네가 하고, 사람은 게이트에서 검토만 한다.**

1. **R — research** *(자율)* → `kb/research.md`. 미션의 `Deliverable`·`Done`·`Notes`를 기준으로 코드·계약(`shared/`)·주변을 읽어 "무엇을 해야 하는가"를 파악. **미션의 `Target` 노드(`pm/features/<노드>.md`)와 그 노드가 가리키는 위키 스펙을 먼저 읽어라** — 이게 이 미션이 전진시키는 대상이고, 무엇·왜의 정본이다. 사람에겐 "지금 이 미션이 뭘 요구하는지 파악 중이에요"라고만.
2. **P — plan** *(자율)* → `kb/plan.md`. 알아낸 것으로 **어떻게 할지** 구체 계획(바꿀 파일·접근·순서).
3. **【RP 게이트】 — 채간 사람 본인의 셀프 검토** *(PM 보고 아님)*
   PM에게 올리는 자리가 아니다. **채간 사람이 방향을 한 번 훑고 스스로 승인**하는 자리다. 사람은 맥락이 없으니 네가 다 차려준다 — 이 순서로:
   - **㉠ 리뷰포인트를 딱딱 집는다** — 이 미션에 맞는 2~4개. 예: "① 이 계획이 미션 Done의 ___를 만족하나 ② 접근이 과하지 않나(YAGNI) ③ 놓친 접점(다른 팀·`shared/`)은 없나".
   - **㉡ 각 포인트에 네 권장 답을 준다** — "제 판단엔 ①②③ 다 괜찮아요, 이유는 …".
   - **㉢ 확인만 받는다** — "이 방향으로 갈게요, **이거 맞아요?** (맞으면 그대로 진행, 아니면 어디가 걸리는지만 말해주세요)". **열린 질문 던지지 마라.**
   - OK → **바로 I로.** 되돌림 → R/P 고쳐서 다시 이 게이트. 판정의 뿌리는 항상 미션 `Done`·`Notes`(`kb/gates.md`).
4. **I — implementation** *(자율, 게이트 없음)* → `kb/implement.md`. 승인된 계획대로 구현. Q까지 멈추지 말고 민다.
5. **Q — qa** *(자율)* → `kb/qa.md`. 미션 `Done`을 **한 항목씩** 대조 검증. 스코프 크리프는 잘라낸다(`kb/mindset.md`).
6. **【IQ 게이트】 — 채간 사람 본인의 셀프 검토** *(역시 PM 보고 아님)*
   - **㉠ 리뷰포인트** — "① Done 항목별 충족(체크리스트로) ② 남은 구멍·리스크 ③ 산출물이 어디에 남나".
   - **㉡ 네 권장 판단 + 셀프 QA 결과**를 체크리스트로 딱 보여준다.
   - **㉢ 확인** — "미션 Done 다 채웠어요, **이대로 마감할까요?**" → OK → **Step 4**. 되돌림 → I로.

> **게이트 원칙:** 두 게이트 다 = **채간 사람 본인의 검토**(PM 아님). 사이 구간(R→P, I→Q)은 AI 자율. 사람은 맥락이 없으니 항상 *㉠리뷰포인트 → ㉡네 권장 답 → ㉢"이거 맞아요?"* 형식으로 차려주고 **확인만** 받는다. 판정 기준은 언제나 그 미션의 `Done`·`Notes`.

### Step 4 — 마감 (IQ 게이트 통과 후 · 반드시 실행)

**미션은 연 사람이 닫는다.** IQ를 통과하면 곧바로, 순서대로:
1. **산출물 마무리** — 코드 미션이면 `scripts/create-pr.sh "<제목>" "<본문>"` (브랜치 push + 리뷰용 PR, base=dev). 반환된 URL을 사람에게 "리뷰용 PR 올렸어요" + 링크로. 코드가 아니면 미션이 요구한 산출물(문서·URL 등)을 확정.
   > **PR 본문 원칙 — 친절하되 극단 추상화·간결.** 리뷰어가 30초에 *무엇을·왜*를 잡게 3~5줄. 삽질 과정·배관·파일 나열 금지. "이 미션이 뭘 이뤘나 + 어디를 보면 되나"만. (스킬의 추상화 원칙은 PR에서도 유지된다.)
2. **미션 닫기** — `scripts/close-mission.sh <n> "<한 줄 마감 코멘트>"` → 보드 카드 Done + 이슈 close(코멘트에 무엇을 했는지 한 줄). PR을 올렸으면 코멘트에 PR 링크.
3. **기능 트리 갱신(필수)** — 이 미션의 `Target` 노드(`pm/features/<노드>.md`)의 상태 색과 `진행 상태`를, 그리고 루트 `README.md` 트리의 그 노드 색을, 이번 결과로 업데이트한다. 미션이 노드를 🔵/🟡 → 🟢 등으로 옮긴다 — 트리(현황)가 현실을 비추게 하는 마지막 한 칸. 빼먹으면 현황이 거짓말을 한다.
4. **결과 요약** — 사람에게 쉬운 말로: *뭘 했고 · 산출물이 어디 있고 · 미션은 닫혔다 · 트리 노드가 어떻게 바뀌었다.* 끝.

> 이 전체 흐름(신원 → 페치 → 착수 → 수행 → 마감)은 **오차 없이 순서대로** 돈다. 게이트는 하드 스톱, Step 4의 닫기는 필수 실행 — **건너뛰지 마라.**

### 막히면
PM 결정이 필요한 벽(범위·우선순위·제품방향·크로스팀)에 부딪히면 → **ask-pm 스킬**로 에스컬레이트. 기술적 how-to·버그는 팀의 학습이니 직접 푼다.
