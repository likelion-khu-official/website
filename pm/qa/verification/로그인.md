## 근거 (시나리오별 상세)

### AUTH-001 — pass

`find frontend/src -iname "*middleware*"`로 확인한 결과 **이 프로젝트엔 Next.js `middleware.ts`가 아예 없다.** 가드는 전부 클라이언트 컴포넌트 안에서 이뤄진다: `frontend/src/app/admin/page.tsx` → `AdminDashboard`(`frontend/src/components/admin/AdminDashboard.tsx:38-70`)가 마운트되면 `useEffect`에서 `refreshSession()`(`POST /api/admin/auth/refresh`)을 호출하고, 쿠키가 없거나 무효하면 401/`INVALID_REFRESH_TOKEN`을 받아 54-59행에서 `router.replace('/admin/login')`으로 보낸다. 이 요청이 실패하는 동안 `admins`·`currentAdmin` state는 채워지지 않으므로(48-51행은 성공 경로에서만 실행) 보호 데이터가 화면에 그려지는 순간은 없다 — 잠깐 "불러오고 있어요…" 로딩 텍스트만 보이고 곧바로 로그인 화면으로 이동한다. `/admin` 자체가 404로 뜨는 코드 경로는 없다(정상적으로 페이지는 렌더되고, 그 안에서 클라이언트가 리다이렉트).

noindex·nofollow는 `frontend/src/app/admin/layout.tsx:3-6`의 `metadata.robots = {index:false, follow:false}` — `/admin` 하위 전체에 적용되는 레이아웃 레벨 메타라 어드민 전 영역에 걸린다.

### AUTH-002 — pass (criteria 재작성함)

이 시나리오는 원래 "미들웨어(쿠키 존재 여부만 검사)가 있고, 그게 `/admin/login`↔`/admin`을 서로 되돌려 리다이렉트 루프를 만든다"고 서술돼 있었다. **코드를 직접 확인한 결과 이 전제가 사실이 아니었다** — `find frontend/src -iname "*middleware*"` 결과 0건, `middleware.ts` 자체가 프로젝트에 없다. 그래서 이 세션에서 `pm/qa/criteria/일반사용자.yaml`의 AUTH-002를 실제 아키텍처에 맞게 다시 썼다(아래는 새 버전 기준 검증).

실제 가드는 전부 클라이언트 컴포넌트다: `AdminDashboard.tsx`(`frontend/src/components/admin/AdminDashboard.tsx:38-70`)가 마운트되며 "불러오고 있어요…" 를 먼저 보여주고, `useEffect` 안에서 `refreshSession()`을 호출한다. 가짜 `access_token` 쿠키는 `JwtAuthenticationFilter.doFilterInternal()`(`JwtAuthenticationFilter.java:30-38`)의 `jwtProvider.parseClaims(token)`에서 서명 검증에 실패해 `Optional.empty()`가 되고(21-23행 주석: "쿠키가 없거나 무효해도 예외를 던지지 않고 그냥 익명으로 진행"), 요청은 익명으로 필터체인을 통과한다. `refreshSession()`은 access_token이 아니라 **refresh_token 쿠키**로 동작하므로(`AdminAuthService.refresh()`, `AdminAuthService.java:82-100`), refresh_token이 없으면 `InvalidRefreshTokenException` → 401 → `AdminDashboard.tsx:54-59`의 `router.replace('/admin/login')`이 **정확히 한 번** 실행된다. `/admin/login`(`LoginForm.tsx`)은 마운트 시 세션을 확인하거나 `/admin`으로 되돌리는 로직이 없으므로 여기서 흐름이 끝난다 — 되돌아갈 코드 경로가 없어 루프가 원천적으로 불가능하다. 로딩 중엔 `admins`·`currentAdmin` state가 비어 있어(48-51행은 성공 경로에서만 채워짐) 보호 데이터가 그려지는 순간도 없다.

### AUTH-003 — pass

`AdminAuthService.login()`(`AdminAuthService.java:52-70`): 이메일이 없거나(53-54행) 비밀번호가 틀리면(60-66행) 모두 `InvalidCredentialsException`을 던진다 — 존재하지 않는 이메일과 틀린 비밀번호가 같은 예외·같은 메시지로 합쳐져 계정 존재 여부가 안 드러난다. `InvalidCredentialsException`(`admin/exception/InvalidCredentialsException.java`)의 기본 메시지가 정확히 "이메일 또는 비밀번호가 올바르지 않아요." — `GlobalExceptionHandler.handleInvalidCredentials()`(`GlobalExceptionHandler.java:82-85`)가 `401 {code:'INVALID_CREDENTIALS'}`로 응답.

잠금은 `admin.recordFailedLogin(maxAttempts, Duration.ofMinutes(lockoutDurationMinutes))`(61행) — `application.yml:62-63`에서 `admin.lockout.max-attempts: 5`(기본값), `admin.lockout.duration-minutes: 15`(기본값). 5회째 실패 시 `admin.isLocked()`가 참이 돼 `AccountLockedException` → `handleAccountLocked()`(88-90행) → `423 LOCKED {code:'ACCOUNT_LOCKED'}`.

### AUTH-006 — pass

`find frontend/src/app/admin -type f`로 admin 하위 라우트를 전부 나열했다: `forgot-password`·`invite/[token]`·`login`·`page.tsx`·`reset-password/[token]`뿐이고 `signup`·`register` 같은 공개 가입 라우트가 없다. 백엔드 `AdminAuthController`에도 회원가입 엔드포인트가 없다(계정 생성은 초대 수락 `POST /api/admin/invitations/{token}/accept` 경로뿐이며 이는 초대 토큰이 있어야만 가능 — 공개 가입이 아니다).

### AUTH-007 — pass (PR #121, dev 미머지)

`gh pr diff 121 --repo likelion-khu-official/website`로 확인. `MemberAuthController.login()`(diff의 `MemberAuthController.java:31-34`) → `MemberAuthService.login()`(`MemberAuthService.java:41-56`, PR diff): 존재하지 않는 학번은 `InvalidCredentialsException("학번 또는 비밀번호가 올바르지 않아요.")`, 틀린 비밀번호도 동일 메시지·동일 예외(53-56행) — 존재 구분 없음. `maxAttempts`/`lockoutDurationMinutes`는 admin과 **같은 설정 키**(`admin.lockout.max-attempts`/`admin.lockout.duration-minutes`)를 그대로 재사용해 5회 실패 시 잠금(`AccountLockedException`).

`refresh()`(diff, `MemberAuthController.java` 41-46행 + `MemberAuthService.java` refresh 메서드): 쿠키가 없으면 `refreshTokenOf()`가 `null`을 반환하고 `MemberAuthService.refresh(null)`이 즉시 `InvalidRefreshTokenException`을 던져 401. `changePassword()`(`MemberAuthController.java` 54-61행)에 `@PreAuthorize("hasRole('MEMBER')")`가 붙어 있어 미인증 요청은 401 UNAUTHENTICATED로 걸린다 — then의 세 조건 모두 PR diff 코드로 확인했다. 단 이 PR은 아직 dev에 머지되지 않았고, 공개 화면(로그인 폼)도 없어 FE 진입점이 없다는 criteria의 note와 일치한다.
