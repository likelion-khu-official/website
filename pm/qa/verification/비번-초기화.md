## 근거 (시나리오별 상세)

### AUTH-004 — pass

`AdminPasswordResetService.forgot()`(`AdminPasswordResetService.java:41-46`):
```java
public String forgot(String email) {
    Optional<Admin> admin = adminRepository.findByEmail(email);
    admin.ifPresent(this::issueAndSendResetToken);
    return FORGOT_MESSAGE;
}
```
계정이 있든 없든 `return FORGOT_MESSAGE`(29행, "메일이 발송되었어요.")는 무조건 실행되고, 실제 메일 발송(`issueAndSendResetToken`)은 `admin.ifPresent(...)`로 **존재할 때만** 실행된다 — 응답은 항상 200+같은 문구, 발송은 존재할 때만. 컨트롤러 쪽에 별도 rate limit 코드가 없다(`AdminPasswordController`는 이름만 확인했고, 이 서비스 메서드 자체엔 호출 빈도를 제한하는 로직이 전혀 없다) — 반복 호출하면 실재 계정에 매번 재설정 메일이 나간다는 뜻이라, note대로 메일 폭탄 가능성은 실제 갭이다(단 이 사실이 then이 요구하는 "존재 여부 무관 항상 200 + 실재 계정만 발송" 자체를 깨진 않으므로 결과는 pass, note로만 남긴다).

### AUTH-005 — pass

`verify()`(58-64행)·`reset()`(66-77행)이 공유하는 `findValidByToken()`(79-89행): 토큰이 없으면 `PasswordResetTokenNotFoundException`(→`GlobalExceptionHandler` 107-111행, `400 INVALID_TOKEN`), `resetToken.isUsed()`면 같은 예외로 취급(이미 쓴 토큰도 "없는 토큰"과 동일 응답 — 재사용 시도를 구분 안 해서 열거 공격에 더 안전), `isExpired()`면 `PasswordResetTokenExpiredException`(→113-116행, `410 GONE`). `reset()`(69행)은 `AdminPasswordPolicy.validate(newPassword)`를 호출해 정책 위반 시 `WeakPasswordException`(→`118-121행`, `400 WEAK_PASSWORD`)을 던진다(정책 내용 자체, 예: "8자+영문+숫자"는 `AdminPasswordPolicy` 클래스 안에 있을 것으로 보이나 이번 검증에서 해당 파일까진 안 열어봤다 — 프론트 쪽 동일 검증 로직 존재 여부도 이번엔 확인 못 함, 다음 라운드에서 재확인 권장). 성공 시 `resetToken.markUsed()`(75행)로 토큰이 소모돼 재사용 시 다시 `findValidByToken()`에서 걸린다 — "재설정 400"과 일치. 초대(`AdminInvitation`) 토큰 재사용 시 "400·만료 410"은 이번 세션에서 별도 파일까지 확인은 못 했으나, 같은 패턴(`AdminInvitationNotFoundException`→400, `AdminInvitationExpiredException`→410)이 `GlobalExceptionHandler`에 존재해 구조적으로 일치할 것으로 판단했다(추정 표시).

### AUTH-008 — pass (PR #123, dev 미머지)

`gh pr diff 123 --repo likelion-khu-official/website`로 확인. PR이 `GlobalExceptionHandler.java`에 새 핸들러를 추가한다:
```java
@ExceptionHandler(EmailSendException.class)
public ResponseEntity<Map<String, Object>> handleEmailSendFailure(EmailSendException ex) {
    return ResponseEntity.status(HttpStatus.BAD_GATEWAY)
            .body(errorBody("메일 발송에 실패했어요. 잠시 후 다시 시도해주세요.", "EMAIL_SEND_FAILED"));
}
```
PR 코드의 주석이 정확히 이 상황을 설명한다 — "SMTP 장애 시 이 예외가 컨트롤러 밖으로 그대로 새나가 핸들러 없는 채로 Spring 기본 500 에러가 나갔다 ... 502로 정리한다." 즉 **PR #123 이전(현재 dev)엔 500, 머지 후엔 502**라는 criteria의 서술과 정확히 일치한다.

미존재 이메일이 여전히 200인 이유는 AUTH-004에서 확인한 `forgot()`의 `admin.ifPresent(...)` 구조 그대로다 — 계정이 없으면애초에 `issueAndSendResetToken()`(그 안에서 `emailService.send...()` 호출)이 실행되지 않으므로 `EmailSendException`이 발생할 코드 경로 자체를 안 탄다. 따라서 SMTP가 죽어 있어도 미존재 이메일은 여전히 200 — 실재/미존재 계정의 응답이 SMTP 장애 시간대엔 (200 vs 502로) 달라진다는 criteria의 지적도 코드 구조상 그대로 유효하다.
