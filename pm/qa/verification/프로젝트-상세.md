## 근거 (시나리오별 상세)

### PROJ-003 — 숨김/없는 프로젝트 id로 `GET /api/projects/{id}` → 404, 존재·숨김 미구분

무엇을 확인했나: PR #121(open, `feat/117-member-auth` → `dev`, 아직 머지 안 됨)의 프로젝트 상세 조회 로직이 "숨김"과 "존재하지 않음"을 같은 응답으로 처리하는지, 그리고 그 응답 형태가 시나리오가 요구하는 `{success:false, code:'PROJECT_NOT_FOUND'}`인지.

어느 파일을 봤나(전부 PR #121 diff):
- `backend/src/main/java/likelion/khu/website/project/ProjectController.java`(신규) — `@GetMapping("/api/projects/{id}")`가 `projectService.getPublicDetail(id)`를 그대로 반환, 인증 요구 없음.
- `backend/src/main/java/likelion/khu/website/project/ProjectService.java`의 `getPublicDetail()`:
  ```java
  public ProjectDetailResponse getPublicDetail(Long id) {
      Project project = projectRepository.findByIdAndHiddenFalse(id)
              .orElseThrow(ProjectNotFoundException::new);
      return toDetailResponse(project);
  }
  ```
  쿼리 조건이 `findByIdAndHiddenFalse` 하나뿐이다 — id가 존재하지 않는 경우와, id는 존재하지만 `hidden=true`인 경우 둘 다 이 쿼리는 빈 `Optional`을 반환하고, 두 경우 모두 똑같이 `ProjectNotFoundException`을 던진다. 즉 코드 구조상 "존재하지 않음"과 "숨김"을 애초에 구분할 방법이 없다 — 시나리오가 요구하는 "존재·숨김 구분 없음"과 정확히 일치한다.
- `backend/src/main/java/likelion/khu/website/common/GlobalExceptionHandler.java` diff:
  ```java
  @ExceptionHandler(ProjectNotFoundException.class)
  public ResponseEntity<Map<String, Object>> handleProjectNotFound(ProjectNotFoundException ex) {
      return ResponseEntity.status(HttpStatus.NOT_FOUND).body(errorBody(ex.getMessage(), "PROJECT_NOT_FOUND"));
  }
  ```
  `errorBody()`는 `Map.of("success", false, "message", message, "code", code)`를 만든다 — 응답이 정확히 `{success:false, code:'PROJECT_NOT_FOUND', message:...}` 형태다. 이 예외 핸들러는 PR #121에서 새로 추가된 것으로, PR 본문에 "이전엔 `ResponseStatusException`을 그대로 던져 success/code 없는 Spring 기본 에러 바디가 나갔다"는 배경 설명이 있다 — 즉 이번 PR이 이 계약 갭을 직접 고친 부분이다.
- `backend/src/test/java/likelion/khu/website/project/ProjectControllerTest.java`의 `get_HiddenProject_Returns404`, `get_NonExistentId_Returns404` 두 테스트가 각각 숨긴 프로젝트/존재하지 않는 id로 조회 시 `status().isNotFound()`를 검증한다 — 응답 바디의 code까지는 이 두 테스트에서 직접 assert하지 않지만, 예외 핸들러가 하나뿐이므로 두 경로 모두 같은 `PROJECT_NOT_FOUND` 바디로 귀결된다.

왜 그렇게 판단했나: 조회 쿼리 단계에서부터 "존재하지 않음"과 "숨김"이 하나의 `Optional.empty()`로 합쳐지고, 그 뒤 예외 처리도 단일 핸들러라 응답 형태가 시나리오 기대와 정확히 일치한다. **PR #121은 아직 dev에 머지되지 않은 open 상태**이므로 지금 dev를 배포하면 이 엔드포인트 자체가 없다는 점을 결과에 명시했다.

### PROJ-004 (상세 관점) — 숨김/복원/삭제 후 상세 API 재호출 시 반영

무엇을 확인했나: 어드민의 숨김·복원, 멤버의 삭제가 `getPublicDetail()`의 결과(200 vs 404)에 실제로 영향을 주는지.

어느 파일을 봤나(전부 PR #121 diff):
- `ProjectService.setHidden(id, hidden)`이 `project.setHidden(hidden)`으로 엔티티 플래그만 바꾼다. `getPublicDetail()`은 매 호출마다 `findByIdAndHiddenFalse(id)`로 다시 조회하므로, `hidden=true`로 바뀌면 다음 상세 조회부터 즉시 404가 되고, `hidden=false`로 되돌리면(복원) 같은 쿼리가 다시 해당 프로젝트를 찾아 200을 반환한다 — 별도 캐시가 없어 플래그 변경이 조회 결과에 직접 반영되는 구조다.
- `ProjectService.delete(id, memberId)`가 `projectRepository.deleteById(id)`로 행 자체를 지운다. 삭제 후에는 `findByIdAndHiddenFalse`든 관리자용 `findById`든 아무것도 찾지 못해 이후 상세 조회는 항상 `ProjectNotFoundException` → 404다.
- 테스트 `hidden_ByAdmin_HidesFromPublicButKeptInStorage`가 어드민이 숨긴 직후 `GET /api/projects/{id}`가 `status().isNotFound()`가 됨을 같은 테스트 안에서 검증한다(목록 재확인과 함께). `delete_ByParticipant_Returns200` 계열 테스트들이 참여 멤버의 삭제 권한을, `delete_NonExistentId_Returns404`가 이미 없는 id 삭제 시 404를 검증한다.

왜 그렇게 판단했나: 숨김→상세 404, 삭제→상세 404(행 자체 소멸), 복원→다시 200 세 갈래 모두 `hidden` 플래그 하나와 매번 재조회하는 쿼리 구조로 대칭적으로 설명된다. 목록 관점(`pm/qa/verification/프로젝트-목록.md`)과 마찬가지로, 복원(hidden:false) 케이스를 명시적으로 재조회까지 검증하는 테스트 메서드는 diff에서 별도로 찾지 못했지만, `setHidden()`이 `true`/`false`를 대칭적으로 처리하고 `getPublicDetail()`이 캐시 없이 매번 재조회하는 구조라 코드상 자명하다고 판단했다.

### PROJ-005 (상세 관점) — 상세 구성(이미지전부+참여멤버+기수+기간+스택+GitHub링크)

무엇을 확인했나: `find frontend/src/app/projects -type f`로 상세 화면 파일(예: `[id]/page.tsx`)이 실제로 있는지.

어느 파일을 봤나: `frontend/src/app/projects` 디렉터리 안에는 `page.tsx`(목록 플레이스홀더) 하나뿐이고, 동적 라우트(`[id]`) 디렉터리 자체가 없다 — 상세 화면 파일이 아예 존재하지 않는다.

참고로 백엔드 쪽은 PR #121의 `ProjectDetailResponse` DTO가 이미 상세 화면에 필요한 필드를 전부 갖고 있다:
```java
public class ProjectDetailResponse {
    private Long id;
    private String title;
    private String summary;
    private List<ProjectImageResponse> images;
    private List<ProjectParticipantResponse> participants;
    private Integer cohort;
    private LocalDate startDate;
    private LocalDate endDate;
    private Set<String> techStack;
    private String githubUrl;
    private boolean hidden;
    ...
}
```
`images`는 대표 여부와 무관하게 프로젝트의 이미지 전부를 리스트로 담고(`projectImageRepository.findAllByProjectIdOrderByIdAsc`), `participants`는 참여 멤버 이름·파트를 담는다(`ProjectParticipantResponse`). 기간은 `startDate`/`endDate` 두 필드로 표현된다.

왜 그렇게 판단했나: 시나리오는 "화면 구현되면"이 전제인 spec-only 항목이고, 상세 페이지 라우트 자체가 없어 렌더링을 검증할 수 없으므로 not-run이 맞다. 목록 쪽과 동일한 이유로, "API는 이미 준비돼 있고 FE 화면만 없다"는 점을 근거에 남겼다.

### PROJ-006 (상세 관점) — 익명이 상세 관련 쓰기 API 직접 호출 → 401 UNAUTHENTICATED

무엇을 확인했나: `/api/projects/{id}`에 대한 PATCH(수정)·DELETE(삭제), `/api/admin/projects/{id}/hidden`에 대한 PATCH(숨김)를 익명이 호출했을 때의 인증 경계.

어느 파일을 봤나(전부 PR #121 diff, 상세 리소스 대상 쓰기):
- `SecurityConfig.java` diff — 추가된 permitAll은 `.requestMatchers(HttpMethod.GET, "/api/projects", "/api/projects/*").permitAll()` 하나뿐이라 GET만 열려 있다. `/api/projects/{id}`에 대한 PATCH·DELETE, `/api/admin/projects/{id}/hidden`에 대한 PATCH는 이 matcher에 안 걸리고 파일 하단의 기존 `.anyRequest().authenticated()`로 떨어진다(이 줄 자체는 PR #121 변경 범위 밖 — 이미 dev 현재 `SecurityConfig.java`에 존재).
- `authenticationEntryPoint`가 401 `{success:false, code:'UNAUTHENTICATED', message:'로그인이 필요해요.'}`를 반환하는 동일한 로직이 이 경로에도 적용된다.
- `ProjectController.update()`·`delete()`에는 `@PreAuthorize("hasRole('MEMBER')")`, `setHidden()`에는 `@PreAuthorize("hasAnyRole('ADMIN','SUPER_ADMIN')")`이 붙어 있지만, 인증 자체가 안 된 요청은 이 평가 전에 걸러진다.
- `ProjectControllerTest.java`의 `update_Unauthenticated_Returns401`, `delete_Unauthenticated_Returns401`, `hidden_Unauthenticated_Returns401` 세 테스트가 각각 PATCH `/api/projects/{id}`, DELETE `/api/projects/{id}`, PATCH `/api/admin/projects/{id}/hidden`을 인증 없이 호출 시 `status().isUnauthorized()`를 직접 검증한다.

왜 그렇게 판단했나: 상세 리소스에 대한 세 쓰기 엔드포인트 모두 permitAll 범위 밖에 있고, 각각에 대응하는 401 테스트가 diff에 존재해 코드와 테스트가 일치한다. PR이 아직 dev에 머지되지 않았다는 점은 위와 동일하게 남긴다.
