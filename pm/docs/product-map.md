# 제품 지도 — 표면·역할·소유권

> 이 사이트가 **몇 개의 화면 층으로 나뉘고, 누가 무엇을 할 수 있고, 어떤 데이터를 누가 소유하나**를 한 곳에 고정한다. 정책이 흔들릴 때 돌아오는 기준점.
> 배경 결정: [`feed-write-policy.md`](./feed-write-policy.md)(블로그 글쓰기 대안 검토) · 브리프: [`brief.md`](./brief.md).

## 한 줄 정의

**3면 구조.** 공개(읽기) · 멤버(부원 셀프관리) · 어드민(최고관리자). 로그인은 **멤버·어드민 표면 뒤에만** 있고, 공개 화면은 인증을 아예 안 만진다.

---

## 왜 3면인가 (왜 로그인을 넣나)

원래 2면(공개 + 어드민)이었다. 블로그 글쓰기 신원 문제로 시작했지만, 진짜 이유는 **콘텐츠 소유권을 분산**하려는 것:

- 지금 랜딩·프로젝트·멤버 콘텐츠가 전부 **PM 한 명 병목**이다(그래서 아직 플레이스홀더 — 출시 못 하는 진짜 이유). 브리프 제약 "콘텐츠는 운영진이 채운다"가 현실에선 PM 혼자로 쏠린다.
- **멤버 표면 = 이 병목을 없애는 그릇.** 부원이 자기 프로필·프로젝트·글을 직접 올린다.
- 블로그 작성자 신원(사칭 방지)은 로그인 세션에서 자동으로 나온다 — 드롭다운·매직링크·OTP 다 불필요해짐.

**비용(감수):** 계정 생애주기(학기마다 온보딩/졸업). 완화 — 계정은 **어드민이 발급**(공개 회원가입 없음), 규모 20~30명, 인증 기반은 어차피 #74가 만든다.

**설계 원칙: 얇게.** 멤버 표면은 지금 필요한 것만. 단, **앞으로 부원이 하는 모든 기능이 여기 들어간다**는 걸 전제로 그릇만 열어둔다(과잉설계 금지, 확장 여지만).

## 세 표면

| 표면 | 대상 | URL | 인증 | 하는 일 |
|---|---|---|---|---|
| **공개** | 방문자 전원(신입후보·타동아리·기업) | `/*` | 없음 | 랜딩 6기둥 읽기, 블로그 읽기, 지원폼 제출, 모집알림 신청 |
| **멤버** | 부원(로그인) | `/member/*` | 멤버 로그인 | 글쓰기, 내 프로필 관리, 우리 팀 프로젝트 관리 |
| **어드민** | 최고관리자 | `/admin/*` | 최고관리자 role | 블로그 검수, 계정 발급, 지원자 열람, 전체 콘텐츠 관리 |

- 일반 방문자는 로그인 창을 볼 일이 없다 — **로그인 기능이 있는 것조차 모른다.** 공개 화면의 보안 핸들링 = 0.
- 멤버·어드민은 URL로만 진입, 진입하면 로그인 벽부터.

## 역할 × 권한

| | 방문자 | 멤버 | 최고관리자 |
|---|---|---|---|
| 랜딩·블로그 읽기 | ✅ | ✅ | ✅ |
| 지원폼 제출 / 모집알림 신청 | ✅ | ✅ | ✅ |
| 블로그 글쓰기(→ 검수 대기) | ❌ | ✅ | ✅ |
| 내 프로필 편집 | ❌ | ✅(본인) | ✅(전체) |
| 프로젝트 등록·편집 | ❌ | ✅(참여한 것) | ✅(전체) |
| 블로그 검수(게시·숨김) | ❌ | ❌ | ✅ |
| 계정 발급 / 지원자 열람 | ❌ | ❌ | ✅ |

- 권한 티어는 **멤버 / 최고관리자 두 단계로 시작.** 운영진 세분화(예: 파트장)는 필요해지면 그때. 지금 안 쪼갠다.

## 데이터 엔티티 · 소유권 (누가 편집하나)

| 엔티티 | 소유·편집 | 게시 전 검수 | 비고 |
|---|---|---|---|
| **Account** | 최고관리자 발급 | — | 공개 가입 없음. 멤버↔부원 연결 |
| **Member**(프로필) | 부원 본인 | 없음(사후 숨김 가능) | 이름·소개·이미지·파트 |
| **Post**(블로그) | 부원 작성 | **있음**(어드민 승인 후 공개) | 공식 얼굴로 나가서 검수 유지 |
| **Project** | 참여 부원 | v1은 신뢰 기반(사후 숨김) | 이미지·참여자·소개글 |
| **Comment** | 방문자/부원 | 없음 | 기존 유지 |

- **소유권 원칙:** 자기 것은 자기가(부원 self-service), 공식 채널로 **나가는** 것만 검수(블로그). 프로젝트는 일단 신뢰 기반 + 어드민 사후 숨김 — 문제 생기면 블로그처럼 검수 붙일 수 있게 상태기계는 열어둔다.

## 인증 모델 (얇게)

- **#74(어드민 파운데이션)가 로그인·세션·role을 제공.** 멤버 로그인은 그 위의 **role 티어 하나**다 — 별도 인증 시스템 아님.
- **계정 발급:** 최고관리자가 어드민 화면에서 부원 계정 생성(공개 회원가입 없음). 초기엔 8명 수동 시드.
- 비번 리셋 등은 필요 최소한으로. 세부는 #74 설계에서.

---

## 의존 · 시퀀싱

```
#74 어드민 파운데이션 (로그인·최고관리자 role·관리자 레이아웃)   ← 모든 것의 선행
      │
      ├─▶ 멤버 role 티어 + /member/* 껍데기 (로그인 가드)        ← 멤버 표면의 문
      │        │
      │        ├─▶ 블로그 글쓰기 이관 (작성자 = 세션)            ← #94·#95 대체본
      │        ├─▶ 내 프로필 관리
      │        └─▶ 프로젝트 관리
      │
      └─▶ #96 블로그 검수 어드민 화면 (이미 발주, #74 서브이슈)
```

**라이브 미션 처리(2026-07-10 결정):**
- **#94(BE)·#95(FE) 블로그 글쓰기 = 홀드.** 작성자 신원이 드롭다운/본문 → **로그인 세션**으로 바뀌므로 재설계 후 재개. (매직링크로 한 번 갈아엎었기에 세 번째 churn을 막으려 정책 확정까지 멈춤.) 홀드 코멘트 발송 완료.
- **#76(멤버 API)** — 드롭다운용 의존이 사라짐. 멤버 표면·프로필 관리 관점에서 범위 재정의 필요.
- **#96(검수)** — 영향 없음. 검수는 어느 구조에서든 필요. 그대로.

---

## 비범위 (일부러 안 하는 것)

- **공개 회원가입** — 계정은 어드민 발급만.
- **권한 세분화**(파트장 등) — 멤버/최고관리자 2단계로 시작.
- **프로젝트 검수 파이프라인** — v1은 신뢰 기반, 필요해지면 블로그식으로.
- **봇 가드·OTP·소셜로그인** — 안 함.
- **멤버 표면의 미래 기능**(출석·과제·내부자료 등) — 그릇만 열어두고 지금은 안 만든다.
