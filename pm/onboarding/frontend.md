# 프론트엔드 팀 — 업무 방식

이 문서는 이 프로젝트에서 **프론트엔드 팀이 일하는 방식**만 다룹니다.
(프로젝트 배경·목표는 `pm/docs/brief.md`, 첫 미션은 GitHub 이슈로 전달됩니다.)

> [!NOTE]
> 저장소 구조와 Claude로 일하는 법은 [**개발자 온보딩 — 모노레포 & AI-native**](dev-monorepo-ai.md)를 함께 읽어주세요.

---

## 1. 책임과 권한

프론트엔드 팀은 **화면 구현 · API 연동 · 반응형 · SEO**를 담당하며, 각 페이지의 **통합 드라이버**입니다 — 디자인 시안과 백엔드 API를 끌어와 페이지를 끝까지 책임집니다.

디자인의 *의도*(위계·강조·구성)는 따르되, 그것을 화면으로 옮기는 *구현*(반응형·상태·미세조정)은 프론트엔드의 영역입니다. 무엇을 우선 만들지는 PM이 정합니다.

## 2. 일하는 방식

| 원칙 | 내용 |
|---|---|
| **병렬 (릴레이 아님)** | 디자인·백엔드의 완성을 기다리지 않습니다. 거친 와이어프레임과 API mock만 있으면 곧바로 시작하며, 프로젝트 뼈대·라우팅·배포는 가장 먼저 깝니다. |
| **계약은 코드로** | 백엔드와의 API는 `shared/`에 **타입으로** 합의합니다. 백엔드 완성 전에는 그 타입으로 **mock**을 사용해 진행하고, 완성되면 실제 API로 교체합니다. |
| **의도 vs 구현** | 디자인의 *의도*는 따르되, 반응형·상태·엣지 케이스는 프론트가 채웁니다. 의도를 바꿔야 할 때만 디자인과 논의합니다. |
| **완료(Done)의 기준** | "화면이 떴다"가 아니라 **"반응형·실데이터·SEO까지 실서버에서 동작하고, 디자인 결과 QA를 통과"**입니다. |

## 3. 소통 방식

| 상황 | 채널 |
|---|---|
| **미션 전달** | PM이 GitHub 이슈로 전달합니다: `[FE] 미션 (~목표일)`. 이슈는 PM의 로드맵이므로 팀이 열고 닫지 않습니다. |
| **API 조율 (백엔드)** | API 합의·변경은 **PR·카카오톡에서 백엔드와 직접** 진행합니다. PM을 거치지 않습니다. |
| **시안 조율 (디자인)** | 시안 리뷰·의도 확인은 **Figma·카카오톡에서 디자인과 직접** 진행합니다. |
| **인프라** | 배포·인프라 관련은 **장찬욱(인프라)**과 직접 소통합니다. |

## 4. PM(김우진)의 역할

| 한다 | 하지 않는다 |
|---|---|
| 무엇·왜·우선순위 결정 | "어떻게"에 대한 지시 (구현은 팀의 몫) |
| 미션 전달, 막힌 것 해소 | 팀 간 기술 인터페이스 중계 (팀끼리 직접) |
| 콘텐츠 수급, 팀 간 통합 | 코드 마이크로매니징 |

> PM은 방향과 우선순위를 제시하는 사람이며, 팀의 작업을 어깨너머로 관리하지 않습니다.
